# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project shape

The Primer is a Socratic AI learning companion for children. The codebase is a **Rust workspace under `src/`** (not the repo root) — every cargo command must be run from `src/`. The workspace targets Rust **edition 2024** (requires Rust 1.85+ for default builds; the `speech` feature pulls vendored silero which needs 1.87+, pinned in `rust-toolchain.toml`). No system dependencies for default builds: SQLite is bundled, TLS uses rustls. `--speech` mode additionally needs system `espeak-ng`.

**Phase 0.1 is done; Phase 0.3 is substantially progressed.** Working today: streaming generation against Claude or Ollama, SQLite session persistence with resume-by-UUID, long-term memory (rolling LLM summary + FTS5 retrieval), engagement classifier, concept extractor, comprehension classifier, learner-model persistence, per-child vocabulary tracking with Leitner-box spaced repetition, locale-scoped knowledge base, graceful inference-error handling with retry, and a working voice round-trip POC (`--speech`). Still ahead: knowledge-base bootstrapping (Phase 0.2), session-time-based break suggestions (rest of Phase 0.3), local llama.cpp inference, voice-loop hardening, hardware integration. See [README.md](README.md) for the user-facing status, [ROADMAP.md](ROADMAP.md) for the phase plan, and [primer_technical_spec.md](primer_technical_spec.md) for the long-form vision.

## Common commands

All commands run from `src/`:

```bash
cargo build                                          # build the workspace
cargo run --bin primer                               # REPL with stub backend (no API key needed)
cargo run --bin primer -- --backend cloud --name Binti --age 8                # uses ANTHROPIC_API_KEY env var
cargo run --bin primer -- --backend cloud --model claude-opus-4-7             # override the default cloud model
cargo run --bin primer -- --backend ollama --model llama3.2                   # local Ollama at http://localhost:11434
cargo run --bin primer -- --session-db /tmp/explicit.db                       # override default session DB location
cargo run --bin primer -- --resume <uuid>                                     # resume a past session (no greeting)
cargo run --bin primer -- --no-persist                                        # in-memory session, nothing written to disk
cargo test                                           # run all tests (parser + dialogue manager coverage)
cargo test -p primer-pedagogy                        # tests for one crate
cargo test -p primer-pedagogy decide_intent          # single test by name substring
cargo clippy --workspace --all-targets               # lint
cargo fmt                                            # format
RUST_LOG=debug cargo run --bin primer                # verbose tracing output
cargo run --example tts_hello --features piper -- --onnx <path>.onnx --config <path>.onnx.json --out hello.wav   # Piper TTS smoke
```

CLI flags exposed by `primer-cli`: `--backend stub|cloud|ollama`, `--model` (Anthropic id for cloud, defaults to `claude-sonnet-4-6`; required local tag for ollama), `--ollama-url` (default `http://localhost:11434`), `--name`, `--age`, `--language <pack_id>` (locale pack id, default `en`; scopes both knowledge-base FTS tables and the learner's stored locale — see locale gotcha below), `--knowledge-db <path>` (defaults to `:memory:`), `--session-db <path>` (defaults to `~/.primer/<slugified-name>.db`, created if missing; pass an explicit path only to override), `--resume <uuid>` (resume a past session by UUID; reads from `--session-db`; errors if file or id is missing; no greeting on resume), `--no-persist` (in-memory session, nothing written to disk; mutually exclusive with `--resume` and `--session-db`), `--api-key` (or `ANTHROPIC_API_KEY` env), `--classifier-backend stub|cloud|ollama` (default: same as `--backend`), `--classifier-model <name>` (default: same as `--model`), `--classifier-timeout-ms <ms>` (default: 3000), `--extractor-backend stub|cloud|ollama` (default: same as `--backend`), `--extractor-model <name>` (default: same as `--model`), `--extractor-timeout-ms <ms>` (default: 5000), `--comprehension-backend stub|cloud|ollama` (default: same as `--backend`), `--comprehension-model <name>` (default: same as `--model`), `--comprehension-timeout-ms <ms>` (default: 5000), `--vocab-max-per-prompt <N>` (top-K most-overdue concepts to inject into the system prompt as passive vocab review hints; default 4, set 0 to disable), `--embedder-backend none|stub|fastembed|ollama` (default `none` = BM25-only, the pre-Phase-0.2.5 behaviour; `stub` is for testing the hybrid pipeline only and dilutes BM25 with hash noise; `fastembed` requires `--features primer-cli/embedding`, `ollama` requires `--features primer-cli/ollama-embedding`), `--embedder-model <name>` (default `bge-m3` for fastembed, `nomic-embed-text` for ollama), `--embedder-ollama-url <url>` (default `http://localhost:11434`), `--verbose` (prints `[intent]` / `[classifier]` / `[extractor]` / `[comprehension]` debug lines to stderr while stdout stays clean). Voice-mode flags (only with `--features primer-cli/speech`): `--speech`, `--whisper-model <path>`, `--voice-onnx <path>`, `--voice-config <path>`, `--voice <model-id>`, `--mic-silence-ms <ms>`. Type `quit`, `exit`, or `bye` to end the REPL.

Env files are auto-loaded at startup. Two locations checked, in order: (1) project-local `.env` (dotenvy searches cwd and ancestors), then (2) user-global `~/.primer_env`. Earlier sources win, so a per-repo `.env` overrides the home file. Copy `.env.example` → `.env` for a per-repo config, or drop `ANTHROPIC_API_KEY=...` into `~/.primer_env` to share across projects. Both `.env` and `*.local` variants are gitignored.

## Architecture: trait-based hardware abstraction

The central design principle is that **the pedagogical engine is decoupled from any specific inference backend, speech engine, or knowledge store via traits in `primer-core`**. Backend selection is a runtime config choice, not a code change. This is what allows Phase 0 (cloud) to share code with Phase 1 (llama.cpp / QNN NPU / RKNN NPU) and Phase 2 (Whisper + Piper).

The twelve crates form a layered dependency graph:

```
primer-cli  →  primer-pedagogy  →  primer-core  ←  primer-inference, primer-speech, primer-knowledge, primer-storage, primer-classifier, primer-comprehension, primer-extractor, primer-embedding, primer-kb-load
```

- **`primer-core`** — the only crate that defines public traits (`InferenceBackend`, `KnowledgeBase`, `Named`, `VoiceActivityDetector`, `SpeechToText`, `StreamingSpeechToText`, `TextToSpeech`, `StreamingTextToSpeech`, `LearnerStore`, `SessionStore`, `Embedder`) plus shared types (`LearnerModel` umbrella struct → `LearnerProfile` + `Vec<ConceptState>` + `LearningPreferences` + latest engagement snapshot; `Session`, `Turn`, `PedagogicalIntent`, `EngagementState`, `UnderstandingDepth`, `Prompt`, `Passage`, `AudioChunk`). The `vocab` module exposes the Leitner-box scheduler (`apply_box_transition`, `is_due`, `due_concepts`); `i18n::Locale` is the locale-pack identifier used everywhere user-visible strings render or per-locale storage is keyed; `consts` holds every tunable default (retry budgets, classifier/extractor/comprehension settings, vocab box intervals); `llm_util` holds `extract_first_json_object` + `truncate_to_chars` shared across the structured-output crates; `retry::retry_with_backoff` is the HTTP-neutral retry helper. All speech traits inherit from `Named`, so a backend that implements both the one-shot and streaming variant of a trait writes its `name()` impl exactly once. Everything else depends on this.
- **`primer-inference`** — `StubBackend` (canned Socratic responses, no model), `CloudBackend` (Anthropic Messages API), and `OllamaBackend` (local Ollama `/api/chat`, useful for prototype testing against real models without integrating llama.cpp directly). `LlamaCppBackend`, `QnnBackend`, `RknnBackend` are TODOs.
- **`primer-speech`** — Phase 2 speech backends. Stubs always available; real backends behind features: Silero VAD (`silero`), Whisper STT (`whisper`), Piper TTS (`piper`). Pure helper modules `vad_debounce` and `phrase_split` carry the streaming state machines so they can be unit-tested without a backend dep. The Piper backend uses `PhraseSplitter` to chunk LLM output on `. ! ?` and synthesise per-phrase via the vendored `piper-rs` (the upstream 0.1.9 release was patched in-tree at `src/vendor/piper-rs/` for `ort 2.0.0-rc.10` API compatibility; the patch lives behind `[patch.crates-io]` in the workspace `Cargo.toml`). One voice per `PiperTts` instance — `open_session(voice)` errors on a `model_id` mismatch; multi-voice runtime switching is deferred. Smoke binary `cargo run --example tts_hello --features piper` synthesises a fixed phrase to a 16-bit PCM WAV to validate a voice is wired correctly.
- **`primer-knowledge`** — `SqliteKnowledgeBase` using FTS5 with BM25 ranking, hybrid-augmented by an in-Rust cosine over per-passage f32 vectors when an `Embedder` is wired (Phase 0.2.5). **Per-locale tables**: each locale gets its own `passages_<pack_id>` (FTS5) + `passages_<pack_id>_meta` pair plus `embeddings_<pack_id>` (1:1 with content_rowid, BLOB column for the vector), opened via `SqliteKnowledgeBase::open_for_locale(path, locale)`. This keeps BM25 statistics locale-pure and lets the tokenizer choice diverge per locale without affecting others. A pre-locale legacy `passages` table is migrated into the requested locale's pair on first open. The retrieve path negates FTS5's negative rank so higher-score = more-relevant in the public API. **Schema is at user_version 3.** Migrations are idempotent CREATE-IF-NOT-EXISTS: v2 added `sources` (cross-locale licence + attribution registry); v3 added `embedding_models` (cross-locale, validates dim on rebind) + `embeddings_<pack>`. The `retrieve_hybrid` method runs BM25 (top `bm25_top_k`) and a vector cosine (top `vector_top_k`) and fuses both via `primer_core::rrf::fuse` with `rrf_k=60` (industry standard). When no embedder is wired, `retrieve_hybrid`'s default trait impl falls back to BM25-only — every consumer can call it unconditionally.
- **`primer-embedding`** — concrete `Embedder` backends. `StubEmbedder` (deterministic FNV+xorshift hash → L2-normalised f32 vector; default dim 384; model id `"stub-fxhash-v1"`) is always built and is the runtime fallback when a real-model download fails. Behind the `fastembed` feature: `FastEmbedBackend` wraps `fastembed-rs 5.7.0` (pinned because that's the latest fastembed line that uses `ort = "=2.0.0-rc.10"`, which our vendored silero/whisper/piper need; v5.9+ moved to rc.11/rc.12 and is incompatible until we bump those vendor patches). Default real model: BGE-M3 (1024-dim multilingual, ~570 MB int8 — covers EN/DE/JA/HI in one model so the initial test cohort never hits a model swap). Behind the `ollama` feature: `OllamaEmbedder` calls `/api/embeddings` (developer-prototyping path; not the recommended runtime backend). The `Embedder` trait inherits from `Named` so a single backend writes its `name()` once. Mismatched dim under an existing `model_id` is a hard error in both `primer-knowledge` and `primer-storage` — silent fallback would invisibly degrade retrieval and that's exactly the bug class we want to prevent.
- **`primer-kb-load`** — JSONL ingestion + auto-seed-on-empty + `--reembed` backfill for the knowledge base. `load_jsonl(kb, path)` ingests passages (idempotent: skips already-present ids). `auto_seed_if_empty(kb, locale)` discovers `seed_passages.<pack_id>.jsonl` via `$PRIMER_SEED_DIR` → `$XDG_DATA_HOME/primer/seed/` → walking up from `CARGO_MANIFEST_DIR` and loads it on a freshly-empty KB. `reembed_kb(kb, embedder, force, batch_size)` backfills embeddings for passages missing one. Standalone binary `primer-kb-load --reembed --knowledge-db <path> --locale <pack>` is feature-gated on `fastembed`.
- **`primer-storage`** — `SqliteSessionStore` persists every conversation `Session` (one DB per child, separate from the RAG corpus on privacy grounds). Schema is normalised: every categorical text column (`speaker`, `pedagogical_intent`, `concept`) is a foreign key into a lookup table. The `speakers`, `pedagogical_intents`, and `understanding_depths` lookup tables are seeded and validated against the canonical Rust enums on every `open()` — drift is a hard error. `concepts` is open-vocabulary and lazily populated. The trait exposes `save_session`, `load_session(id) -> Option<Session>`, `retrieve_session_turns(session_id, query, k, exclude_indices_at_or_after)` — the last two underpin the `--resume` flow and the FTS5-backed long-term memory layer — `save_comprehensions(...)` for `turn_comprehensions`, and `update_turn_concepts(...)` for the post-response extractor backfill. **Schema is at version 8.** Migrations layer additively, each idempotent and transaction-wrapped: v2 added `sessions.summary` + `sessions.summary_through_turn_index` plus a `turn_text_fts` FTS5 virtual table over `turns.text` with triggers; v3 added `turn_classifications` for `EngagementAssessment` persistence; v4 added the `understanding_depths` lookup plus `learners` and `learner_concepts` for learner-model persistence (adoption of an existing session's `learner_id` is a CLI-layer concern, not part of the migration); v5 added `comprehension_classifiers` + `turn_comprehensions` (UNIQUE `(turn_id, concept_id, classifier_id)` so re-classification is a no-op while a different classifier id lands as a parallel row); v6 added `learners.locale` + `concepts.concept_language_tag` (i18n preparation — every concept is stored with the locale it was extracted under so future multilingual sessions don't conflate); v7 added `learner_concepts.box_level INTEGER NOT NULL DEFAULT 0` for the Leitner-box scheduler; v8 added `embedding_models` + `embeddings_turns` (per-turn f32 vectors for hybrid long-term-memory retrieval). Each `apply_vN_migrations` runs on every open, brings older DBs up to current, and rolls back on partial failure. The `LearnerStore` trait (also implemented by `SqliteSessionStore`) exposes `load_learner` + `save_learner`; `SessionStore::most_recent_session_learner_id` supports the CLI's first-run-after-v4 adoption flow. `DialogueManager` accepts an optional `&dyn SessionStore`; when set, it saves after every turn (success or mid-stream error), on `open_session`, on `resume_session`, and on `close_session`. Save failures `tracing::warn!` instead of propagating. Implementation pattern mirrors `primer-knowledge`: a single `rusqlite::Connection` wrapped in `Mutex`, async trait methods with synchronous bodies — acceptable at conversation turn rates; revisit under profiling if a multi-process consumer ever appears.
- **`primer-classifier`** — engagement classifier trait + implementations. `EngagementClassifier` is the trait; `LlmEngagementClassifier` wraps `Arc<dyn InferenceBackend>` (a separate dep so the crate stays free of `primer-inference` build dep); `StubEngagementClassifier` powers tests with `with_response`/`with_script` constructors. Soft-fail policy: classifier errors never propagate up — log via `tracing::warn!` and return `EngagementAssessment::unknown_low_confidence(...)`. Configuration in `ClassifierSettings` (history_depth, blocking_timeout, confidence_threshold, recent_child_turns, max_output_chars, generation_max_tokens, generation_temperature, generation_top_p) — every numeric backed by `consts.rs` defaults. Schema v3 in `primer-storage` persists every classification to `turn_classifications` for offline analysis / future training data (schema is now at v4; v3 tables are unchanged).
- **`primer-extractor`** — concept-extractor trait + implementations. `ConceptExtractor` is the trait; `LlmConceptExtractor` wraps `Arc<dyn InferenceBackend>` (separate dep so the crate stays free of `primer-inference` build dep); `StubConceptExtractor` powers tests with `with_response`/`with_script` constructors. One LLM call per completed exchange returns `{ child_concepts, primer_concepts }` separately so each turn's `concepts` field is populated correctly. Soft-fail policy: extractor errors return `ConceptExtraction::empty()` and `tracing::warn!` — never propagate. Configuration in `ExtractorSettings` (blocking_timeout, recent_context_turns, max_output_chars, max_concepts_per_speaker, generation_max_tokens, generation_temperature, generation_top_p) — every numeric backed by `consts.rs` defaults.
- **`primer-comprehension`** — comprehension-classifier trait + implementations. `ComprehensionClassifier` is the trait; `LlmComprehensionClassifier` wraps `Arc<dyn InferenceBackend>` (a separate dep so the crate stays free of `primer-inference` build dep); `StubComprehensionClassifier` powers tests with `with_response`/`with_script` constructors. One LLM call per completed exchange returns a JSON array of per-concept assessments `{concept, depth, confidence, evidence}` against the candidate set the extractor surfaced (deduped union of `child_concepts ∪ primer_concepts`, capped at `max_concepts_per_call`). Soft-fail policy throughout — comprehension errors return `ComprehensionResult::empty()` and `tracing::warn!`, never propagate. Configuration in `ComprehensionSettings` (blocking_timeout, recent_context_turns, max_output_chars, max_concepts_per_call, confidence_threshold, generation_max_tokens, generation_temperature, generation_top_p, evidence_max_chars) — every numeric backed by `consts.rs` defaults. Schema v5 in `primer-storage` persists every assessment to `turn_comprehensions` for offline depth-trajectory analysis without involving the LLM.
- **`primer-pedagogy`** — the Primer's "soul". Two modules:
  - `prompt_builder` builds the system prompt (encoding the Socratic method as instructions, varying by age, engagement, and intent), assembles messages from session turns, and contains `decide_intent()` — the heuristic that picks the next `PedagogicalIntent`. `build_prompt` takes a `summary` (rolling pre-window summary) and a slice of `retrieved_older` turns; both are injected as system-prompt sections so the chat-message timeline (the last `context_window_turns` turns) stays linear.
  - `dialogue_manager` orchestrates a session: record child turn → decide intent → retrieve knowledge → retrieve long-term memory (summary + FTS-relevant older turns) → build prompt → call inference → record Primer turn → refresh summary if due → update learner model. New entry point `resume_session(loaded)` replaces `open_session()` for resumed flows: it swaps in the loaded `Session`, clears `ended_at`, refreshes the summary **only if the loaded one is stale** (covers fewer pre-window turns than the current range; a current summary is preserved verbatim — no point burning an LLM call on identical work), and persists. Two summary-refresh cadences: active-conversation `refresh_summary_if_due` uses a K-threshold (re-summarize when `context_window_turns` new pre-window turns have accumulated, default K=20) so per-turn dialogue doesn't trigger an LLM call every time the boundary advances; resume `refresh_summary_if_stale` uses a simple "is the summary out of date?" check (any uncovered pre-window content triggers a refresh). Summarization goes through `InferenceBackend::summarize` — stub returns a canned string; cloud/ollama default-impl through `generate`.

  The `DialogueManager` now owns the classifier flow: at the START of each turn it awaits the previous turn's classifier task (with bounded `blocking_timeout`, default 3000ms — bumped from an original 500ms after smoke testing showed even small models like gemma4:e4b need >500ms for the classification call, which meant `turn_classifications` rows mostly never landed and the `[classifier]` verbose line never printed) so engagement state is fresh before `decide_intent`. After saving the response it spawns a fresh `tokio::spawn(classifier.classify(...))` task that persists to `turn_classifications` and feeds `learner.recent_assessments` for the next turn. Sequential, in the natural inter-turn pause — memory-constrained-friendly. Classifier and storage are held as `Arc<dyn ...>` since the spawn requires `'static`; backend and knowledge stay as borrowed `&dyn`.

  The extractor flow mirrors the classifier exactly: at the end of each successful response the dialogue manager spawns `tokio::spawn(extractor.extract(...))` for the just-completed (child, primer) exchange. The task self-persists via the new `SessionStore::update_turn_concepts(session_id, turn_index, &concepts)` method (idempotent FK link into `turn_concepts`). At the start of the next turn `await_pending_extraction` waits up to `extractor_settings.blocking_timeout` (default 5000ms — looser than the classifier's 3000ms because no in-turn decision depends on extracted concepts; bumped from an initial 1500ms after smoke testing showed even small models like gemma4:e4b need >1500ms when chained with comprehension) and applies the result to in-memory `LearnerModel.concepts` (encounter_count++, depth=Aware on first sight). On timeout the task is detached so DB persistence still completes.

  The post-response background path is now `(classifier task) || (extractor → comprehension chain)`: extraction runs first inside the same spawn; comprehension runs only if extraction surfaced any concepts. The await at the start of the next turn applies extraction first (so any new concepts are inserted at `Aware` in `LearnerModel.concepts`) and then comprehension (so `apply_comprehension` can promote depths via monotonic max **and** advance `box_level` via `vocab::apply_box_transition`). Combined timeout is `extractor_settings.blocking_timeout + comprehension_settings.blocking_timeout` (default 5000 + 5000 = 10 000 ms); detach-on-timeout so DB persistence still completes. The 10s budget covers even small-model two-call latency in practice; the natural inter-turn pause for a child reading + thinking comfortably absorbs it.

  `build_turn_prompt` calls `primer_core::vocab::due_concepts` over the in-memory `LearnerModel.concepts` to pick up to `vocab_max_per_prompt` overdue concepts and injects them as a passive vocab-review section in the system prompt — the LLM weaves them in only if topically relevant; no drilling, no quizzing. The `build_system_prompt_with_pack_and_vocab` builder takes the due-vocab slice alongside the summary and retrieved-older turns; an empty slice omits the section.
- **`primer-cli`** — the REPL binary (binary name `primer`, not `primer-cli`).

`DialogueManager` holds `&dyn InferenceBackend` and `&dyn KnowledgeBase` (borrowed references, not owned), so swapping backends is a CLI/config decision and the engine is testable with stubs.

## Conventions and gotchas worth knowing

- **The workspace root is `src/Cargo.toml`, not the repo root.** All workspace deps are pinned there via `[workspace.dependencies]`; per-crate `Cargo.toml` files use `.workspace = true`.
- The binary is named `primer` (see `[[bin]]` in `primer-cli/Cargo.toml`), so `cargo run --bin primer`.
- **Real-backend streaming is live.** `CloudBackend::generate_stream` consumes Anthropic SSE (`event:`/`data:` framing, hand-rolled `SseBuffer` + `parse_anthropic_event`). `OllamaBackend::generate_stream` consumes NDJSON (one JSON object per `\n`-terminated line, `NdjsonBuffer` + `parse_ollama_line`). Both forward chunks via a `futures::channel::mpsc::unbounded` driven by a spawned tokio task. The stub backend still emits one final chunk by design — that's a degenerate but valid stream and shouldn't be "fixed".
- **`CloudBackend` currently maps `Role::System` to `"user"` in the messages array** (system instructions are a top-level field on the Anthropic API). `OllamaBackend` instead prepends a `system`-role message because Ollama's chat API has no separate system field. Watch this divergence when reworking prompt assembly.
- **`DialogueManager` exposes both `respond_to` and `respond_to_streaming(input, FnMut(&str))`.** The non-streaming method is now a thin wrapper around the streaming one with a no-op callback. On a mid-stream error, the partial Primer turn is **not** recorded into the session — the child turn stays, the Primer turn is dropped, and the error is returned.
- **Cloud model defaults to `claude-sonnet-4-6`** but is now overridable via `--model`. For ollama, `--model` is required (e.g., `--model llama3.2`).
- **`update_learner_model` in `dialogue_manager/learner_update.rs` is a deliberate placeholder** — it only does a crude word-count heuristic for `EngagementState`. Concept extraction has shipped (`primer-extractor`), comprehension classification has shipped (`primer-comprehension`, with monotonic-max depth promotion via `apply_comprehension`), and vocabulary spaced repetition has shipped (`primer-core::vocab` Leitner-box scheduler). The remaining placeholders here are the engagement-state heuristic itself and session-time-based break suggestions (the only Phase 0.3 item still outstanding per ROADMAP.md).
- **`dialogue_manager` is a directory module.** [src/crates/primer-pedagogy/src/dialogue_manager/](src/crates/primer-pedagogy/src/dialogue_manager/) — `mod.rs` holds the struct + type aliases + glue; per-axis files split the impl block: `lifecycle.rs` (new / open / resume / close / accessors), `turn.rs` (respond_to_streaming orchestrator + 7 private helpers), `background.rs` (await / drain / apply_*_outcome), `retrieval.rs` (retrieve_knowledge / retrieve_long_term_memory), `summary.rs` (refresh_summary_*), `learner_update.rs` (placeholder heuristic), `apply.rs` (pure free fns + their unit tests). Tests split per axis: `tests/{lifecycle_tests,turn_tests,background_tests}.rs` plus shared `test_support.rs` (mocks + builders). When adding a new method to `DialogueManager`, place it in the file matching its responsibility and use `pub(super)` visibility for cross-module access.
- **`decide_intent()` is the brain of the Socratic behaviour.** It is the most important function to test rigorously when adding pedagogical features. Characterization tests are in `prompt_builder::tests` (18 tests pinning current behaviour, including factual-question routing and the `Encouragement` / `SessionClose` split for disengagement); add to them when you change intent routing.
- Knowledge base default is `:memory:` (empty). Retrieval just returns no passages and the prompt builder gracefully omits the knowledge section. Bootstrapping a corpus is a Phase 0.2 task.
- Errors: use `primer_core::error::PrimerError` (variants per subsystem) and the `Result<T>` alias. Wrap external errors via `.map_err(|e| PrimerError::Inference(format!("...: {e}")))`-style.
- **`PrimerError::Inference` carries a typed `InferenceError`.** Six variants: `Auth`, `RateLimited { retry_after: Option<Duration> }`, `ServiceUnavailable`, `NetworkUnavailable`, `ModelNotFound { model }`, `Other(String)`. Variant fields are locale-neutral data; user-facing rendering is `primer_core::i18n::render_inference_error(err, locale)`. `From<&str>` and `From<String>` impls let producers say `format!(...).into()` for unmapped conditions; the result lands in `Other`. **`Other`'s inner string is dev-facing only — it never reaches users**; the i18n layer substitutes a generic message and the call site (CLI) logs the full error via `tracing::warn!`.
- **Retries happen at the request-send + status-check phase only.** Once `generate_stream` starts consuming bytes from a 2xx response, mid-stream errors propagate cleanly and the partial Primer turn drops at the dialogue-manager layer (existing behaviour). Both cloud and Ollama wrap the pre-stream phase in `primer_core::retry::retry_with_backoff`. The retry helper is HTTP-neutral (lives in `primer-core`, no `reqwest` dep) — backends translate their own status codes via `classify_anthropic_status` / `classify_ollama_status` before invoking it.
- **Retry budget defaults are tuned for conversational latency.** 3 attempts × ~250 ms × 2 ≈ ~1.75 s worst case before failure. `Retry-After` headers ≤ 5 s are honored exactly; longer waits surface immediately so children aren't left staring at a 30-second silent gap. Defaults in `primer_core::consts::retry`. Each retry decision emits `tracing::info!(target: "primer::retry", attempt, kind, delay_ms)` — a future voice-mode change can subscribe and play a "give me a moment" bridging phrase.
- **`primer_core::i18n::render_inference_error` is the single i18n boundary** for the inference-error surface. Phase 0.1 ships English-only via `Locale::English`. Adding locales is purely additive — no producer touches anything when the i18n PR lands. **Discipline:** never embed user-facing English in `InferenceError` variant fields (no `hint: String`); never render `Other`'s inner string to users. The `.map_err` wrapping site (formerly at `dialogue_manager.rs:534`, now in the inference-call path of `dialogue_manager/turn.rs::stream_inference_response`) that re-wrapped typed errors as `Other` was removed during the Phase 0.1 work — DO NOT reintroduce a similar wrap when adding new error-producing call sites; just propagate via `?`. See [docs/background_research/i18n_design.md](docs/background_research/i18n_design.md) for the long-form i18n vision.
- **Categorical text columns are normalised into lookup tables and stored as integer foreign keys.** Applies to every SQLite schema in this codebase (sessions, learner-model concepts when Phase 0.3 lands, etc.). For closed Rust enums (`Speaker`, `PedagogicalIntent`, `UnderstandingDepth`), the Rust enum is the single source of truth; the DB lookup table is a derived projection that the storage layer regenerates and validates against on every `open()`. Drift (mismatched name on a known id, unknown id) is a hard error — there is **no** API exposed for mutating lookup tables. Adding a new variant means: edit the Rust source, recompile, next `open()` seeds the new row. Retired integer IDs are never reused.
- **Schema migrations are idempotent column-add patterns wrapped in a single transaction.** Each `apply_vN_migrations` checks `pragma_table_info` before each ALTER and uses `CREATE VIRTUAL TABLE IF NOT EXISTS` / `CREATE TRIGGER IF NOT EXISTS`. Bumping `USER_VERSION` is one half of a migration; the other half is making the new objects safe to re-create on a re-open. The whole body runs inside `conn.unchecked_transaction()` so a partial failure (e.g. disk-full between the FTS create and a trigger create) rolls back to the pre-migration state. Reject `existing_version > USER_VERSION` (newer-than-this-build); silently update old `existing_version != USER_VERSION` to current after running migrations. **Adding a new schema version** = bump `USER_VERSION` in `schema.rs`, add `apply_vN_migrations` following the same pattern, and call it from the open path (the existing v2..v7 chain is the template).
- **FTS5 query inputs are sanitized.** Child input is fed through `sanitize_fts_phrase`: tokens are stripped of non-alphanumeric chars, FTS5 reserved keywords (`AND`, `OR`, `NOT`, `NEAR`) are dropped, surviving tokens are quoted and OR-joined. Empty result short-circuits the query. This is what allows `retrieve_session_turns` to take raw child input safely. BM25 ranking + `LIMIT k` keeps the OR-join from producing noise.
- **Long-term memory is a system-prompt concern, not a chat-message concern.** The chat `messages` array passed to inference is exactly `session.recent_turns(context_window_turns)`. Summary and retrieved-older turns are injected into the system prompt. This keeps the timeline the model sees linear and the context budget bounded even across hours of conversation.
- **Sessions persist by default.** Without `--session-db`, the CLI defaults to `~/.primer/<slugified-name>.db`. The directory is created on first run; an unwritable home is a hard error rather than a silent in-memory fallback. The first time the default-path file is created the CLI prints a one-line banner explaining where the data lives and how to opt out. `--resume <uuid>` requires the file to exist (no auto-create on a typo) and errors clearly when the id isn't present. `--no-persist` skips disk entirely (in-memory `:memory:` store); it conflicts with `--session-db` and `--resume` at parse time.
- **Slugged learner names are NFC-normalized and Unicode-aware.** `slug()` in `primer-cli` accepts any character Unicode classifies as alphanumeric (Latin, Cyrillic, CJK, etc.), lowercased where case applies. The input is NFC-normalized first so `José` (precomposed `é`) and `José` (decomposed `e` + combining acute) map to the same filename — otherwise two copies of the same child get two session DBs. Filesystems on macOS/Linux handle Unicode filenames natively; this is the right tradeoff for a children's product whose audience explicitly includes non-English names.
- **The engagement classifier is one model behind the chat.** Classification of turn N happens after the Primer's response to turn N is complete; its result is applied at the start of turn N+1's intent decision. This is a deliberate latency tradeoff for memory-constrained-device support and is invisible to children but worth knowing when reasoning about why a misclassified turn doesn't immediately yank the intent.
- **Concept extraction lags one turn for in-memory state.** The extractor task spawned at the end of turn N writes to `turn_concepts` directly, but the in-memory `LearnerModel.concepts` is only updated when the next turn starts (the same await-with-bounded-timeout pattern as the classifier). This means the system prompt for turn N does not include concepts surfaced *in* turn N — only those from earlier turns. Acceptable for v1; if it ever surprises a child ("I just told you about photosynthesis and you're acting like I didn't"), the fix is to apply the extraction synchronously before save_session.
- **Comprehension assessments lag one turn just like concepts.** `apply_comprehension` runs in the same `await_pending_post_response` step as `apply_extraction`, so depth promotions for turn N's exchange are visible to the system prompt at turn N+1, not turn N. Same combined timeout applies.
- **`apply_comprehension` is monotonic-max.** A weak exchange never demotes `concept.depth`; explicit forgetting is a Phase 0.3+ concern handled algorithmically over `turn_comprehensions`, not via another LLM call.
- **Per-concept comprehension assessments are persisted even when sub-threshold.** `confidence_threshold` (default 0.6) only gates the in-memory apply step; the longitudinal `turn_comprehensions` record gets every assessment, with confidence scores intact for offline filtering.
- **`extract_first_json_object` and `truncate_to_chars` live in `primer-core::llm_util`.** All three structured-output crates (`primer-classifier`, `primer-extractor`, `primer-comprehension`) import from there. Don't re-introduce per-crate copies.
- **`update_turn_concepts` exists because `save_session` skips already-persisted turns.** The append-only invariant on `save_session` (skip turns already on disk by index) means concept additions to a previously-saved turn would be silently dropped. `update_turn_concepts` is the explicit backfill path for the post-response extractor task. Do not move concept persistence back into `save_session` — that would require changing the skip rule and breaking the append-only invariant.
- **`InferenceBackend::generate(prompt, params)` does not take a model parameter.** Each backend instance owns its model name at construction. So `--classifier-model X` (without `--classifier-backend Y`) constructs a *separate* backend instance of the same type with model X — it does not reuse the main backend with a different model on a per-call basis.
- **`Named` is the single source of `name()` across every speech trait** (`VoiceActivityDetector`, `SpeechToText`, `StreamingSpeechToText`, `TextToSpeech`, `StreamingTextToSpeech`). Adding a new speech trait means inheriting from `Named` so backends never have to write `name()` twice.
- **`piper-rs 0.1.9` is vendored at `src/vendor/piper-rs/`.** The upstream crate was built against pre-`rc.10` `ort`; the rest of the workspace pins `ort = "=2.0.0-rc.10"` to keep `silero-vad-rust 6.2`'s `ndarray` consistent. The vendored copy carries a small patch fixing `Value::from_array` (now wants owned arrays), `Session::run` (now `&mut self`), and `try_extract_tensor` (returns `(&Shape, &[T])` now). `[patch.crates-io] piper-rs = { path = "vendor/piper-rs" }` in `src/Cargo.toml` makes it the resolved version. Until upstream releases an `rc.10`-compatible version, this fork stays.
- **`--speech` mode is gated by the `speech` feature on `primer-cli`** which pulls all four `primer-speech` features (`silero`, `whisper`, `piper`, `cpal`). Default builds stay light. Build with `~/.cargo/bin/cargo build --features primer-cli/speech`. Then pass `--speech --whisper-model <path> --voice-onnx <path.onnx> --voice-config <path.onnx.json>` at runtime. `clap` enforces that model paths travel with `--speech`.
- **Build with rustup, not Homebrew rust.** `rust-toolchain.toml` pins Rust 1.87+ but this pin is honored only by rustup proxy binaries. If Homebrew rust shadows on `PATH`, `cargo` silently uses Homebrew's 1.86 and silero will fail to compile. Always invoke as `~/.cargo/bin/cargo` for primer commands.
- **`silero-vad-rust 6.2.1` is vendored at `src/vendor/silero-vad-rust/`** with three patches: (a) `is_multiple_of(N)` → `% N == 0` — the upstream `unsigned_is_multiple_of` API is still unstable on rustc 1.97-nightly so the toolchain pin alone was not enough; (b) `load-dynamic` removed from the vendor's ort feature list to allow static linking; (c) crate-level `#![allow(unused_variables, dead_code)]` to silence two upstream warnings. Drop the vendor patch once upstream catches up.
- **`primer-speech`'s ort dep** explicitly lists `download-binaries` and `copy-dylibs` in its ort features (with `default-features = false`; without this explicit list those features are stripped and ONNX Runtime cannot be found at link time). Together with removing `load-dynamic` from the vendored silero, this achieves a fully static `libonnxruntime` link — no `dlopen` failure at runtime. The first build downloads the ONNX Runtime binary from cdn.pyke.io; subsequent builds use the cached copy.
- **`espeak-ng` must be installed system-wide** for `--speech` mode to produce phonemes. On macOS: `brew install espeak-ng`. On Debian/Ubuntu: `apt install espeak-ng-data`. The `speech_loop::run` function probes `/opt/homebrew/share`, `/usr/local/share`, and `/usr/share` at startup and sets `PIPER_ESPEAKNG_DATA_DIRECTORY` to the parent of the first complete `espeak-ng-data` directory found; `espeak-rs` ships an incomplete subset that causes phoneme failures for most voices.
- **Voice id must be explicitly wired.** `LoopBackends` has a `voice: VoiceProfile` field; production wires `model_id` from `cfg.voice_id` (the `--voice` flag). Before this fix, `run_loop` hardcoded `VoiceProfile::default()` (= en_US-amy-medium) and Piper rejected any non-default voice with a model-id mismatch. Always pass `--voice <model-id>` matching your `.onnx` filename.
- **The voice loop never lets the Primer speak over a child** (LATENT_THINK keeps the mic open across the LLM call; cancel-on-SpeechStart aborts the LLM if the child resumes). It also **never lets the child speak over the Primer** — an `Arc<AtomicBool>` `is_speaking` flag gates the audio thread during SPEAK: `run_loop` sets it true before SPEAK; after the flush sentinel returns it `await`s a `DrainHook` — a `tokio::task::spawn_blocking` wrapper around `primer_speech::wait_for_drain` — that polls the speaker ringbuf's `occupied_len` until it reads zero on 3 consecutive 10 ms checks (then waits an 80 ms grace for cpal's own output buffer, max 5 s sanity cap, with the grace included in the cap so total wallclock is bounded); only then does `run_loop` clear the flag. Going through `spawn_blocking` is what keeps the synchronous wait off the tokio worker — calling `std::thread::sleep` directly inside the on_audio callback (which runs in the same task as `run_loop`) would block other tasks for up to 5 s and panic outright on a single-threaded runtime. The audio thread checks the flag every 5ms and while true drains-and-discards mic samples, drops any active whisper session, and clears local buffers. Without this, the Primer's TTS leaked back through the mic and caused endless self-conversation. The speaker producer is wrapped in `Arc<Mutex<HeapProd<f32>>>` so both the on_audio push path and the drain hook can access it; the mutex is uncontended (push and observe never overlap). Both invariants are tested via the mock-driven `speech_loop` tests in `primer-cli`.
- **Speaker ringbuf is 240,000 samples** (~5 s at 48 kHz). The previous 24,000-sample (~500 ms) buffer dropped samples when on_audio produced faster than cpal drained — cutting off all but the first phrase of a multi-phrase response. Never shrink below ~5 s worth of samples.
- **Markdown is stripped before TTS.** `strip_markdown_for_tts` removes paired `*emphasis*`, `**strong**`, and `` `code` `` markers before pushing to Piper; bare unmatched markers stay; a `*` (or run of `**`) flanked by digits is replaced with " times " so multiplication and exponent notation read aloud (e.g. `5*3=15` → "5 times 3=15"). The visible transcript on stdout retains markdown.
- **The audio capture path for `--speech` runs on a dedicated `std::thread`**, not a tokio task — it owns the silero VAD and the active whisper streaming session and emits `VadEvent`s plus, on `SpeechEnd`, the finalized transcript through a channel that the `run_loop` consumes via a `ChannelStt` wrapper. This keeps the test/production separation clean (mocks bypass cpal entirely).
- **Resampler tail-handling is non-trivial.** rubato's `FftFixedIn` buffers FFT state internally. `on_audio` carries a `leftover: Vec<f32>` buffer across calls so phrase tails are prepended to the next call rather than zero-padded mid-stream; at end-of-turn, an empty-Vec flush sentinel zero-pads any leftover then drives 4 silence chunks (~186 ms input silence) through the resampler to drain its FFT-buffered output. Without both, the last syllable/word of each phrase and often the entire last phrase were silently discarded.
- **The on_audio push loop bails on cpal output stream errors.** When synthesis outruns cpal drain, the loop retries (5 ms sleep between attempts) via `primer_speech::push_all_with_bail` until cpal's ringbuf accepts the samples. If the cpal output stream errors (device removed, host audio service crashed), an `Arc<AtomicBool>` flag plumbed from `cpal::err_callback` to `SpeakerSink::errored_flag()` flips and the loop bails with a `tracing::warn!` about the discarded sample count — instead of spinning forever on a `push_slice` that returns 0. The earlier 1-second drop-on-timeout was the actual mechanism cutting off the end of long Primer responses.
- **Voice rate is 0.9 (length_scale ≈ 1.111) and inter-phrase silence is 200 ms.** Slightly slower than Piper's default; 200 ms of silence inserted between `AudioChunk`s (each chunk = one phrase). Adjust `VoiceProfile.rate` and the inter-chunk sleep in `speech_loop` if the voice feels too slow or too breathless.
- **Vocabulary spaced repetition is driven entirely by the comprehension classifier — no extra LLM call.** `primer_core::vocab::apply_box_transition(box_level, depth, confidence)` advances the box on a strong above-threshold reading, resets it on `Aware` or sub-confidence, and pins at `MAX_BOX_LEVEL` (= 4, 30-day interval). `due_concepts(concepts, now, max_n)` returns the top-N most-overdue `ConceptState`s using the box-interval table `[1d, 3d, 7d, 14d, 30d]`. The dialogue manager calls this in `build_turn_prompt` and feeds the slice into the prompt builder's vocab section. The LLM is told to weave words in only if topically relevant — never drill, never quiz. **Vocabulary entries are `ConceptState` rows, not a separate table** — keeping the schema flat preserves the Phase 4 anonymisation/contribution path the roadmap calls out.
- **Locale is per-learner and persisted.** `--language <pack_id>` sets `Locale` for the session; on first save it's written to `learners.locale` (schema v6). On `--resume`, if the CLI's `--language` differs from the resumed learner's stored locale the CLI errors and explains the two ways to resolve (drop `--language` or pass `--language <stored>`). The reason is that `concepts.concept_language_tag` would otherwise silently start tagging new rows with the wrong locale, corrupting the longitudinal vocabulary record. The knowledge base is keyed by locale via `SqliteKnowledgeBase::open_for_locale` so passages stay BM25-pure per language.
- **The `LearnerModel` is an umbrella struct, not a leaf.** It contains `LearnerProfile` (name, age, native language, interests), `Vec<ConceptState>` (the longitudinal mastery record), `LearningPreferences` (per-child tunables, e.g. `early_disengagement_threshold`), and the latest engagement snapshot. When persisting or extending, edit the leaf type that owns the field — the umbrella struct is just composition. `LearningPreferences` is open-vocabulary (children's interests don't fit a closed enum) so it serialises as JSON, while categorical leaves (`UnderstandingDepth`, `EngagementState`) are closed enums backed by lookup tables.
- **Hybrid retrieval is opt-in via the `embedding` cargo feature AND the `--embedder-backend` flag.** Two layers: (1) compile-time, the `embedding` cargo feature decides whether `FastEmbedBackend` is built at all; (2) runtime, `--embedder-backend none|stub|fastembed|ollama` (default `none`) decides which embedder the dialogue manager uses. `none` is BM25-only — exactly the pre-Phase-0.2.5 behaviour. `stub` constructs a deterministic FNV-hash embedder useful only for testing the hybrid pipeline structurally; in production it dilutes BM25 with semantic noise and should not be the default. The user's stated preference is for hybrid retrieval default-on once CI validates the cdn.pyke.io ort-runtime download; flip both `default = ["embedding"]` in `primer-cli/Cargo.toml` and the `--embedder-backend` default at that point. Without the feature, an explicit `--embedder-backend fastembed` exits with a build hint.
- **`embedding` and `speech` cargo features both pin `ort = "=2.0.0-rc.10"`** so `--features embedding,speech` builds cleanly. `primer-speech`'s vendored silero/whisper/piper pin rc.10; `primer-embedding`'s `fastembed` feature pins `fastembed-rs = "=5.7.0"`, the latest line that *also* uses rc.10 (5.9+ moved to rc.11/rc.12). **Bumping fastembed past 5.7.0 will break the speech build** until the speech vendor patches are upgraded in lockstep. Pinning the workspace ort to exactly rc.10 in two unrelated dep paths is the load-bearing constraint that keeps the two feature stacks compatible.
- **Hybrid retrieval = BM25 leg + vector cosine leg, fused via Reciprocal Rank Fusion.** `primer_core::rrf::fuse(&[ranked_lists], k)` is the only fusion implementation in the codebase. Industry default `k = 60` is the only value used; smaller weighs the very top more, larger flattens. Both `KnowledgeBase::retrieve_hybrid` and `SessionStore::retrieve_session_turns_hybrid` carry a default trait impl that falls back to BM25-only when no `Embedder` is wired — every consumer can call them unconditionally. `HybridParams::default()` is `(bm25_top_k=20, vector_top_k=20, final_top_k=5, rrf_k=60.0)`.
- **Embedder model identity is persisted in a per-DB `embedding_models` lookup table; mismatched dim or unrecorded model is a hard error.** Both `primer-knowledge` (schema v3) and `primer-storage` (schema v8) carry their own `embedding_models` table. The first call that writes a vector under model name X registers `(name=X, dim=N)`; subsequent writes that report a different dim under the same name fail loudly. This is the load-bearing invariant that prevents silent quality regressions when a user re-opens a DB with a different embedder. To deliberately swap models on an existing DB: `primer-kb-load --reembed --force` (KB) — symmetric session-DB tool is a TODO.
- **Embedding-on-save runs in a `tokio::spawn`, fire-and-forget, after `save_session`.** `DialogueManager::spawn_embedding_task` embeds the most-recent (child, primer) turns and calls `SessionStore::save_turn_embedding` for each. Idempotent at the storage layer (upsert by `turn_id`), so re-spawning over already-embedded turns is a no-op write. The just-saved turn may not have a vector yet at the next `retrieve_session_turns_hybrid` call, but that's fine — recent turns are inside the context window so the model already sees them directly; only older turns (where retrieval matters) reliably have vectors.
- **`StubEmbedder` produces hash vectors, not semantic vectors.** Two synonyms produce orthogonal vectors. Workspace tests use the stub to assert *structural* hybrid behaviour (RRF fusion correctness, schema migrations, dim-mismatch handling), never semantic similarity. The semantic-quality test for the real model lives behind `--features fastembed --ignored` and runs only on demand.
- **First-run BGE-M3 download is ~570 MB into `~/.cache/primer/models/` and uses the same `cdn.pyke.io` ONNX-runtime download as the speech feature.** Construction failure falls back to BM25-only with a `tracing::warn!` — the conversation still works, which is strictly better than refusing to start. `--embedder-backend none` skips the download entirely; `--embedder-backend stub` skips the download AND stays in-process but provides no semantic signal.

## Pedagogical principles encoded in the system prompt

These constrain prompt-builder and dialogue-manager changes — if a change makes the Primer more answer-y or more engagement-maximising, it is wrong:

- The Primer asks more questions than it answers; pure factual questions get a direct answer **then** a Socratic pivot.
- The Primer never tries to maximise engagement. It detects frustration/disengagement and offers breaks, scaffolding, or session close — never guilt.
- All learner data is local; cloud inference sends turns per-request only.
- Comprehension is verified through transfer questions, application, and contradiction probing — not assumed from a confident-sounding response.
