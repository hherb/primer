# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project shape

The Primer is a Socratic AI learning companion for children. The codebase is a **Rust workspace under `src/`** (not the repo root) — every cargo command must be run from `src/`. The workspace targets Rust **edition 2024** and pins toolchain **1.88** in `rust-toolchain.toml` (raised from 1.87 once `primer-gui` pulled in Tauri 2.11, whose transitive deps — darling 0.23, plist 1.9, serde_with 3.20, time 0.3.47 — require 1.88+; the speech vendor's earlier 1.87 floor is satisfied by any 1.88+ build). No system dependencies for default builds: SQLite is bundled, TLS uses rustls. `--speech` mode additionally needs system `espeak-ng`.

**Phase 0.1 is done; Phase 0.2 is done; Phase 0.3 is done.** Hybrid retrieval pipeline, JSONL ingestion + multi-file auto-seed, a 56-passage CC0 hand-drafted seed corpus across all five planned clusters (space, body, how-things-work, life, earth/weather), AND a 35-article Simple English Wikipedia layer (CC-BY-SA-3.0; physics fundamentals, chemistry, biology, earth science, health) all auto-load alongside each other on a fresh KB. Working today: streaming generation against Claude, Ollama, or any OpenAI-compatible server (oMLX, LM Studio, vLLM, etc. via `--backend openai-compat`), SQLite session persistence with resume-by-UUID, long-term memory (rolling LLM summary + FTS5 retrieval), hybrid retrieval (BM25 + dense-vector RRF, opt-in via `--embedder-backend`), multi-layer auto-seed corpus, tuned BM25-only AND hybrid retrieval defaults via diagnostic sweeps over the 91-passage corpus (hybrid: 100% loose / 100% strict recall on all 91 queries / 24 strict-subset canonical mappings — `KNOWN_FAILING_QUERIES_HYBRID` is now empty after the issue #45 corpus expansion that added the `seed:en:flowers` passage and a stomach-growl sentence to `seed:en:digestion`), engagement classifier, concept extractor, comprehension classifier, learner-model persistence, per-child vocabulary tracking with Leitner-box spaced repetition, locale-scoped knowledge base, graceful inference-error handling with retry, session-time-based break suggestions, and a working voice round-trip POC (`--speech`). Still ahead: local llama.cpp inference, voice-loop hardening, hardware integration. See [README.md](README.md) for the user-facing status, [ROADMAP.md](ROADMAP.md) for the phase plan, and [primer_technical_spec.md](primer_technical_spec.md) for the long-form vision.

## Common commands

All commands run from `src/`:

```bash
cargo build                                          # build the workspace
cargo run --bin primer                               # REPL with stub backend (no API key needed)
cargo run --bin primer -- --backend cloud --name Binti --age 8                # uses ANTHROPIC_API_KEY env var
cargo run --bin primer -- --backend cloud --model claude-opus-4-7             # override the default cloud model
cargo run --bin primer -- --backend ollama --model llama3.2                   # local Ollama at http://localhost:11434
cargo run --bin primer -- --backend openai-compat --model mlx-community/Qwen3-8B-4bit   # OpenAI-compatible server (oMLX, LM Studio, vLLM, etc.)
cargo run --bin primer -- --session-db /tmp/explicit.db                       # override default session DB location
cargo run --bin primer -- --resume <uuid>                                     # resume a past session (no greeting)
cargo run --bin primer -- --no-persist                                        # in-memory session, nothing written to disk
cargo test                                           # run all tests (parser + dialogue manager coverage)
cargo test -p primer-pedagogy                        # tests for one crate
cargo test -p primer-pedagogy decide_intent          # single test by name substring
cargo clippy --workspace --all-targets               # lint
cargo fmt                                            # format
RUST_LOG=debug cargo run --bin primer                # verbose tracing output
cargo run --example tts_hello --features piper -- --onnx <path>.onnx --config <path>.onnx.json --out hello.wav   # Piper TTS smoke
```

CLI flags exposed by `primer-cli`: `--backend stub|cloud|ollama|openai-compat`, `--model` (Anthropic id for cloud, defaults to `claude-sonnet-4-6`; required local tag for ollama; required for openai-compat), `--ollama-url` (default `http://localhost:11434`), `--openai-compat-url <url>` (default `http://localhost:8000`; or env `OPENAI_COMPAT_URL`), `--openai-compat-api-key <key>` (or env `OPENAI_COMPAT_API_KEY`; optional for local servers like oMLX/LM Studio, required for remote providers like Together/Groq), `--name`, `--age`, `--language <pack_id>` (locale pack id, default `en`; scopes both knowledge-base FTS tables and the learner's stored locale — see locale gotcha below), `--knowledge-db <path>` (defaults to `:memory:`), `--session-db <path>` (defaults to `~/.primer/<slugified-name>.db`, created if missing; pass an explicit path only to override), `--resume <uuid>` (resume a past session by UUID; reads from `--session-db`; errors if file or id is missing; no greeting on resume), `--no-persist` (in-memory session, nothing written to disk; mutually exclusive with `--resume` and `--session-db`), `--api-key` (or `ANTHROPIC_API_KEY` env), `--classifier-backend stub|cloud|ollama|openai-compat` (default: same as `--backend`), `--classifier-model <name>` (default: same as `--model`), `--classifier-timeout-ms <ms>` (default: 3000), `--extractor-backend stub|cloud|ollama|openai-compat` (default: same as `--backend`), `--extractor-model <name>` (default: same as `--model`), `--extractor-timeout-ms <ms>` (default: 5000), `--comprehension-backend stub|cloud|ollama|openai-compat` (default: same as `--backend`), `--comprehension-model <name>` (default: same as `--model`), `--comprehension-timeout-ms <ms>` (default: 5000), `--vocab-max-per-prompt <N>` (top-K most-overdue concepts to inject into the system prompt as passive vocab review hints; default 4, set 0 to disable), `--session-break-after-mins <N>` (minutes between break-suggestion nudges; default 30; must be ≥1; engagement-state overrides win), `--embedder-backend none|stub|fastembed|ollama|openai-compat` (default `none` = BM25-only, the pre-Phase-0.2.5 behaviour; `stub` is for testing the hybrid pipeline only and dilutes BM25 with hash noise; `fastembed` requires `--features primer-cli/embedding`, `ollama` requires `--features primer-cli/ollama-embedding`, `openai-compat` requires `--features primer-cli/openai-compat-embedding`), `--embedder-model <name>` (default `bge-m3` for fastembed, `nomic-embed-text` for ollama), `--embedder-ollama-url <url>` (default `http://localhost:11434`), `--embedder-openai-compat-url <url>` (falls back to `--openai-compat-url`), `--embedder-openai-compat-model <name>` (required when embedder backend is `openai-compat`), `--verbose` (prints `[intent]` / `[classifier]` / `[extractor]` / `[comprehension]` debug lines to stderr while stdout stays clean). Voice-mode flags (only with `--features primer-cli/speech`): `--speech`, `--whisper-model <path>`, `--voice-onnx <path>`, `--voice-config <path>`, `--voice <model-id>`, `--mic-silence-ms <ms>`. Type `quit`, `exit`, or `bye` to end the REPL.

Env files are auto-loaded at startup. Two locations checked, in order: (1) project-local `.env` (dotenvy searches cwd and ancestors), then (2) user-global `~/.primer_env`. Earlier sources win, so a per-repo `.env` overrides the home file. Copy `.env.example` → `.env` for a per-repo config, or drop `ANTHROPIC_API_KEY=...` into `~/.primer_env` to share across projects. Both `.env` and `*.local` variants are gitignored.

## Local pre-commit hook (optional, recommended)

A version-controlled pre-commit hook lives at [.githooks/pre-commit](.githooks/pre-commit). It runs `cargo fmt --all -- --check` from `src/` on commits that touch `.rs` files (and fast-skips otherwise). Opt in once per clone:

```bash
git config core.hooksPath .githooks
```

Bypass a single commit with `git commit --no-verify`. Disable permanently with `git config --unset core.hooksPath`. The CI step in [.github/workflows/ci.yml](.github/workflows/ci.yml) is the source of truth; the hook is an early-warning copy so drift fails locally instead of after a CI round-trip. The hook skips (with a one-line warning) when `cargo` isn't on `PATH` and `$CARGO` is unset, so docs-only contributors aren't forced to install rustup.

**Branch protection is the structural fix** for `main`. The recommended setup, configurable from GitHub repo settings, is to require the `cargo test (default features)` status check before merging to `main`. The local hook complements but does not replace this — the local hook closes the loop for direct `git commit` use; branch protection closes it at the merge boundary regardless of contributor discipline. Closes issue #96.

## Architecture: trait-based hardware abstraction

The central design principle is that **the pedagogical engine is decoupled from any specific inference backend, speech engine, or knowledge store via traits in `primer-core`**. Backend selection is a runtime config choice, not a code change. This is what allows Phase 0 (cloud) to share code with Phase 1 (llama.cpp / QNN NPU / RKNN NPU) and Phase 2 (Whisper + Piper).

The fifteen crates form a layered dependency graph:

```
primer-cli  ┐
primer-gui  ┴→  primer-engine  →  primer-pedagogy  →  primer-core  ←  primer-inference, primer-speech, primer-knowledge, primer-storage, primer-classifier, primer-comprehension, primer-extractor, primer-embedding, primer-kb-load

primer-qnn-sys (sibling — Phase 1.2 FFI scaffold; will be pulled into primer-inference behind a `qnn` cargo feature once step 1.2.2 lands the safe wrapper)
```

`primer-engine` holds shared wiring helpers — backend construction, path resolution, learner reconciliation — so the REPL (`primer-cli`) and the Tauri desktop binary (`primer-gui`, scaffolded today; engine wiring lands in step 3 of the GUI plan) construct identical session stacks without copy-pasting setup code. `primer-gui` is the Tauri 2.x desktop frontend; today it ships an empty window only (step 2 of 10).

- **`primer-core`** — the only crate that defines public traits (`InferenceBackend`, `KnowledgeBase`, `Named`, `VoiceActivityDetector`, `SpeechToText`, `StreamingSpeechToText`, `TextToSpeech`, `StreamingTextToSpeech`, `LearnerStore`, `SessionStore`, `Embedder`) plus shared types (`LearnerModel` umbrella struct → `LearnerProfile` + `Vec<ConceptState>` + `LearningPreferences` + latest engagement snapshot; `Session`, `Turn`, `PedagogicalIntent`, `EngagementState`, `UnderstandingDepth`, `Prompt`, `Passage`, `AudioChunk`). The `vocab` module exposes the Leitner-box scheduler (`apply_box_transition`, `is_due`, `due_concepts`); `i18n::Locale` is the locale-pack identifier used everywhere user-visible strings render or per-locale storage is keyed; `consts` holds every tunable default (retry budgets, classifier/extractor/comprehension settings, vocab box intervals); `llm_util` holds `extract_first_json_object` + `truncate_to_chars` shared across the structured-output crates; `retry::retry_with_backoff` is the HTTP-neutral retry helper. All speech traits inherit from `Named`, so a backend that implements both the one-shot and streaming variant of a trait writes its `name()` impl exactly once. Everything else depends on this.
- **`primer-inference`** — `StubBackend` (canned Socratic responses, no model), `CloudBackend` (Anthropic Messages API), `OllamaBackend` (local Ollama `/api/chat`, useful for prototype testing against real models without integrating llama.cpp directly), and `OpenAiCompatBackend` (any server speaking the OpenAI `/v1/chat/completions` SSE format — oMLX, LM Studio, vLLM, llama.cpp `--server`, Together, Groq, OpenRouter). `LlamaCppBackend`, `QnnBackend`, `RknnBackend` are TODOs (QnnBackend = Phase 1.2 step 1.2.2; the FFI scaffold underneath it lives in the sibling `primer-qnn-sys` crate as of step 1.2.1).
- **`primer-qnn-sys`** — low-level FFI scaffold for the Phase 1.2 Qualcomm NPU backend. Exposes hand-rolled `extern "C"` declarations for the six Genie C-API entry points the Primer needs (`GenieDialogConfig_createFromJson`, `GenieDialogConfig_free`, `GenieDialog_create`, `GenieDialog_setTokenCallback`, `GenieDialog_query`, `GenieDialog_free`) plus a `GenieLibrary` struct that `libloading::Library::new`s `libGenie.so` at runtime and eagerly resolves the six function pointers. **`target_os = "android"`-gated**: on every other target, `GenieLibrary::open` returns `GenieLibraryError::PlatformUnsupported(os)` without touching the filesystem, which is what keeps `cargo check` green from macOS/Linux desktops and CI runners. The hand-rolled binding path is the documented fallback in the Phase 1.2 plan's risk register ("`bindgen` produces unusable bindings for some Genie types → hand-roll the function declarations (only ~6 functions needed)") — it is also load-bearing here because the QAIRT SDK is gated behind a Qualcomm developer-portal login and cannot be vendored into a public AGPL repo without a licence-redistribution pass that step 1.2.0 unblocks. Swapping to `build.rs`-generated `bindgen` over vendored headers is a straightforward follow-up: the public surface (raw types + `extern "C"` signatures) is identical. No safety wrappers live here — the `tokio::sync::Mutex<GenieDialogSession>`, the C-ABI token callback that bridges into `futures::channel::mpsc::UnboundedSender`, the `Drop` impl, and the `InferenceBackend` impl all land in `primer-inference::qnn` in step 1.2.2.
- **`primer-speech`** — Phase 2 speech backends. Stubs always available; real backends behind features: Silero VAD (`silero`), Whisper STT (`whisper`), Piper TTS (`piper`). Pure helper modules `vad_debounce` and `phrase_split` carry the streaming state machines so they can be unit-tested without a backend dep. The Piper backend uses `PhraseSplitter` to chunk LLM output on `. ! ?` and synthesise per-phrase via the vendored `piper-rs` (the upstream 0.1.9 release was patched in-tree at `src/vendor/piper-rs/` for `ort 2.0.0-rc.10` API compatibility; the patch lives behind `[patch.crates-io]` in the workspace `Cargo.toml`). One voice per `PiperTts` instance — `open_session(voice)` errors on a `model_id` mismatch; multi-voice runtime switching is deferred. Smoke binary `cargo run --example tts_hello --features piper` synthesises a fixed phrase to a 16-bit PCM WAV to validate a voice is wired correctly. The `voice_loop` module (behind the `voice-loop` cargo feature) holds the shared LISTEN → LATENT_THINK → SPEAK state machine that both `primer-cli` (`--speech` mode) and `primer-gui` (Voice mode toggle) consume via different `LoopObserver` implementations — CLI uses `StdoutObserver`, GUI uses `TauriEventObserver` emitting `primer://voice/*` events. `run_loop` returns a `LoopHandle` with `stop_tx` (CLI Ctrl+C / GUI End-voice-mode) and `cancel_response_tx` (GUI Stop button + Esc; aborts the in-flight LLM + TTS and returns to LISTEN). A sibling `run_loop_borrowed` keeps the `'r`-lifetime `&mut DialogueManager` shape for tests and for the CLI (which owns the DM directly and doesn't need Arc<Mutex<>>). The cpal/silero/whisper/piper audio-backend builder lives in `voice_loop::backends::build_local_backends` so both binaries share one construction path; the GUI wraps it in `primer-gui::voice::backends`, the CLI calls it directly from `primer-cli/src/speech_loop/mod.rs`.
- **`primer-knowledge`** — `SqliteKnowledgeBase` using FTS5 with BM25 ranking, hybrid-augmented by an in-Rust cosine over per-passage f32 vectors when an `Embedder` is wired (Phase 0.2.5). **Per-locale tables**: each locale gets its own `passages_<pack_id>` (FTS5) + `passages_<pack_id>_meta` pair plus `embeddings_<pack_id>` (1:1 with content_rowid, BLOB column for the vector), opened via `SqliteKnowledgeBase::open_for_locale(path, locale)`. This keeps BM25 statistics locale-pure and lets the tokenizer choice diverge per locale without affecting others. A pre-locale legacy `passages` table is migrated into the requested locale's pair on first open. The retrieve path negates FTS5's negative rank so higher-score = more-relevant in the public API. **Schema is at user_version 3.** Migrations are idempotent CREATE-IF-NOT-EXISTS: v2 added `sources` (cross-locale licence + attribution registry); v3 added `embedding_models` (cross-locale, validates dim on rebind) + `embeddings_<pack>`. The `retrieve_hybrid` method runs BM25 (top `bm25_top_k`) and a vector cosine (top `vector_top_k`) and fuses both via `primer_core::rrf::fuse` with `rrf_k=60` (industry standard). When no embedder is wired, `retrieve_hybrid`'s default trait impl falls back to BM25-only — every consumer can call it unconditionally.
- **`primer-embedding`** — concrete `Embedder` backends. `StubEmbedder` (deterministic FNV+xorshift hash → L2-normalised f32 vector; default dim 384; model id `"stub-fxhash-v1"`) is always built and is the runtime fallback when a real-model download fails. Behind the `fastembed` feature: `FastEmbedBackend` wraps `fastembed-rs 5.7.0` (pinned because that's the latest fastembed line that uses `ort = "=2.0.0-rc.10"`, which our vendored silero/whisper/piper need; v5.9+ moved to rc.11/rc.12 and is incompatible until we bump those vendor patches). Default real model: BGE-M3 (1024-dim multilingual, ~570 MB int8 — covers EN/DE/JA/HI in one model so the initial test cohort never hits a model swap). Behind the `ollama` feature: `OllamaEmbedder` calls `/api/embeddings` (developer-prototyping path; not the recommended runtime backend). Behind the `openai-compat` feature: `OpenAiCompatEmbedder` calls `/v1/embeddings` with native batch support (sends the full batch in a single request, unlike `OllamaEmbedder` which loops one-at-a-time) — works with oMLX, LM Studio, vLLM, and remote providers. The `Embedder` trait inherits from `Named` so a single backend writes its `name()` once. Mismatched dim under an existing `model_id` is a hard error in both `primer-knowledge` and `primer-storage` — silent fallback would invisibly degrade retrieval and that's exactly the bug class we want to prevent.
- **`primer-kb-load`** — JSONL ingestion + auto-seed-on-empty + `--reembed` backfill for the knowledge base. `load_jsonl(kb, path)` ingests passages (idempotent: skips already-present ids). `auto_seed_if_empty(kb, locale)` discovers ALL `*.<pack_id>.jsonl` files in the first matching seed dir via `$PRIMER_SEED_DIR` → `$XDG_DATA_HOME/primer/seed/` → walking up from `CARGO_MANIFEST_DIR` and loads each in lexicographic order, so `seed_passages.en.jsonl` (CC0 hand-drafted) and `wiki_passages.en.jsonl` (CC-BY-SA-3.0 Wikipedia layer) auto-load alongside each other. `discover_seed_files(locale)` returns the full list; `discover_seed_jsonl(locale)` is a back-compat shim that returns only the canonical `seed_passages.<pack>.jsonl`. `reembed_kb(kb, embedder, force, batch_size)` backfills embeddings for passages missing one. Standalone binary `primer-kb-load --reembed --knowledge-db <path> --locale <pack>` is feature-gated on `fastembed`. The Python ingestion pipeline at `data/ingest/` regenerates the wiki layer (lead-only chunking via the MediaWiki extracts API; batched 20-per-request with disambiguation-page detection); the developer runs it once and commits the resulting JSONL.
- **`primer-storage`** — `SqliteSessionStore` persists every conversation `Session` (one DB per child, separate from the RAG corpus on privacy grounds). Schema is normalised: every categorical text column (`speaker`, `pedagogical_intent`, `concept`) is a foreign key into a lookup table. The `speakers`, `pedagogical_intents`, and `understanding_depths` lookup tables are seeded and validated against the canonical Rust enums on every `open()` — drift is a hard error. `concepts` is open-vocabulary and lazily populated. The trait exposes `save_session`, `load_session(id) -> Option<Session>`, `retrieve_session_turns(session_id, query, k, exclude_indices_at_or_after)` — the last two underpin the `--resume` flow and the FTS5-backed long-term memory layer — `save_comprehensions(...)` for `turn_comprehensions`, and `update_turn_concepts(...)` for the post-response extractor backfill. **Schema is at version 8.** Migrations layer additively, each idempotent and transaction-wrapped: v2 added `sessions.summary` + `sessions.summary_through_turn_index` plus a `turn_text_fts` FTS5 virtual table over `turns.text` with triggers; v3 added `turn_classifications` for `EngagementAssessment` persistence; v4 added the `understanding_depths` lookup plus `learners` and `learner_concepts` for learner-model persistence (adoption of an existing session's `learner_id` is a CLI-layer concern, not part of the migration); v5 added `comprehension_classifiers` + `turn_comprehensions` (UNIQUE `(turn_id, concept_id, classifier_id)` so re-classification is a no-op while a different classifier id lands as a parallel row); v6 added `learners.locale` + `concepts.concept_language_tag` (i18n preparation — every concept is stored with the locale it was extracted under so future multilingual sessions don't conflate); v7 added `learner_concepts.box_level INTEGER NOT NULL DEFAULT 0` for the Leitner-box scheduler; v8 added `embedding_models` + `embeddings_turns` (per-turn f32 vectors for hybrid long-term-memory retrieval). Each `apply_vN_migrations` runs on every open, brings older DBs up to current, and rolls back on partial failure. The `LearnerStore` trait (also implemented by `SqliteSessionStore`) exposes `load_learner` + `save_learner`; `SessionStore::most_recent_session_learner_id` supports the CLI's first-run-after-v4 adoption flow. `DialogueManager` accepts an optional `&dyn SessionStore`; when set, it saves after every turn (success or mid-stream error), on `open_session`, on `resume_session`, and on `close_session`. Save failures `tracing::warn!` instead of propagating. Implementation pattern mirrors `primer-knowledge`: a single `rusqlite::Connection` wrapped in `Mutex`, async trait methods with synchronous bodies — acceptable at conversation turn rates; revisit under profiling if a multi-process consumer ever appears.
- **`primer-classifier`** — engagement classifier trait + implementations. `EngagementClassifier` is the trait; `LlmEngagementClassifier` wraps `Arc<dyn InferenceBackend>` (a separate dep so the crate stays free of `primer-inference` build dep); `StubEngagementClassifier` powers tests with `with_response`/`with_script` constructors. Soft-fail policy: classifier errors never propagate up — log via `tracing::warn!` and return `EngagementAssessment::unknown_low_confidence(...)`. Configuration in `ClassifierSettings` (history_depth, blocking_timeout, confidence_threshold, recent_child_turns, max_output_chars, generation_max_tokens, generation_temperature, generation_top_p) — every numeric backed by `consts.rs` defaults. Schema v3 in `primer-storage` persists every classification to `turn_classifications` for offline analysis / future training data (schema is now at v4; v3 tables are unchanged).
- **`primer-extractor`** — concept-extractor trait + implementations. `ConceptExtractor` is the trait; `LlmConceptExtractor` wraps `Arc<dyn InferenceBackend>` (separate dep so the crate stays free of `primer-inference` build dep); `StubConceptExtractor` powers tests with `with_response`/`with_script` constructors. One LLM call per completed exchange returns `{ child_concepts, primer_concepts }` separately so each turn's `concepts` field is populated correctly. Soft-fail policy: extractor errors return `ConceptExtraction::empty()` and `tracing::warn!` — never propagate. Configuration in `ExtractorSettings` (blocking_timeout, recent_context_turns, max_output_chars, max_concepts_per_speaker, generation_max_tokens, generation_temperature, generation_top_p) — every numeric backed by `consts.rs` defaults.
- **`primer-comprehension`** — comprehension-classifier trait + implementations. `ComprehensionClassifier` is the trait; `LlmComprehensionClassifier` wraps `Arc<dyn InferenceBackend>` (a separate dep so the crate stays free of `primer-inference` build dep); `StubComprehensionClassifier` powers tests with `with_response`/`with_script` constructors. One LLM call per completed exchange returns a JSON array of per-concept assessments `{concept, depth, confidence, evidence}` against the candidate set the extractor surfaced (deduped union of `child_concepts ∪ primer_concepts`, capped at `max_concepts_per_call`). Soft-fail policy throughout — comprehension errors return `ComprehensionResult::empty()` and `tracing::warn!`, never propagate. Configuration in `ComprehensionSettings` (blocking_timeout, recent_context_turns, max_output_chars, max_concepts_per_call, confidence_threshold, generation_max_tokens, generation_temperature, generation_top_p, evidence_max_chars) — every numeric backed by `consts.rs` defaults. Schema v5 in `primer-storage` persists every assessment to `turn_comprehensions` for offline depth-trajectory analysis without involving the LLM.
- **`primer-pedagogy`** — the Primer's "soul". Two modules:
  - `prompt_builder` builds the system prompt (encoding the Socratic method as instructions, varying by age, engagement, and intent), assembles messages from session turns, and contains `decide_intent()` — the heuristic that picks the next `PedagogicalIntent`. `build_prompt` takes a `summary` (rolling pre-window summary) and a slice of `retrieved_older` turns; both are injected as system-prompt sections so the chat-message timeline (the last `context_window_turns` turns) stays linear.
  - `dialogue_manager` orchestrates a session: record child turn → decide intent → retrieve knowledge → retrieve long-term memory (summary + FTS-relevant older turns) → build prompt → call inference → record Primer turn → refresh summary if due → update learner model. New entry point `resume_session(loaded)` replaces `open_session()` for resumed flows: it swaps in the loaded `Session`, clears `ended_at`, refreshes the summary **only if the loaded one is stale** (covers fewer pre-window turns than the current range; a current summary is preserved verbatim — no point burning an LLM call on identical work), and persists. Two summary-refresh cadences: active-conversation `refresh_summary_if_due` uses a K-threshold (re-summarize when `context_window_turns` new pre-window turns have accumulated, default K=20) so per-turn dialogue doesn't trigger an LLM call every time the boundary advances; resume `refresh_summary_if_stale` uses a simple "is the summary out of date?" check (any uncovered pre-window content triggers a refresh). Summarization goes through `InferenceBackend::summarize` — stub returns a canned string; cloud/ollama default-impl through `generate`.

  The `DialogueManager` now owns the classifier flow: at the START of each turn it awaits the previous turn's classifier task (with bounded `blocking_timeout`, default 3000ms — bumped from an original 500ms after smoke testing showed even small models like gemma4:e4b need >500ms for the classification call, which meant `turn_classifications` rows mostly never landed and the `[classifier]` verbose line never printed) so engagement state is fresh before `decide_intent`. After saving the response it spawns a fresh `tokio::spawn(classifier.classify(...))` task that persists to `turn_classifications` and feeds `learner.recent_assessments` for the next turn. Sequential, in the natural inter-turn pause — memory-constrained-friendly. Classifier and storage are held as `Arc<dyn ...>` since the spawn requires `'static`; backend and knowledge stay as borrowed `&dyn`.

  The extractor flow mirrors the classifier exactly: at the end of each successful response the dialogue manager spawns `tokio::spawn(extractor.extract(...))` for the just-completed (child, primer) exchange. The task self-persists via the new `SessionStore::update_turn_concepts(session_id, turn_index, &concepts)` method (idempotent FK link into `turn_concepts`). At the start of the next turn `await_pending_extraction` waits up to `extractor_settings.blocking_timeout` (default 5000ms — looser than the classifier's 3000ms because no in-turn decision depends on extracted concepts; bumped from an initial 1500ms after smoke testing showed even small models like gemma4:e4b need >1500ms when chained with comprehension) and applies the result to in-memory `LearnerModel.concepts` (encounter_count++, depth=Aware on first sight). On timeout the task is detached so DB persistence still completes.

  The post-response background path is now `(classifier task) || (extractor → comprehension chain)`: extraction runs first inside the same spawn; comprehension runs only if extraction surfaced any concepts. The await at the start of the next turn applies extraction first (so any new concepts are inserted at `Aware` in `LearnerModel.concepts`) and then comprehension (so `apply_comprehension` can promote depths via monotonic max **and** advance `box_level` via `vocab::apply_box_transition`). Combined timeout is `extractor_settings.blocking_timeout + comprehension_settings.blocking_timeout` (default 5000 + 5000 = 10 000 ms); detach-on-timeout so DB persistence still completes. The 10s budget covers even small-model two-call latency in practice; the natural inter-turn pause for a child reading + thinking comfortably absorbs it.

  `build_turn_prompt` calls `primer_core::vocab::due_concepts` over the in-memory `LearnerModel.concepts` to pick up to `vocab_max_per_prompt` overdue concepts and injects them as a passive vocab-review section in the system prompt — the LLM weaves them in only if topically relevant; no drilling, no quizzing. The `build_system_prompt_with_pack_and_vocab` builder takes the due-vocab slice alongside the summary and retrieved-older turns; an empty slice omits the section.
- **`primer-cli`** — the REPL binary (binary name `primer`, not `primer-cli`).

`DialogueManager` holds `&dyn InferenceBackend` and `&dyn KnowledgeBase` (borrowed references, not owned), so swapping backends is a CLI/config decision and the engine is testable with stubs.

## Conventions and gotchas worth knowing

- **The workspace root is `src/Cargo.toml`, not the repo root.** All workspace deps are pinned there via `[workspace.dependencies]`; per-crate `Cargo.toml` files use `.workspace = true`.
- The binary is named `primer` (see `[[bin]]` in `primer-cli/Cargo.toml`), so `cargo run --bin primer`.
- **Real-backend streaming is live.** `CloudBackend::generate_stream` consumes Anthropic SSE (`event:`/`data:` framing, hand-rolled `SseBuffer` + `parse_anthropic_event`). `OllamaBackend::generate_stream` consumes NDJSON (one JSON object per `\n`-terminated line, `NdjsonBuffer` + `parse_ollama_line`). Both forward chunks via a `futures::channel::mpsc::unbounded` driven by a spawned tokio task. The stub backend still emits one final chunk by design — that's a degenerate but valid stream and shouldn't be "fixed".
- **`CloudBackend` currently maps `Role::System` to `"user"` in the messages array** (system instructions are a top-level field on the Anthropic API). `OllamaBackend` instead prepends a `system`-role message because Ollama's chat API has no separate system field. `OpenAiCompatBackend` also uses a `"system"` role message (same as Ollama). Watch this divergence when reworking prompt assembly.
- **`OpenAiCompatBackend` speaks the OpenAI chat-completions SSE format (`/v1/chat/completions`).** SSE parsing uses `OpenAiSseBuffer` which yields `data:` payloads and ignores comment/event/id lines. `data: [DONE]` terminates the stream. `finish_reason != null` also terminates. The SSE format is structurally different from Anthropic's (which uses `event:` + `data:` pairs with typed event names like `content_block_delta`); the parsers are deliberately separate. `classify_openai_compat_status` handles error classification (401/403 → Auth, 429 → RateLimited, 404+model → ModelNotFound, 5xx → ServiceUnavailable). Optional `Authorization: Bearer` header for remote providers. Retry wraps pre-stream phase only, same as Cloud and Ollama.
- **`OpenAiCompatEmbedder` sends the full batch in one request** via the standard `/v1/embeddings` endpoint (unlike `OllamaEmbedder` which loops one-at-a-time). Feature-gated behind `openai-compat` in `primer-embedding/Cargo.toml`. Results are sorted by `index` to match input order (the spec allows out-of-order responses). Dim-mismatch is a hard error, same as `OllamaEmbedder`.
- **`DialogueManager` exposes both `respond_to` and `respond_to_streaming(input, FnMut(&str))`.** The non-streaming method is now a thin wrapper around the streaming one with a no-op callback. On a mid-stream error, the partial Primer turn is **not** recorded into the session — the child turn stays, the Primer turn is dropped, and the error is returned.
- **Cloud model defaults to `claude-sonnet-4-6`** but is now overridable via `--model`. For ollama, `--model` is required (e.g., `--model llama3.2`).
- **`update_learner_model` in `dialogue_manager/learner_update.rs` is a deliberate placeholder** — it only does a crude word-count heuristic for `EngagementState`. Concept extraction has shipped (`primer-extractor`), comprehension classification has shipped (`primer-comprehension`, with monotonic-max depth promotion via `apply_comprehension`), vocabulary spaced repetition has shipped (`primer-core::vocab` Leitner-box scheduler), and session-time break suggestions have shipped (`PedagogicalIntent::SuggestBreak` + `primer_core::session_timing`). The remaining placeholder here is the engagement-state heuristic itself (crude word-count, not classifier-driven).
- **`dialogue_manager` is a directory module.** [src/crates/primer-pedagogy/src/dialogue_manager/](src/crates/primer-pedagogy/src/dialogue_manager/) — `mod.rs` holds the struct + type aliases + glue; per-axis files split the impl block: `lifecycle.rs` (new / open / resume / close / accessors), `turn.rs` (respond_to_streaming orchestrator + 7 private helpers), `background.rs` (await / drain / apply_*_outcome), `retrieval.rs` (retrieve_knowledge / retrieve_long_term_memory), `summary.rs` (refresh_summary_*), `learner_update.rs` (placeholder heuristic), `apply.rs` (pure free fns + their unit tests). Tests split per axis: `tests/{lifecycle_tests,turn_tests,background_tests}.rs` plus shared `test_support.rs` (mocks + builders). When adding a new method to `DialogueManager`, place it in the file matching its responsibility and use `pub(super)` visibility for cross-module access.
- **`decide_intent()` is the brain of the Socratic behaviour.** It is the most important function to test rigorously when adding pedagogical features. Characterization tests are in `prompt_builder::tests` (18 tests pinning current behaviour, including factual-question routing and the `Encouragement` / `SessionClose` split for disengagement); add to them when you change intent routing.
- Knowledge base default path is `:memory:`, but `auto_seed_if_empty` runs unconditionally on startup and loads every `*.<pack>.jsonl` it discovers (the in-repo CC0 hand-drafted seed plus the CC-BY-SA-3.0 Wikipedia layer = 91 passages today for `en`). So a fresh CLI run starts with a populated in-memory KB, not an empty one. If discovery finds no seed files the KB stays empty and retrieval just returns no passages — the prompt builder gracefully omits the knowledge section in that case. The discovery search order is `$PRIMER_SEED_DIR` → `$XDG_DATA_HOME/primer/seed/` → walking up from `CARGO_MANIFEST_DIR`.
- Errors: use `primer_core::error::PrimerError` (variants per subsystem) and the `Result<T>` alias. Wrap external errors via `.map_err(|e| PrimerError::Inference(format!("...: {e}")))`-style.
- **`PrimerError::Inference` carries a typed `InferenceError`.** Six variants: `Auth`, `RateLimited { retry_after: Option<Duration> }`, `ServiceUnavailable`, `NetworkUnavailable`, `ModelNotFound { model }`, `Other(String)`. Variant fields are locale-neutral data; user-facing rendering is `primer_core::i18n::render_inference_error(err, locale)`. `From<&str>` and `From<String>` impls let producers say `format!(...).into()` for unmapped conditions; the result lands in `Other`. **`Other`'s inner string is dev-facing only — it never reaches users**; the i18n layer substitutes a generic message and the call site (CLI) logs the full error via `tracing::warn!`.
- **Retries happen at the request-send + status-check phase only.** Once `generate_stream` starts consuming bytes from a 2xx response, mid-stream errors propagate cleanly and the partial Primer turn drops at the dialogue-manager layer (existing behaviour). Both cloud and Ollama wrap the pre-stream phase in `primer_core::retry::retry_with_backoff`. The retry helper is HTTP-neutral (lives in `primer-core`, no `reqwest` dep) — backends translate their own status codes via `classify_anthropic_status` / `classify_ollama_status` before invoking it.
- **Retry budget defaults are tuned for conversational latency.** 3 attempts × ~250 ms × 2 ≈ ~1.75 s worst case before failure. `Retry-After` headers ≤ 5 s are honored exactly; longer waits surface immediately so children aren't left staring at a 30-second silent gap. Defaults in `primer_core::consts::retry`. Each retry decision emits `tracing::info!(target: "primer::retry", attempt, kind, delay_ms)` — a future voice-mode change can subscribe and play a "give me a moment" bridging phrase.
- **`primer_core::i18n::render_inference_error` is the single i18n boundary** for the inference-error surface. Phase 0.1 ships English-only via `Locale::English`. Adding locales is purely additive — no producer touches anything when the i18n PR lands. **Discipline:** never embed user-facing English in `InferenceError` variant fields (no `hint: String`); never render `Other`'s inner string to users. The `.map_err` wrapping site (formerly at `dialogue_manager.rs:534`, now in the inference-call path of `dialogue_manager/turn.rs::stream_inference_response`) that re-wrapped typed errors as `Other` was removed during the Phase 0.1 work — DO NOT reintroduce a similar wrap when adding new error-producing call sites; just propagate via `?`. See [docs/background_research/i18n_design.md](docs/background_research/i18n_design.md) for the long-form i18n vision.
- **Categorical text columns are normalised into lookup tables and stored as integer foreign keys.** Applies to every SQLite schema in this codebase (sessions, learner-model concepts via `learner_concepts` since schema v4, etc.). For closed Rust enums (`Speaker`, `PedagogicalIntent`, `UnderstandingDepth`), the Rust enum is the single source of truth; the DB lookup table is a derived projection that the storage layer regenerates and validates against on every `open()`. Drift (mismatched name on a known id, unknown id) is a hard error — there is **no** API exposed for mutating lookup tables. Adding a new variant means: edit the Rust source, recompile, next `open()` seeds the new row. Retired integer IDs are never reused.
- **Schema migrations are idempotent column-add patterns wrapped in a single transaction.** Each `apply_vN_migrations` checks `pragma_table_info` before each ALTER and uses `CREATE VIRTUAL TABLE IF NOT EXISTS` / `CREATE TRIGGER IF NOT EXISTS`. Bumping `USER_VERSION` is one half of a migration; the other half is making the new objects safe to re-create on a re-open. The whole body runs inside `conn.unchecked_transaction()` so a partial failure (e.g. disk-full between the FTS create and a trigger create) rolls back to the pre-migration state. Reject `existing_version > USER_VERSION` (newer-than-this-build); silently update old `existing_version != USER_VERSION` to current after running migrations. **Adding a new schema version** = bump `USER_VERSION` in `schema.rs`, add `apply_vN_migrations` following the same pattern, and call it from the open path (the existing v2..v7 chain is the template).
- **FTS5 query inputs are sanitized.** Child input is fed through `sanitize_fts_phrase`: tokens are stripped of non-alphanumeric chars, FTS5 reserved keywords (`AND`, `OR`, `NOT`, `NEAR`) are dropped, surviving tokens are quoted and OR-joined. Empty result short-circuits the query. This is what allows `retrieve_session_turns` to take raw child input safely. BM25 ranking + `LIMIT k` keeps the OR-join from producing noise.
- **Long-term memory is a system-prompt concern, not a chat-message concern.** The chat `messages` array passed to inference is exactly `session.recent_turns(context_window_turns)`. Summary and retrieved-older turns are injected into the system prompt. This keeps the timeline the model sees linear and the context budget bounded even across hours of conversation.
- **Sessions persist by default.** Without `--session-db`, the CLI defaults to `~/.primer/<slugified-name>.db`. The directory is created on first run; an unwritable home is a hard error rather than a silent in-memory fallback. The first time the default-path file is created the CLI prints a one-line banner explaining where the data lives and how to opt out. `--resume <uuid>` requires the file to exist (no auto-create on a typo) and errors clearly when the id isn't present. `--no-persist` skips disk entirely (in-memory `:memory:` store); it conflicts with `--session-db` and `--resume` at parse time.
- **Slugged learner names are NFC-normalized and Unicode-aware.** `slug()` in `primer-cli` accepts any character Unicode classifies as alphanumeric (Latin, Cyrillic, CJK, etc.), lowercased where case applies. The input is NFC-normalized first so `José` (precomposed `é`) and `José` (decomposed `e` + combining acute) map to the same filename — otherwise two copies of the same child get two session DBs. Filesystems on macOS/Linux handle Unicode filenames natively; this is the right tradeoff for a children's product whose audience explicitly includes non-English names.
- **The engagement classifier is one model behind the chat.** Classification of turn N happens after the Primer's response to turn N is complete; its result is applied at the start of turn N+1's intent decision. This is a deliberate latency tradeoff for memory-constrained-device support and is invisible to children but worth knowing when reasoning about why a misclassified turn doesn't immediately yank the intent.
- **Concept extraction lags one turn for in-memory state.** The extractor task spawned at the end of turn N writes to `turn_concepts` directly, but the in-memory `LearnerModel.concepts` is only updated when the next turn starts (the same await-with-bounded-timeout pattern as the classifier). This means the system prompt for turn N does not include concepts surfaced *in* turn N — only those from earlier turns. Acceptable for v1; if it ever surprises a child ("I just told you about photosynthesis and you're acting like I didn't"), the fix is to apply the extraction synchronously before save_session.
- **Comprehension assessments lag one turn just like concepts.** `apply_comprehension` runs in the same `await_pending_post_response` step as `apply_extraction`, so depth promotions for turn N's exchange are visible to the system prompt at turn N+1, not turn N. Same combined timeout applies.
- **`apply_comprehension` is monotonic-max.** A weak exchange never demotes `concept.depth`; explicit forgetting is a Phase 0.3+ concern handled algorithmically over `turn_comprehensions`, not via another LLM call.
- **Per-concept comprehension assessments are persisted even when sub-threshold.** `confidence_threshold` (default 0.6) only gates the in-memory apply step; the longitudinal `turn_comprehensions` record gets every assessment, with confidence scores intact for offline filtering.
- **`extract_first_json_object` and `truncate_to_chars` live in `primer-core::llm_util`.** All three structured-output crates (`primer-classifier`, `primer-extractor`, `primer-comprehension`) import from there. Don't re-introduce per-crate copies.
- **`update_turn_concepts` exists because `save_session` skips already-persisted turns.** The append-only invariant on `save_session` (skip turns already on disk by index) means concept additions to a previously-saved turn would be silently dropped. `update_turn_concepts` is the explicit backfill path for the post-response extractor task. Do not move concept persistence back into `save_session` — that would require changing the skip rule and breaking the append-only invariant.
- **`InferenceBackend::generate(prompt, params)` does not take a model parameter.** Each backend instance owns its model name at construction. So `--classifier-model X` (without `--classifier-backend Y`) constructs a *separate* backend instance of the same type with model X — it does not reuse the main backend with a different model on a per-call basis.
- **`Named` is the single source of `name()` across every speech trait** (`VoiceActivityDetector`, `SpeechToText`, `StreamingSpeechToText`, `TextToSpeech`, `StreamingTextToSpeech`). Adding a new speech trait means inheriting from `Named` so backends never have to write `name()` twice.
- **`piper-rs 0.1.9` is vendored at `src/vendor/piper-rs/`.** The upstream crate was built against pre-`rc.10` `ort`; the rest of the workspace pins `ort = "=2.0.0-rc.10"` to keep `silero-vad-rust 6.2`'s `ndarray` consistent. The vendored copy carries a small patch fixing `Value::from_array` (now wants owned arrays), `Session::run` (now `&mut self`), and `try_extract_tensor` (returns `(&Shape, &[T])` now). `[patch.crates-io] piper-rs = { path = "vendor/piper-rs" }` in `src/Cargo.toml` makes it the resolved version. Until upstream releases an `rc.10`-compatible version, this fork stays.
- **`--speech` mode is gated by the `speech` feature on `primer-cli`** which pulls all four `primer-speech` features (`silero`, `whisper`, `piper`, `cpal`). Default builds stay light. Build with `~/.cargo/bin/cargo build --features primer-cli/speech`. Then pass `--speech --whisper-model <path> --voice-onnx <path.onnx> --voice-config <path.onnx.json>` at runtime. `clap` enforces that model paths travel with `--speech`. **On the macOS-native build (`--features primer-cli/speech,primer-cli/macos-native` on macOS) the `--whisper-model`, `--voice-onnx`, `--voice-config`, and `--voice` flags are not declared at all** — SFSpeechRecognizer + AVSpeechSynthesizer replace whisper + piper, so those assets are never read. `SpeechLoopConfig`'s four asset fields are likewise cfg-gated under the same `cfg(all(target_os = "macos", feature = "macos-native"))` and the config struct drops its lifetime parameter (owned `PathBuf` / `String` rather than borrowed slices, to avoid a `PhantomData<&'a ()>` workaround when the references would otherwise be cfg-gated out). The `--mic-silence-ms` flag stays on both speech builds because Silero remains the VAD on macOS-native too. Closes #112.
- **Voice mode in the GUI is gated by `primer-gui/speech`** which pulls all four `primer-speech` building-block features plus `primer-speech/voice-loop` (the shared state machine) plus `reqwest` (for asset auto-download). Default GUI builds skip cpal entirely. Asset cache lives under `~/.cache/primer/models/` with sub-dirs `voice/<locale>/` and `whisper/`. First-run is consent-gated via the `download_voice_assets` Tauri command + `voice-consent-modal`; the `SpeechSettings.disable_auto_download` flag in `gui-config.json` honours [[project_strict_offline_first]]. Locale-default voice/STT mapping lives in `primer_speech::voice_loop::locale_defaults` — `en` → `en_GB-alba-medium` + Whisper `small.en`, `de` → `de_DE-thorsten-medium` + Whisper `small` (multilingual). Sticky toggle persisted to `gui-config.json` as `speech.voice_mode_enabled` is per-device, not per-learner (matches [[project_personal_device_model]]). Five Tauri commands (`start_voice_mode`, `stop_voice_mode`, `cancel_voice_response`, `download_voice_assets`, `get_voice_state_copy`) and seven events (`primer://voice/{state_change,transcript,response_chunk,response_complete,exit,inference_error,download_progress}`) live in `primer-gui/src/commands/voice.rs`. The frontend controller `ui/voice.js` owns header-toggle clicks, composer swap, Esc-key cancel, and consent-modal flow.
- **`download_voice_assets` IPC takes only `kinds: Vec<String>` — not paths or URLs.** The frontend echoes back only the `kind` strings (`"piper_onnx"` / `"piper_config"` / `"whisper_model"`) from the original `AssetMissing.entries`; the server re-resolves `path` + `suggested_url` via `voice::assets::resolve_requested_kinds`. This keeps the IPC trust boundary tight: a compromised webview cannot direct the host to write outside `~/.cache/primer/models/` or to fetch from a non-canonical URL, because those fields never cross the trust boundary as input. `MissingAsset` is deliberately `Serialize`-only (not `Deserialize`) — if you ever need to round-trip it back through the webview, do not re-add `Deserialize`; add a new echoed-identity type instead. Closes #90.
- **`download_voice_assets` is hardened for timeout, resume, and oversize.** The Tauri wrapper [`primer-gui/src/voice/download.rs::download_one`] builds a `reqwest::Client` with `.timeout(SpeechSettings.download_timeout_secs)` (default 30 min, configurable in `gui-config.json`). When `<dest>.partial` already has bytes on disk, the next attempt sends `Range: bytes=N-`; the server's 206 reply appends, a 200 reply (server ignored the header) truncates and restarts. A 416 reply clears the stale partial so the next attempt starts fresh. Bytes are capped at `approx_size_mb × 150 % × 1 MiB` via `compute_max_bytes`; an oversized stream aborts and the (now-tainted) partial is deleted. **Partial-file policy on failure: only Oversize and 416 clean up the partial; Timeout, Network drop, transient HttpStatus, and I/O errors preserve it so the next click resumes.** This is a deliberate change from the original PR #89 behaviour (which always wiped the partial and thereby defeated resume). On any failure, one final `primer://voice/download_progress` event fires with `error: Some(<kind>)` (`"timeout" | "oversize" | "http_status" | "network" | "io" | "no_url"`); the field is `#[serde(skip_serializing_if = "Option::is_none")]` so the happy-path JSON shape is unchanged. The async core `stream_to_path` is tested by an in-process `tokio::net::TcpListener` server (no external mock dep). Closes #92.
- **`VoiceStateCopy` display strings live in the prompt pack, not in Rust.** The six display strings the `get_voice_state_copy` Tauri command serves (`listen_label/_hint`, `thinking_label/_hint`, `speak_label/_hint`) come from the `[voice_state]` table of each `primer-pedagogy/prompts/<pack>.toml`, surfaced via `PromptPack::voice_state_labels() -> &VoiceStateLabels`. Adding a new locale means adding the table to the new pack — no Rust change needed. The GUI bridge in `primer-gui/src/commands/voice.rs::VoiceStateCopy::for_locale` is a thin clone over the pack accessor, mirroring how `child_label`/`primer_label` work. Empty values in any `[voice_state]` field are a pack-shape error caught at load time via `validate_voice_state_section` so consumers render the strings unconditionally without `Option<&str>` plumbing. Closes #91.
- **Build with rustup, not Homebrew rust.** `rust-toolchain.toml` pins Rust 1.87+ but this pin is honored only by rustup proxy binaries. If Homebrew rust shadows on `PATH`, `cargo` silently uses Homebrew's 1.86 and silero will fail to compile. Always invoke as `~/.cargo/bin/cargo` for primer commands.
- **Builds cleanly on Android ARM64 via Termux** — validated 2026-05-26 on RedMagic 11 Pro for the default-features Phase 0 text REPL (cloud backend; on-device Ollama at 4B Q4 on CPU is functionally correct but too slow for conversational use — the standalone-phone product story is gated on Phase 1.2 `QnnBackend`). Quickstart at [docs/devel/redmagic-termux-quickstart.md](docs/devel/redmagic-termux-quickstart.md). Termux's `pkg install rust` may lag the workspace's 1.88 pin — fall back to `pkg install rustup; rustup default stable` if so. **`/tmp` is not writable in Termux's Android sandbox** — use `$TMPDIR` or `~/` for any `tee`/`mktemp`/log-file write. **`--resume <uuid>` reads from `--session-db`** which defaults to a name-slug-derived path; if you change `--name` (or drop it) on the resume command the CLI looks in the wrong DB file — pass the same `--name` consistently or pass `--session-db` explicitly. **Reasoning-mode Ollama models (DeepSeek-R1, Gemma-thinking, Qwen QwQ, medgemma1.5, etc.) leak chain-of-thought tokens** verbatim into the visible response because `OllamaBackend` does not currently strip the per-model reasoning markers (`<think>…</think>`, `<unused94>thought`, …); proper stream-aware handling is a follow-up.
- **Cross-compile to `aarch64-linux-android` works clean on macOS** — `Finished dev profile in 1m 27s` on a stock Mac with NDK 29 + workspace's pinned 1.88 toolchain. The whole `primer` binary dep graph (12 crates incl. `rusqlite` bundled-SQLite, `reqwest` rustls, `tokio`, `chrono`, default-features `primer-speech`) cross-compiles without any `Cargo.toml` changes. Use `cargo build --target aarch64-linux-android --bin primer` (NOT `--workspace --bin primer` — `primer-gui` is Tauri/webkit2gtk and explicitly out of Android scope). Add the target to the pinned toolchain via `rustup target add aarch64-linux-android --toolchain 1.88` (the default-toolchain target install is not sufficient). The CI workflow at `.github/workflows/ci.yml::android-cross-compile` enforces this as a drift-guard on every push and PR.
- **Hybrid retrieval (`--embedder-backend fastembed`) does NOT build on Android ARM64** as of 2026-05-27. The blocker is upstream `ort-sys 2.0.0-rc.10`: its `build.rs:26` imports `internal::dirs::cache_dir` which is `#[cfg]`-gated to `target_os = "windows" | "linux" | "macos"` only. There is no `target_os = "android"` arm, so the import fails to resolve on Termux. The workspace's pin to `ort = "=2.0.0-rc.10"` is load-bearing (silero/whisper/piper vendor patches require it — see the vendored-crate notes below); bumping ORT requires a lockstep upgrade of those three vendored crates' patches. Unblocking paths: (a) upstream PR to `pyke/ort` adding the Android arm (the Linux arm's `XDG_CACHE_HOME` fallback would likely work as-is); (b) workspace `[patch.crates-io] ort-sys = { path = ... }`; (c) wait for an ORT rc with native Android support and do the lockstep vendor-patch upgrade. **The recommended Android setting today is `--embedder-backend none`** (BM25-only); the retrieval-quality benchmark suite passes at 100% strict recall on the 91-query English benchmark under BM25-only defaults, so this is a graceful fallback, not a feature regression. See `docs/devel/redmagic-termux-quickstart.md` for the full probe write-up.

Drop the vendor patch once upstream catches up.
- **`primer-speech`'s ort dep** explicitly lists `download-binaries` and `copy-dylibs` in its ort features (with `default-features = false`; without this explicit list those features are stripped and ONNX Runtime cannot be found at link time). The first build downloads the ONNX Runtime binary from cdn.pyke.io; subsequent builds use the cached copy.
- **`espeak-ng` must be installed system-wide** for `--speech` (CLI) or voice mode (GUI) to produce phonemes. On macOS: `brew install espeak-ng`. On Debian/Ubuntu: `apt install espeak-ng-data`. Both binaries probe `/opt/homebrew/share`, `/usr/local/share`, and `/usr/share` at startup and set `PIPER_ESPEAKNG_DATA_DIRECTORY` to the parent of the first complete `espeak-ng-data` directory found — CLI in `primer-cli/src/main.rs::probe_espeak_ng_data` (before tokio runtime starts), GUI in `primer-gui/src/lib.rs::probe_espeak_ng_data` (before the Tauri builder spawns workers). `espeak-rs` ships an incomplete subset that causes phoneme failures for most voices. The two probe functions are byte-identical except for logging (CLI uses stderr, GUI uses `tracing::info!`); when refactoring, move the shared implementation to `primer-speech` rather than diverging.
- **Whisper transcription language must be explicitly set from the learner's locale** in `voice_loop::backends::build_local_backends`. `WhisperStt::new(path)` defaults `language` to `"en"`; without `.with_language(locale.pack_id())`, the multilingual `ggml-small.bin` model (used for `de`) is forced into English transcription mode, German audio comes out as approximate English, and the LLM responds in English on a German-locale session. `Locale::pack_id()` returns ISO-639-1 — exactly the form Whisper accepts. Pinned in `whisper::tests::pack_id_is_iso_639_1_for_whisper`.
- **`WhisperStt` reuses a single `WhisperStream` across utterances via a single-slot `StreamCache`** (closes #133). The first `open_session` constructs a fresh stream (≈500 ms wallclock for KV-cache + GPU compute buffer allocation on Apple Silicon); every subsequent `open_session` `take()`s the cached stream and calls `WhisperStream::reset()` (which clears `audio_buf`, `pcmf32_old`, `prompt_tokens`, `n_iter`, `total_samples_processed` while leaving the KV cache intact — that's the load-bearing point). **Re-cache policy is happy-path-only**: `WhisperSession::finalize` re-caches on a successful `flush()`, but **discards on flush error** (and emits `tracing::warn!(target: "primer::speech::whisper", ...)`) — `reset()` does not touch `WhisperState`, so a flush-failed stream may carry decoder state we cannot characterise, and paying one cold-start tax to recover is safer than silently biasing the next utterance with stale state. The `Drop` impl follows the same defensive policy: if `finalize` was never called (push_audio error, caller panic, early return) the dropped stream is discarded with a `tracing::warn!`. The cache is single-slot by design — the voice loop runs one VAD-driven session at a time per backend, so contention is impossible by construction; a racing second `open_session` falls back to a fresh construction, which is functionally identical to the cold-start. Mutex poisoning in the cache is non-fatal: `take()` returns `None` and `put()` drops the value, both with `tracing::warn!`, so the (otherwise invisible) loss of the perf optimisation is debuggable. The cache logic is unit-tested against `StreamCache<i32>` in `whisper::stream_cache_tests` (constructing a real `WhisperStream` needs a GGML model on disk; the multi-utterance correctness smoke is tracked as a follow-up at #166). **Sidesteps `macos-native-26` paths** — SpeechAnalyzer has a different shape and no KV-cache state to reuse. The non-Apple paths still benefit on every cloud + Linux build that uses Whisper.
- **Voice id must be explicitly wired.** `LoopBackends` has a `voice: VoiceProfile` field; production wires `model_id` from `cfg.voice_id` (the `--voice` flag). Before this fix, `run_loop` hardcoded `VoiceProfile::default()` (= en_US-amy-medium) and Piper rejected any non-default voice with a model-id mismatch. Always pass `--voice <model-id>` matching your `.onnx` filename.
- **The voice loop never lets the Primer speak over a child** (LATENT_THINK keeps the mic open across the LLM call; cancel-on-SpeechStart aborts the LLM if the child resumes). It also **never lets the child speak over the Primer** — an `Arc<AtomicBool>` `is_speaking` flag gates the audio thread during SPEAK: `run_loop` sets it true before SPEAK; after the flush sentinel returns it `await`s a `DrainHook` — a `tokio::task::spawn_blocking` wrapper around `primer_speech::wait_for_drain` — that polls the speaker ringbuf's `occupied_len` until it reads zero on 3 consecutive 10 ms checks (then waits an 80 ms grace for cpal's own output buffer, max 5 s sanity cap, with the grace included in the cap so total wallclock is bounded); only then does `run_loop` clear the flag. Going through `spawn_blocking` is what keeps the synchronous wait off the tokio worker — calling `std::thread::sleep` directly inside the on_audio callback (which runs in the same task as `run_loop`) would block other tasks for up to 5 s and panic outright on a single-threaded runtime. The audio thread checks the flag every 5ms and while true drains-and-discards mic samples, drops any active whisper session, and clears local buffers. Without this, the Primer's TTS leaked back through the mic and caused endless self-conversation. The speaker producer is wrapped in `Arc<Mutex<HeapProd<f32>>>` so both the on_audio push path and the drain hook can access it; the mutex is uncontended (push and observe never overlap). Both invariants are tested via the mock-driven `speech_loop` tests in `primer-cli`.
- **Speaker ringbuf is 240,000 samples** (~5 s at 48 kHz). The previous 24,000-sample (~500 ms) buffer dropped samples when on_audio produced faster than cpal drained — cutting off all but the first phrase of a multi-phrase response. Never shrink below ~5 s worth of samples.
- **Markdown is stripped before TTS.** `strip_markdown_for_tts` removes paired `*emphasis*`, `**strong**`, and `` `code` `` markers before pushing to Piper; bare unmatched markers stay; a `*` (or run of `**`) flanked by digits is replaced with " times " so multiplication and exponent notation read aloud (e.g. `5*3=15` → "5 times 3=15"). The visible transcript on stdout retains markdown.
- **The audio capture path for `--speech` runs on a dedicated `std::thread`**, not a tokio task — it owns the silero VAD and the active whisper streaming session and emits `VadEvent`s plus, on `SpeechEnd`, the finalized transcript through a channel that the `run_loop` consumes via a `ChannelStt` wrapper. This keeps the test/production separation clean (mocks bypass cpal entirely).
- **Resampler tail-handling is non-trivial.** rubato's `FftFixedIn` buffers FFT state internally. `on_audio` carries a `leftover: Vec<f32>` buffer across calls so phrase tails are prepended to the next call rather than zero-padded mid-stream; at end-of-turn, an empty-Vec flush sentinel zero-pads any leftover then drives 4 silence chunks (~186 ms input silence) through the resampler to drain its FFT-buffered output. Without both, the last syllable/word of each phrase and often the entire last phrase were silently discarded.
- **The on_audio push loop bails on cpal output stream errors.** When synthesis outruns cpal drain, the loop retries (5 ms sleep between attempts) via `primer_speech::push_all_with_bail` until cpal's ringbuf accepts the samples. If the cpal output stream errors (device removed, host audio service crashed), an `Arc<AtomicBool>` flag plumbed from `cpal::err_callback` to `SpeakerSink::errored_flag()` flips and the loop bails with a `tracing::warn!` about the discarded sample count — instead of spinning forever on a `push_slice` that returns 0. The earlier 1-second drop-on-timeout was the actual mechanism cutting off the end of long Primer responses.
- **Voice rate is 0.9 (length_scale ≈ 1.111) and inter-phrase silence is 200 ms.** Slightly slower than Piper's default; 200 ms of silence inserted between `AudioChunk`s (each chunk = one phrase). Adjust `VoiceProfile.rate` and the inter-chunk sleep in `speech_loop` if the voice feels too slow or too breathless.
- **Vocabulary spaced repetition is driven entirely by the comprehension classifier — no extra LLM call.** `primer_core::vocab::apply_box_transition(box_level, depth, confidence)` advances the box on a strong above-threshold reading, resets it on `Aware` or sub-confidence, and pins at `MAX_BOX_LEVEL` (= 4, 30-day interval). `due_concepts(concepts, now, max_n)` returns the top-N most-overdue `ConceptState`s using the box-interval table `[1d, 3d, 7d, 14d, 30d]`. The dialogue manager calls this in `build_turn_prompt` and feeds the slice into the prompt builder's vocab section. The LLM is told to weave words in only if topically relevant — never drill, never quiz. **Vocabulary entries are `ConceptState` rows, not a separate table** — keeping the schema flat preserves the Phase 4 anonymisation/contribution path the roadmap calls out.
- **Locale is per-learner and persisted.** `--language <pack_id>` sets `Locale` for the session; on first save it's written to `learners.locale` (schema v6). On `--resume`, if the CLI's `--language` differs from the resumed learner's stored locale the CLI errors and explains the two ways to resolve (drop `--language` or pass `--language <stored>`). **The same discipline applies to the GUI's start-new-session path:** `wiring::build_active_session` errors when `cfg.learner.locale` differs from a persisted learner's `profile.locale`. Without this guard, KB/STT/TTS run under the cfg locale (because they read `cfg.learner.locale` directly) while the dialogue manager's prompt pack stays under the persisted locale (because it reads `learner.profile.locale`) — the LLM responds in the persisted language even when the user has just changed Settings → Locale, with KB and voice all consistent with the new locale. User-facing resolution: either revert Settings → Locale to the persisted value, or remove `~/.primer/<slug>.db` to start fresh under the new locale. The reason for hard-erroring rather than silently re-tagging is that `concepts.concept_language_tag` would otherwise start tagging new rows with the wrong locale, corrupting the longitudinal vocabulary record. The knowledge base is keyed by locale via `SqliteKnowledgeBase::open_for_locale` so passages stay BM25-pure per language.
- **The `LearnerModel` is an umbrella struct, not a leaf.** It contains `LearnerProfile` (name, age, native language, interests), `Vec<ConceptState>` (the longitudinal mastery record), `LearningPreferences` (per-child tunables, e.g. `early_disengagement_threshold`), and the latest engagement snapshot. When persisting or extending, edit the leaf type that owns the field — the umbrella struct is just composition. `LearningPreferences` is open-vocabulary (children's interests don't fit a closed enum) so it serialises as JSON, while categorical leaves (`UnderstandingDepth`, `EngagementState`) are closed enums backed by lookup tables.
- **`PedagogyConfig.max_session_minutes` was renamed to `break_suggest_after_minutes`.** It controls the cadence of break-suggestion nudges, not a hard session ceiling. The Primer never enforces a session halt — children can keep going through any number of nudges.
- **Break-suggestion gate state is in-memory.** `DialogueManager.last_break_suggested_at: Option<DateTime<Utc>>` resets on `new` and `resume_session`. A resumed session might cause one extra suggestion if timing aligns badly; that is intentional (see spec at [docs/superpowers/specs/2026-05-05-session-break-suggestion-design.md](docs/superpowers/specs/2026-05-05-session-break-suggestion-design.md)).
- **`PedagogicalIntent::SuggestBreak` (id=9) is wallclock-driven.** `decide_intent_at_with_pack` consults `primer_core::session_timing::should_suggest_break_now` after engagement-state overrides; engagement state wins (a frustrated child past the 30-minute mark gets `Scaffolding`, not `SuggestBreak`). Because `SuggestBreak` is a `PedagogicalIntent` variant it is automatically seeded into the `pedagogical_intents` lookup table on the next DB open — no migration step required.
- **Locale-aware `{minutes}` interpolation in the system prompt.** Each locale's TOML template owns its own unit word (`minutes` / `Minuten`); `pack.break_suggestion_intro(minutes)` renders the substituted string. Pattern to reuse for future locale-aware durations — the template key is `break_suggestion_intro` and the substitution token is `{minutes}`.
- **Prompt packs carry a `[meta] status` field with allow-list `["stable", "preview"]` (default `"stable"` when absent).** The closed `PackStatus` enum exposes this to consumers via `PromptPack::status()`. `load_cached` emits a one-time `tracing::warn!` per `(process, locale)` pair on `Preview` packs; direct `load()` calls do not (the translator-iteration / `PRIMER_PROMPTS_DIR` path is unmuted). **`Locale::Hindi` is shipped as the first preview locale and is deliberately excluded from `Locale::ALL`** so CLI/GUI pickers don't surface it to end users — a developer can still pass `--language hi` explicitly. Flipping a preview to stable means editing both the TOML (`status = "stable"`) and the `Locale::ALL` slice (`+ Self::Hindi`) in one commit. See [docs/localisation/hi/README.md](docs/localisation/hi/README.md) for the preview checklist.
- **Hybrid retrieval is opt-in via the `embedding` cargo feature AND the `--embedder-backend` flag.** Two layers: (1) compile-time, the `embedding` cargo feature decides whether `FastEmbedBackend` is built at all; (2) runtime, `--embedder-backend none|stub|fastembed|ollama` (default `none`) decides which embedder the dialogue manager uses. `none` is BM25-only — exactly the pre-Phase-0.2.5 behaviour. `stub` constructs a deterministic FNV-hash embedder useful only for testing the hybrid pipeline structurally; in production it dilutes BM25 with semantic noise and should not be the default. The user's stated preference is for hybrid retrieval default-on once CI validates the cdn.pyke.io ort-runtime download; flip both `default = ["embedding"]` in `primer-cli/Cargo.toml` and the `--embedder-backend` default at that point. Without the feature, an explicit `--embedder-backend fastembed` exits with a build hint.
- **`embedding` and `speech` cargo features both pin `ort = "=2.0.0-rc.10"`** so `--features embedding,speech` builds cleanly. `primer-speech`'s vendored silero/whisper/piper pin rc.10; `primer-embedding`'s `fastembed` feature pins `fastembed-rs = "=5.7.0"`, the latest line that *also* uses rc.10 (5.9+ moved to rc.11/rc.12). **Bumping fastembed past 5.7.0 will break the speech build** until the speech vendor patches are upgraded in lockstep. Pinning the workspace ort to exactly rc.10 in two unrelated dep paths is the load-bearing constraint that keeps the two feature stacks compatible.
- **Hybrid retrieval = BM25 leg + vector cosine leg, fused via Reciprocal Rank Fusion.** `primer_core::rrf::fuse(&[ranked_lists], k)` is the only fusion implementation in the codebase. Industry default `k = 60` is the only value used; smaller weighs the very top more, larger flattens. Both `KnowledgeBase::retrieve_hybrid` and `SessionStore::retrieve_session_turns_hybrid` carry a default trait impl that falls back to BM25-only when no `Embedder` is wired — every consumer can call them unconditionally. `HybridParams::default()` reads from `primer_core::consts::retrieval` (the consts are the single source of truth — change them, not the impl); a `default_mirrors_consts`-style drift test pins the alignment.
- **Both BM25-only and hybrid retrieval defaults are tuned against the 91-passage corpus.** BM25-only: `KB_FINAL_TOP_K = 5`, `KB_BM25_ONLY_MIN_SCORE = 0.5` (24-cell sweep at `tests/retrieval_sweep.rs`). Hybrid: `KB_BM25_TOP_K = 30`, `KB_VECTOR_TOP_K = 30`, `KB_FINAL_TOP_K = 5`, `RRF_K = 60` (54-cell sweep at `tests/retrieval_sweep_hybrid.rs` under `--features fastembed --ignored`). The benchmark dataset has **91 queries with 24 strict-subset canonical-id mappings**. After the issue #45 corpus expansion (`seed:en:flowers` added; stomach-growl sentence added to `seed:en:digestion`; `"why do flowers smell nice"` re-targeted from `wiki-simple:en:plant` → `seed:en:flowers`), hybrid achieves **100% loose / 100% strict on all 91 queries / 24 mappings** — the previously skipped 2 paraphrases ("tummy growl", "flowers smell nice") now pass under both BM25-only and hybrid, and serve as permanent regression guarantees. `KNOWN_FAILING_QUERIES_HYBRID` is therefore empty today; new entries there should be investigated before adding (they represent a real semantic regression or a corpus-coverage gap the dense leg can't bridge). The other 2 of the original #45 four ("what is inside a tiny bug" → insects, "why does the brain need oxygen from the lungs" → brain) remain in `KNOWN_FAILING_QUERIES` (BM25-only diagnostic) but pass under hybrid — i.e. they continue to demonstrate the hybrid lift that the BM25-only sweep records as a corpus-coverage gap. Lifting the BM25-only strict miss for "how does the sun shine" was the prior closed example (former issue #42). Hybrid regression test ships in two flavours at `tests/retrieval_quality_hybrid.rs`: a `StubEmbedder` structural sanity check (always built; no recall floor — stub vectors are noise on synonyms) and a real-`fastembed` recall floor under `--features fastembed`.

The regression tests assert pass/fail at production defaults (not a measured percentage). After the issue #64 stress-paraphrase expansion: `KNOWN_FAILING_QUERIES_DE` carries 3 BM25-only misses (`bauch komische geräusche` → verdauung, `gänsehaut wenn mir kalt ist` → haut, `ebbe und flut am meer` → mond) — the BM25 leg picks lexically adjacent passages (energie/schall/saugetiere; energie/klima/temperatur; meer/fluss/wetter) instead of the semantically correct one. `KNOWN_FAILING_QUERIES_DE_HYBRID` carries 2 of those 3 — the gänsehaut and ebbe/flut paraphrases — both genuine corpus-coverage gaps: the Klexikon `haut` article doesn't describe the goosebumps reflex (only skin layers, pigments, Sonnenbrand); the Klexikon `mond` article doesn't discuss tides (only orbit, phases, eclipses). The shared `Cluster` enum is reused but the `Wiki` variant doesn't appear in German — Klexikon IS the wiki for `de`, there's no parallel hand-drafted seed layer; if a German hand-drafted seed is ever added, the cluster-floor test enumerates explicit variants and must be updated.
- **Parallel German retrieval-sweep diagnostics ship at `tests/retrieval_sweep_de.rs` (24-cell BM25) and `tests/retrieval_sweep_hybrid_de.rs` (54-cell hybrid under `--features fastembed`).** Same algorithmic-winner rule as the EN sweeps (max strict_recall → max loose_recall → min top_k → max min_score for BM25; analogue for hybrid). The DE BM25 sweep is sized to five clusters — `Cluster::Wiki` is omitted because Klexikon IS the wiki for `de`. On today's 66-passage Klexikon corpus + 31-query benchmark (post-issue-#64): BM25 strict recall remains 100% on the 22 "easy" canonical queries from `top_k=5` onward, but the 3 stress paraphrases pull aggregate strict recall to ~88% — they are the documented BM25-vs-hybrid-lift demonstration. Production `(top_k=5, min_score=0.5)` is a comfortable choice for the non-paraphrase queries; the paraphrases are excluded from the regression assertion via `KNOWN_FAILING_QUERIES_DE`. Hybrid hits 100% strict on the 23 non-corpus-gap canonical queries (lifts the verdauung paraphrase); the 2 genuine corpus gaps (gänsehaut, ebbe/flut) stay in `KNOWN_FAILING_QUERIES_DE_HYBRID` and are excluded from the hybrid recall assertion. Run with `--ignored sweep_retrieval_params_de --nocapture` (BM25) and `--ignored sweep_retrieval_params_hybrid_de --nocapture` (hybrid, downloads ~570 MB BGE-M3 on first run).
- **Sweep harness implementation is shared across locales at `tests/common/sweep.rs`.** All four per-locale sweep files (`retrieval_sweep{,_de,_hybrid,_hybrid_de}.rs`) are thin `#[ignore]` shims that hand a `Bm25SweepConfig` or `HybridSweepConfig` to `run_bm25_sweep` / `run_hybrid_sweep`. The helper is one source of truth for the grid constants (`BM25_TOP_K_GRID`, `BM25_MIN_SCORE_GRID`, the four `HYBRID_*_GRID`s, and `HYBRID_PREFERRED_RRF_K`), the lex-selection rule (now uniformly `.partial_cmp(...).unwrap_or(Ordering::Equal)` — closes #67), the print format, and the self-validation step. Pre-refactor harness pairs were 95% bit-identical Rust (~1000 lines across 4 files); post-refactor the helper is 557 lines + four ~50-line shims. Byte-identical printed output to the pre-refactor harnesses on the same corpus + benchmark.
- **`KB_BM25_ONLY_MIN_SCORE = 0.5` is a no-op for recall today, but defended by a tripwire test.** The 24-cell sweep showed every *correct* top-K hit comfortably exceeds 1.5; the sub-1.5 scores that exist are 5th-place noise on marginal queries (worst top-1 across the 91-query benchmark is 2.06 after the issue #45 corpus expansion — was 2.11 before; the new `seed:en:flowers` and the expanded `seed:en:digestion` shifted the worst-case query but margin remains comfortably above the 0.5 floor; min over all top-K = 0.95). The floor is kept as defensive insurance against future corpus expansion diluting term frequencies. The `#[ignore]`'d diagnostic at `primer-kb-load/tests/bm25_floor_tripwire.rs` probes the actual top-K score distribution — asserts `min > floor` and `median > floor × 2.0` — and fires loudly when the margin closes. - **Embedder model identity is persisted in a per-DB `embedding_models` lookup table; mismatched dim or unrecorded model is a hard error.** Both `primer-knowledge` (schema v3) and `primer-storage` (schema v8) carry their own `embedding_models` table. The first call that writes a vector under model name X registers `(name=X, dim=N)`; subsequent writes that report a different dim under the same name fail loudly. This is the load-bearing invariant that prevents silent quality regressions when a user re-opens a DB with a different embedder. To deliberately swap models on an existing DB: `primer-kb-load --reembed --force` (KB) — symmetric session-DB tool is a TODO.
- **Embedding-on-save runs in a `tokio::spawn`, fire-and-forget, after `save_session`.** `DialogueManager::spawn_embedding_task` embeds the most-recent (child, primer) turns and calls `SessionStore::save_turn_embedding` for each. Idempotent at the storage layer (upsert by `turn_id`), so re-spawning over already-embedded turns is a no-op write. The just-saved turn may not have a vector yet at the next `retrieve_session_turns_hybrid` call, but that's fine — recent turns are inside the context window so the model already sees them directly; only older turns (where retrieval matters) reliably have vectors.
- **`StubEmbedder` produces hash vectors, not semantic vectors.** Two synonyms produce orthogonal vectors. Workspace tests use the stub to assert *structural* hybrid behaviour (RRF fusion correctness, schema migrations, dim-mismatch handling), never semantic similarity. The semantic-quality test for the real model lives behind `--features fastembed --ignored` and runs only on demand.
- **First-run BGE-M3 download is ~570 MB into `~/.cache/primer/models/` and uses the same `cdn.pyke.io` ONNX-runtime download as the speech feature.** Construction failure falls back to BM25-only with a `tracing::warn!` — the conversation still works, which is strictly better than refusing to start. `--embedder-backend none` skips the download entirely; `--embedder-backend stub` skips the download AND stays in-process but provides no semantic signal.
- **Wikipedia-shaped ingestion lives in `data/ingest/` (Python).** Pure functions where possible; network is the test boundary, injected via `http_client`. The pipeline is parameterised by a frozen `WikiSource` dataclass (`pack_id`, `api_url`, `web_base_url`, `id_prefix`, `human_label`, `license`, `topic_tags`, `disambiguation_patterns`, `fetch_strategy`, `batch_size`) so multiple Wikipedia-shaped sources share one fetch / passage-emit / JSONL-write code path. Two presets ship today: `SIMPLE_ENGLISH` (Phase 0.2; Simple English Wikipedia; `text_extracts` strategy uses `action=query&prop=extracts&exintro=1&explaintext=1` with batch_size=20 = MediaWiki's per-IP cap) and `KLEXIKON` (Phase 0.3+; German children's wiki at `klexikon.zum.de`; `klexikon_wikitext` strategy uses `action=parse&prop=wikitext&section=0` + `strip_klexikon_wikitext` because Klexikon's MediaWiki has no TextExtracts extension; batch_size=1, no batching since `parse` takes one `page` per call). Run as `python3 simple_wikipedia.py --language en` or `--language de`; defaults read whitelist from `simple_wikipedia_whitelist.txt` (en) or `klexikon_whitelist.txt` (de) next to the script and write to `../seed/wiki_passages.<pack>.jsonl`. Re-runs are deterministic (sorted by `id`, `ensure_ascii=True`); diffs only reflect real article-content changes upstream. Both fetch strategies raise loudly on missing pages, empty extracts, AND disambiguation pages (matched per-source via `_EN_DISAMBIGUATION_PATTERNS` / `_DE_DISAMBIGUATION_PATTERNS` against the first 300 chars; cure is a more specific title). The wikitext stripper drops `<gallery>` blocks, `[[Datei:...]] [[File:...]] [[Bild:...]] [[Image:...]] [[Kategorie:...]] [[Category:...]]` "drop blocks" via balanced-bracket scanning (image captions can themselves contain nested links), references, templates, HTML comments, while flattening plain `[[link]]` / `[[a|b]]` to display text and stripping `'''bold'''` / `''italic''` markers.
- **`slugify` uses `str.casefold()` not `str.lower()`** so German titles fold ß → ss correctly. The previous `lower()` left ß un-decomposed (NFD has no decomposition for ß) and the ASCII-strip then removed it as a non-alphanumeric — silently corrupting `Straße` to `strae`. Diacritics (ä, ö, ü, é) still NFD-decompose to base letter + combining mark, then the combining mark is stripped — so `Straße` → `strasse`, `Größe` → `grosse`, `Ökologie` → `okologie`. The choice of base-letter vs. long-form transliteration (ä→a vs. ä→ae) was deliberate: cleaner-looking ids, low collision risk in any single whitelist, and `_assert_unique_slugs` catches any real collision before HTTP traffic.
- **`_klexikon_canonical_url` percent-encodes non-ASCII titles per RFC 3986.** `Vögel` becomes `wiki/V%C3%B6gel`, not `wiki/Vögel`. The construction is `web_base_url + urllib.parse.quote(title.replace(" ", "_"), safe="/:")` — `safe="/:"` keeps the path separator unencoded for namespace prefixes (e.g. `Datei:Foo`); `_` is already in the unreserved set so `quote` leaves it alone. The Simple English path is unaffected — its TextExtracts response carries a pre-encoded `fullurl` field that flows through unchanged. Browsers handle either form, but the percent-encoded form is what `source_url` (which flows through to the `sources` table and any attribution UI) should carry.
- **Klexikon was chosen over regular `de.wikipedia.org` for the German layer.** Klexikon (https://klexikon.zum.de) is a hand-curated German children's wiki for ages 8-13, written in plain language by educators — exactly the analogue of Simple English Wikipedia for English. Regular de-Wikipedia is dense adult prose and would have given the wrong vocabulary level for the Primer's audience. Klexikon's license is CC-BY-SA-4.0 (per the site's About page footer); the existing Simple English layer ships under CC-BY-SA-3.0 for parity with the Phase 0.2 ingest. Both labels are correct for their respective sources.
- **`auto_seed_if_empty` loads ALL matching `*.<pack>.jsonl` files in the seed dir** — this is what makes the Wikipedia layer auto-load alongside the hand-drafted seed. The German Klexikon layer (`wiki_passages.de.jsonl`, 66 articles after the 2026-05-10 corpus expansion) auto-loads on a `--language de` session without code change today. The `discover_seed_jsonl(locale) -> Option<PathBuf>` API still exists for back-compat; new code should prefer `discover_seed_files(locale) -> Vec<PathBuf>`. Seed-dir discovery happens at runtime via `$PRIMER_SEED_DIR` → `$XDG_DATA_HOME/primer/seed/` → `$CARGO_MANIFEST_DIR/.../data/seed/` — `cargo run` from `src/` populates `CARGO_MANIFEST_DIR` automatically; running the binary directly needs `PRIMER_SEED_DIR` set or files staged in the XDG path.
- **`retry_http_get` is the single HTTP-call retry boundary in `data/ingest/`.** Every strategy fetcher wraps its `http_client.get(...)` through `retry.retry_http_get`, retrying on HTTP 429 / 5xx with exponential backoff (3 attempts × 0.5 s × 2 + ±10% jitter ≈ ~1.5 s worst case before failure). `Retry-After` is honoured for delta-seconds form (HTTP-date silently drops to `compute_delay`); waits beyond the 30 s budget surface immediately as `RetryCapExceeded`. Network errors (`requests.exceptions.*`) propagate unchanged — out of scope for issue #38. Constants live in `data/ingest/retry.py` (`DEFAULT_*` module-level + `RetrySettings.default()`); a drift-guard test in `tests/test_retry.py` pins the alignment. Module-level `_RETRY_SETTINGS = RetrySettings.default()` in `data/ingest/wiki/fetch.py` is the single source of truth across all three strategy fetchers — tune there, not at call sites.
- **`data/ingest/simple_wikipedia.py` is the CLI entry point only** (~186 lines: `main` pipeline orchestrator + argparse helpers + `__main__` block; the back-compat shim that lived between PR #60 and #61 was dropped after the test imports were migrated). The ingest implementation lives under `data/ingest/wiki/`: `wiki/source.py` (`WikiSource` dataclass + presets + slugify + duplicate checks + `read_whitelist` + `to_passage` emitter), `wiki/strip.py` (Klexikon wikitext → plain text; pure functions), `wiki/fetch.py` (HTTP dispatch + per-strategy fetchers + `_RETRY_SETTINGS`). Code in this area imports directly from the submodule (`from wiki.fetch import fetch_lead`, `from wiki.source import KLEXIKON`, `from wiki.strip import strip_klexikon_wikitext`); `wiki/__init__.py` deliberately exposes no re-exports, so `from wiki import KLEXIKON` will fail.
- **macOS-native speech backend** lives in `primer-speech/src/macos/` behind the `macos-native` cargo feature on `primer-speech`, propagated through `primer-cli/macos-native` and `primer-gui/macos-native`. `MacosSpeechToText` uses `SFSpeechRecognizer` with `requiresOnDeviceRecognition = true` (hard error if the locale isn't on-device — never falls back to network per `[[project_strict_offline_first]]`); `MacosTextToSpeech` uses `AVSpeechSynthesizer.writeUtterance:toBufferCallback:` and `PhraseSplitter` for streaming. Silero stays as the VAD (macOS-26-only `SpeechDetector` would break our macOS 13 floor). Locales supported: `en-US`, `de-DE` only; Hindi is deferred until Apple ships on-device `hi-IN` (likely SpeechAnalyzer on macOS 26+).
- **`VoiceQuality` enum ordering is `Default < Enhanced < Premium`** in `primer-speech/src/macos/voice.rs`, matching Apple's `AVSpeechSynthesisVoiceQuality` ranking. `select_voice` keeps the max-quality match across all voices for the locale; the earlier `Default < Premium < Enhanced` ordering silently discarded Premium installs when an Enhanced voice was also present. Pinned by `voice_quality_ordering_matches_apple` in the same file's `tests` module. Quietly bundled into PR #134 (which was titled around SpeechAnalyzer); separately noted here for #138 so a future bisect for "voice quality changed" lands on the rationale rather than on a tangential commit.
- **macOS-native builder selection and PCM contract.** Three voice-loop builders live in sibling modules under `voice_loop/` so each file is gated by exactly one cargo feature and stays under the 500-line guideline: `backends.rs::build_local_backends` (whisper + piper); `backends_macos_native.rs::build_local_backends_macos_native` (SFSpeechRecognizer + Silero + AVSpeechSynthesizer); `backends_macos_native_26.rs::build_local_backends_macos_native_26` (SpeechAnalyzer + derived VAD + AVSpeechSynthesizer). All three are re-exported at `voice_loop::*` — **call sites should use the public re-export (e.g. `voice_loop::build_local_backends_macos_native`), not the inner module path** (`voice_loop::backends_macos_native::…`), so a future internal rename doesn't break callers. Shared mic/speaker/closure helpers live in `voice_loop::backends_common` (`SpeakerPipeline::start`, `MicPipeline::start`, `make_on_audio`, `make_drain_hook`) — every builder consumes them so the post-mic construction is one verified-once helper, not three verbatim copies. CLI/GUI pick which builder via runtime config + `cfg!(all(target_os = "macos", feature = "macos-native"))`. The PCM-callback chunk-size assumption is validated at build time via the `examples/tts_macos_pcm_smoke.rs` example — re-run after each macOS major release. Closes #149.
- **macOS-native streaming TTS uses an unbounded `mpsc::channel` of `SynthesisEvent`s.** AVSpeechSynthesizer's PCM callbacks (running on the GCD main queue) send `SynthesisEvent::Audio(chunk)` for each callback fire and `SynthesisEvent::PhraseEnd` when the zero-frame EOS sentinel arrives. The consumer drains via `try_recv` interleaved with `NSRunLoop::runUntilDate(STREAM_DRAIN_POLL_MS)` on the main path, or `recv_timeout(STREAM_DRAIN_POLL_MS)` on the background path; `STREAM_DRAIN_POLL_MS = 10` lives in `primer_core::consts::speech`. **The channel must stay unbounded** — both the main-thread path's producer + consumer are literally the same thread, and even the background path's producer is on the GCD main queue, so a bounded `sync_channel` that filled up would deadlock the GCD main queue (a hard "never block" surface in the AVFoundation contract). Unbounded makes the no-block guarantee structural; per-phrase memory footprint is at most ~100 events × ~1 KB = ~100 KB. The structural invariant — at least 2 `Audio` events arrive before the first `PhraseEnd` for any non-trivial phrase — is pinned by `streaming_emits_multiple_audio_events_before_phrase_end` in `tests/macos_tts.rs`; the user-facing per-phrase TTFA win across multi-phrase responses is pinned by `streaming_chunks_reach_speaker_before_phrase_completes` in `voice_loop::state_machine` using a `TimedMockTts` that emits 3 marker-valued chunks at 50 ms wallclock intervals.
- **macOS-native main-thread requirement and `dispatch2` bridge.** AVSpeechSynthesizer dispatches its PCM callbacks to the GCD main queue, so production callers branch at runtime: main-thread callers drive `NSRunLoop::mainRunLoop()` directly inside `synthesize_streaming_main_thread`; tokio-worker callers bounce work through `dispatch2::DispatchQueue::main().exec_async(...)` (safe typed GCD wrapper; closes #125 — replaced the prior hand-rolled `extern "C" { dispatch_async_f; static _dispatch_main_q; }` FFI block). **That background path only works in contexts where the OS main thread is running AppKit's `NSApplicationMain` (or has called `dispatch_main()`)** — `NSRunLoop::runUntilDate` from a plain CLI is **not** sufficient to drain the GCD main queue when no AppKit machinery has wired the integration. Tauri runs `NSApplicationMain` for the GUI, so the GUI's background-path callers work fine. **`primer-cli` solves this by running the tokio runtime as `Builder::new_current_thread()` on the OS main thread when `cfg(all(target_os = "macos", feature = "macos-native"))` — every voice-loop tokio task then polls on main, the synchronous `MacosTtsSession::push_text` runs on main, and synthesis takes the main-thread path** (which drives its own `runUntilDate` and works correctly). The trade-off: background tokio tasks (classifier, extractor, comprehension, embedding) are cooperatively scheduled on main and stall during synthesis; they catch up at the start of the next turn via `await_pending_post_response`, which is exactly when the dialogue manager already expects them, so user-visible behaviour is unchanged. `tests/macos_tts.rs` uses `harness = false` for a similar reason — it owns `main()` so the synthesis (also on main) is naturally on the OS main thread.
- **macOS-native STT threading, shared drain helper, and file size.** SFSpeechRecognizer is simpler than TTS: a swap to a background `NSOperationQueue` in `new()` eliminates the main-thread dependency for STT result callbacks entirely. The shared drain-loop helper (closed #124) now lives at `primer-speech/src/macos/stream_drain.rs::drive_streaming_drain`; both streaming paths call it with a path-specific closure (`try_recv` + `NSRunLoop::runUntilDate` slices on main; `recv_timeout(STREAM_DRAIN_POLL_MS)` on background) absorbing the threading-model difference. The helper owns the deadline check, `on_event` dispatch, and `PhraseEnd`-terminates-the-loop invariant; structural coverage for both branch shapes ships as `#[cfg(test)] mod tests` in `stream_drain.rs`. `tts.rs` is 636 lines after the #125 migration shed the `mod dispatch { ... }` FFI block and the `extern "C" fn trampoline` + `Box::into_raw`/`Box::from_raw` ceremony (still over the 500-line guideline); the deferred file-split is tracked alongside #126 (`spawn_blocking` wrap).
- **macOS-native `dispatch2` closure must whole-struct-capture (RFC 2229 gotcha).** The closure passed to `DispatchQueue::main().exec_async(...)` in `synthesize_streaming_background` must capture the `StreamCtx` *as a whole struct* for the struct-level `unsafe impl Send` to apply — Rust 2021 disjoint capture (RFC 2229) would otherwise split it into per-field captures and inherit `Retained<AVSpeechUtterance>`'s `!Send` bound, defeating `exec_async`'s `F: Send` requirement. The `let owned = ctx;` identity-rebind on the first line of the closure body is the load-bearing way to force whole-struct capture; do not "simplify" it by destructuring `ctx` directly, by accessing `ctx.utterance` / `ctx.tx` field-wise from the closure body, or by hoisting the destructure outside.

- **`macos-native-26` is the new SpeechAnalyzer-backed feature on `primer-speech`** for macOS 26+ (and iOS 26+ when an iOS host application lands). Mutually exclusive with `macos-native`; a `compile_error!` in `primer-speech/src/lib.rs`, `primer-cli/src/main.rs`, and `primer-gui/src/lib.rs` enforces the XOR. Build with `~/.cargo/bin/cargo build --features primer-cli/speech,primer-cli/macos-native-26`. STT and derived VAD events come from `SpeechAnalyzer` + `SpeechTranscriber` + `SpeechDetector` (with `.medium` sensitivity, `.progressiveTranscription` preset) via a Swift sidecar at `crates/primer-speech/swift-sources/Macos26PipelineImpl.swift`, bridged to Rust via `swift-bridge 0.1` and compiled to a `libMacos26Pipeline.a` by `crates/primer-speech/build.rs` using `swiftc`. The bridge exposes one opaque type with `create(locale_bcp47).await` (panics on failure — swift-bridge 0.1.59 can't codegen `Result<OpaqueSwiftType>`), `feed_audio(samples: Vec<f32>)`, `next_result().await -> ResultEvent` (sentinel `stream_done: bool` for EOF), and `stop().await`. **`nextResult()` is single-flighted via a cached `Task<ResultEvent?, Error>` on the Swift side** so `tokio::select!` cancellation in `analyzer.rs::run_consumer_loop` doesn't trigger `AsyncStreamBuffer.swift:508: Fatal error: attempt to await next() on more than one task` — cancelled callers are transparently picked up by the next iteration. VAD events are derived from transcriber activity by `DerivedVadStateMachine` (in `macos26/vad.rs`): first non-empty partial → `SpeechStart`; `isFinal=true` OR 1000 ms inactivity → `SpeechEnd`. Tunable thresholds at `primer_core::consts::speech::macos26` (`SPEECH_END_TIMEOUT = 1000 ms`, `EVENT_POLL_INTERVAL = 100 ms`). The 1000 ms timeout is empirically tuned: 600 ms cut off mid-sentence on natural child-paced speech; 1200 ms felt sluggish on short sentences. With `.progressiveTranscription` preset, genuine `isFinal=true` arrives only on pipeline teardown — the inactivity timer is the load-bearing path for end-of-utterance detection. The audio thread is independent: cpal mic → resample to 16 kHz → tokio mpsc → consumer task which owns the Macos26Pipeline. A small bridge task converts the tokio `TextMessage` stream into the std-mpsc `String` stream that `ChannelStt` expects, dropping volatile partials (only `is_final` results cross to ChannelStt — future refactor can plumb partials through). TTS reuses `MacosTextToSpeech` from `crates/primer-speech/src/macos/tts.rs` — AVSpeechSynthesizer is unchanged on macOS 13+ and 26+, so the existing `macos/` module's TTS submodule is widened (`tts` and `voice` cfg-gated on `any(feature = "macos-native", feature = "macos-native-26")`; `stt`, `locale`, `permissions` stay `macos-native`-only because they use `objc2-speech`/SFSpeechRecognizer). Locale support: `en-US` and `de-DE`; Hindi errors loudly at `Macos26Stt::new` because `SpeechTranscriber` doesn't support `hi-IN` as of macOS 26.5. Asset download is silent and OS-managed via `AssetInventory.assetInstallationRequest(...).downloadAndInstall()` on first use (en-US ships with macOS 26; de-DE downloads on first run) — the macOS path is explicitly the friction-free demo surface, not production children's hardware where `[[project_strict_offline_first]]` applies. `build.rs` adds `-Wl,-rpath,/usr/lib/swift` to the link args so dyld can resolve `@rpath/libswift_Concurrency.dylib` from the dyld shared cache (libswift_Concurrency has no on-disk file on macOS 12+ — it lives only in the cache). `primer-cli` has a tiny `build.rs` of its own that emits the same rpath via `cargo:rustc-link-arg-bins` because `cargo:rustc-link-arg` is package-scoped and doesn't propagate from `primer-speech` to downstream binary crates. **Empirical A/B vs Whisper** (PR #131): ~100× faster time-to-first-partial (~30 ms vs ~3.8 s); ~2× faster final transcript (~800 ms vs ~1.8 s); negative-lag predictive partials. Integration smoke tests at `crates/primer-speech/tests/macos26_smoke.rs` are `#[ignore]`'d (`cargo test -p primer-speech --features macos-native-26 --test macos26_smoke -- --ignored --nocapture`); manual end-to-end requires speaking into the mic. Module is internally Apple-platform-portable via `cfg(target_vendor = "apple")`; rename `macos26/` → `apple26/` is a mechanical follow-up once an iOS host application exists. ## Pedagogical principles encoded in the system prompt

These constrain prompt-builder and dialogue-manager changes — if a change makes the Primer more answer-y or more engagement-maximising, it is wrong:

- The Primer asks more questions than it answers; pure factual questions get a direct answer **then** a Socratic pivot.
- The Primer never tries to maximise engagement. It detects frustration/disengagement and offers breaks, scaffolding, or session close — never guilt.
- All learner data is local; cloud inference sends turns per-request only.
- Comprehension is verified through transfer questions, application, and contradiction probing — not assumed from a confident-sounding response.
