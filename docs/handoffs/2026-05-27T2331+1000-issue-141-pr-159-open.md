## What we shipped this session

**Closed issue #141** — friendlier panic when `xcrun` / `xcode-select` / `swiftc` are missing or fail.

`primer-speech/build.rs` previously called every Xcode-toolchain probe with `.expect("invoke xcrun")`-style messages, so the first user on a dev box without full Xcode (Command Line Tools only, or no Xcode at all) saw an opaque `"No such file or directory"` panic rather than an actionable install hint. PR #159 routes every probe + the `swiftc` invocation through pure helpers and emits a multi-line `cargo:warning=` with the remediation command before the panic.

### Design — pure-helper module loaded two ways

New file [`crates/primer-speech/src/macos26/build_hints.rs`](src/crates/primer-speech/src/macos26/build_hints.rs) (~280 lines including 14 unit tests) holds:

- `XCODE_HINT: &str` / `SWIFTC_HINT: &str` — multi-line install hints (Xcode 16+, App Store, `sudo xcode-select --switch …`, Command-Line-Tools-aren't-enough warning).
- `enum ProbeOutcome { Ok(String), NonZero { code: Option<i32>, stderr: String }, SpawnFailed(String) }` — pure data; signal-terminated processes carry `code: None`.
- `fn classify(result: io::Result<Output>) -> ProbeOutcome` — pure classifier; no side effects.
- `fn format_failure_message(name, args, outcome) -> Option<String>` — `None` on Ok, `Some(human msg)` on either failure variant.
- `fn cargo_warning_lines(hint) -> Vec<String>` — splits a hint into `cargo:warning=…` lines for the caller to `println!`.

The file is loaded **two ways**:

- `build.rs` pulls it in via `#[path = "src/macos26/build_hints.rs"] mod build_hints;` (gated on `feature = "macos-native-26"`). build.rs is its own crate so this is fine.
- The lib crate declares `#[cfg(test)] mod build_hints;` in [`src/macos26/mod.rs`](src/crates/primer-speech/src/macos26/mod.rs) so `cargo test --features macos-native-26` picks up the inline unit tests at `macos26::build_hints::tests`. No production-build cost (the `cfg(test)` gate keeps the module out of release builds).

This pattern means the helpers ship with TDD coverage even though build.rs itself isn't reachable via `cargo test`. **Reuse this when next adding a non-trivial helper that needs to be shared between a build script and the lib crate.**

### What the user sees on failure

Before (xcrun missing):
```
thread 'main' panicked at crates/primer-speech/build.rs:107:14:
invoke xcrun: No such file or directory (os error 2)
```

After:
```
warning: The `macos-native-26` feature requires Xcode 16+ with the macOS 26 SDK.
warning: Install Xcode from the Mac App Store, then run:
warning:     sudo xcode-select --switch /Applications/Xcode.app/Contents/Developer
warning: Command Line Tools alone do NOT include the macOS SDK — full Xcode is required.

error: failed to run custom build command for `primer-speech v0.1.0 (…)`

Caused by:
  ...
  thread 'main' panicked at crates/primer-speech/build.rs:145:21:
  failed to invoke `xcrun`: No such file or directory (os error 2)
```

Same `XCODE_HINT` treatment for: `xcrun` non-zero exit (SDK uninstalled), `xcode-select` spawn failure, `xcode-select` non-zero exit, `swiftc` spawn failure (treated as a missing-Xcode signal). A separate `SWIFTC_HINT` is used for the `swiftc` non-zero-exit path because at that point Xcode is present and the actionable next step is to read the swiftc output above.

## Open decisions / risks

Newly surfaced this session:

- **The `#[path]` + `#[cfg(test)] mod` two-load pattern is the load-bearing structure** for `build_hints` having unit tests at all. Build scripts aren't reachable via `cargo test`; the lib's `cfg(test) mod build_hints;` is what makes the inline `#[cfg(test)] mod tests` block visible to `cargo test --features macos-native-26`. The `#![allow(dead_code)]` at the top of `build_hints.rs` is needed because the helpers are only *used* via `#[path]` from build.rs, not by anything in the lib — without the allow, `cargo clippy -- -D warnings` would fire on every helper item in test mode. If a future refactor consumes a helper from a non-test lib path, the allow can be tightened.
- **The unit tests use `sh -c 'exit 7'` instead of `/bin/false`** because `false` lives at `/bin/false` on Linux and `/usr/bin/false` on macOS. `sh` is universally on PATH and lets us pin a specific exit code, which is a stronger assertion than "non-zero".
- **`format_failure_message` returns `Option<String>` rather than `String`** so it can be called once per outcome variant in a single `match` arm that destructures the outcome by reference. The build.rs site does `format_failure_message(..., ref failure).expect("non-Ok outcome must produce a failure message")` — the expect is structural, not a real failure mode.

## Patterns to reuse, not reinvent

New from this session:

- **Pure helpers shared between `build.rs` and the lib crate's test target: file lives under `src/<mod>/<helper>.rs`, loaded by build.rs via `#[path = "src/<mod>/<helper>.rs"] mod <helper>;` and by the lib via `#[cfg(test)] mod <helper>;` in the parent `mod.rs`. Put `#![allow(dead_code)]` at the top of the helper file so the lib-side `cfg(test)` load doesn't trip `-D warnings`.** This is the *only* way I know of to get TDD coverage on build-script logic without duplicating code, and it keeps the helper's release-build cost at zero. Reach for it whenever build.rs has non-trivial branching logic.
- **For build-script tests that need a non-zero exit status, use `Command::new("sh").args(["-c", "exit 7"]).output()` instead of `/bin/false`.** Pins a specific exit code (stronger assertion than "non-zero") and avoids the `false` path divergence between Linux (`/bin/false`) and macOS (`/usr/bin/false`).
- **Multi-line `cargo:warning=` is the right surface for actionable install hints on build failures.** Cargo renders each `cargo:warning=` line on its own with the `warning:` prefix, so a multi-line hint stays readable. Don't try to cram a multi-line install hint into a single warning string with embedded `\n` — cargo doesn't unescape.
- **For probe commands (xcrun, xcode-select, swiftc), distinguish three outcome variants** (`Ok(stdout)`, `NonZero { code, stderr }`, `SpawnFailed(err_string)`) rather than collapsing into a single Result. Each variant has a meaningfully different remediation surface — "binary missing" needs an install hint, "binary present but failed" needs the stderr in front of the user, "success but stdout malformed" should probably never happen but is the only thing that can hide a real bug.
