## Open decisions / risks

Newly surfaced this session:

- **Branch-protection ↔ paths-ignore interaction is now a documented coupling.** If `cargo test (default features)` is wired as a required status check on `main` while the workflow-level `paths-ignore` from PR #169 is in place, a docs-only PR will be stuck waiting forever for a job that never ran. Two known mitigations, documented inline in `.github/workflows/ci.yml`:
  1. **Skipped-but-required stub job** — a tiny "always-pass" job whose name matches the required-check name, running only on docs-only paths.
  2. **Per-job `paths:` allowlists** — each job opts in to file patterns; non-matching jobs get a `skipped` conclusion that branch protection accepts as passing. More YAML but cleaner semantics.

  The right path depends on which approach the repo owner prefers when flipping branch protection. **Both decisions revisit together.**
- **`**/*.md` safety relies on a runtime grep, not a structural guarantee.** Future code that adds `include_str!("docs/something.md")` or reads a committed `.md` file as test data would silently lose CI coverage on docs-only edits. The inline YAML comment notes the verification; mitigations (a lint, a regression test) are over-engineering until at least one such case appears. Defer.
- **The CodeQL Analyze jobs are NOT skipped on docs-only changes.** Those are configured via the GitHub-default CodeQL setup (no workflow file in `.github/workflows/`), so this PR's filter doesn't reach them. The waste is much smaller (CodeQL Analyze on Rust is faster than the full `cargo test` job) but non-zero. Repo-owner-only fix.

## Patterns to reuse, not reinvent

New from this session:

- **"Verify the safety assumption before encoding a CI exclusion" is the corollary of PR #167's "verify locally first" rule.** Before adding `**/*.md` to a `paths-ignore` list, prove the claim "no Rust test reads a committed `.md` file" with a grep. The proof becomes part of the inline rationale comment so a future contributor adding `include_str!("docs/foo.md")` has a chance to notice. The artifact of the rule is the YAML comment block (lines 3–32 of the new `ci.yml`) — it cites the exact grep command and the file it found (`primer-kb-load/src/lib.rs:510`) so the verification can be re-run, not just trusted.
- **"Document the branch-protection coupling at the point of the workaround"** — when workflow-level `paths-ignore` lands without branch protection in place, the YAML rationale block explicitly names both mitigation patterns (skipped-but-required stub job; per-job `paths:` allowlist) so the next person flipping branch protection on `main` doesn't have to rediscover them. This is the same shape as the inline `option-(a)-vs-(b)` block at `state_machine.rs:847` (the macos-26 design call): document the decision at the load-bearing line, not in a separate doc that nobody reads.

Carried forward (from prior sessions; see prior handoff trail for two-layer drain, capacity-1-channel-as-sync-lever, pure-helper extraction, `#[path]` + `#[cfg(test)] mod` two-load pattern for build-script TDD, gate-narrowing, RFC-2229 whole-struct capture, trait-directive refusal documentation patterns, bundle-struct + free-function shape for shared closure builders, one-file-one-cargo-feature-gate as a quick file-size win, public-re-export protecting call sites from internal renames, generic-single-slot-cache + mechanical-caller-wiring as a TDD shape, `Option<T>` taken in finalize-AND-Drop for consume-on-success-AND-on-panic, put-back-before-propagate for caches keyed by expensive construction, "verify locally first, then encode in CI" for feature-combo drift-guards, and "`cargo clippy -- -D warnings` over `cargo check` for drift-guards when the rotted-lint bug class exists").
