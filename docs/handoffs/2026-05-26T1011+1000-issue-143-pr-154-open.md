## What we shipped this session

**Closed issue #143** — the narrow result-loss race in [`crates/primer-speech/swift-sources/Macos26PipelineImpl.swift::nextResult`](src/crates/primer-speech/swift-sources/Macos26PipelineImpl.swift). Surfaced by the PR #134 round-2 code review and acknowledged in a docstring on `nextResult` itself. The race window is microseconds wide; user-visible impact is at most one missing intermediate transcript per drop event under `.progressiveTranscription` (genuine `isFinal=true` arrives only on teardown).

### The fix — two-layer drain, no continuations needed

The issue body's sketched fix used `withCheckedContinuation`. That isn't needed. The existing `await task.value` already gives us the synchronisation point; we just need to ensure the yielded value survives the narrow window where the awaiter is dropped after the task completes but before the value is consumed.

New `pendingResult: ResultEvent?` field on `Macos26Pipeline`. The inner Task writes into it **before** the defer clears `nextResultTask` (defers run on return, after body assignments). Entry to `nextResult()` becomes:

1. **Layer 1** — if `pendingResult` is Some, drain and return. This is the path that closes the #143 race: a Task that completed in the microsecond gap left `pendingResult` populated and `nextResultTask` cleared; the next call drains it cleanly.
2. **Layer 2** — if a Task is still in flight, await it; the body writes `pendingResult` before clearing `nextResultTask`, so a drain reads the value after the await returns.
3. **Cold path** — spawn a fresh Task. Write `pendingResult` in the body; defers clear `nextResultTask` on return.

`stop()` also clears `pendingResult` so a stop-then-stray-nextResult sequence returns stream_done rather than a stale pre-stop event, mirroring the existing `lastText = ""` clear in `nextResultBridge`'s end-of-stream path.

### Behaviour matrix — all paths covered

| Path | pendingResult | nextResultTask | Outcome |
| --- | --- | --- | --- |
| Cold-path happy: spawn → await → drain | None → Some → None | nil → Some → nil | Event delivered ✓ |
| Two simultaneous awaiters (A dropped before Task completes) | None → Some → None | Some → Some → nil | Event delivered to B ✓ |
| **#143 race**: Task completes between defer + next call | None → **Some persists** → None | Some → nil → nil | Layer-1 drain delivers ✓ |
| End-of-stream | stays None | Some → nil | nil → stream_done sentinel ✓ |
| `iter.next()` throws | stays None | Some → nil | re-throw via `try await` ✓ |
| Outer awaiter dropped during throw | stays None | Some → nil | next call retries (same as before) ✓ |

End-of-stream and throwing both leave `pendingResult` nil — observable behaviour identical to before the cache (re-running `iter.next()` on an exhausted iterator yields nil idempotently; re-throwing is correct error propagation).

## Open decisions / risks

Newly surfaced this session:

- **The Layer-1 → Layer-2 → Cold-path branch order in `nextResult()` is load-bearing.** Layer 1 must run before Layer 2, because a completed Task that wrote `pendingResult` and cleared `nextResultTask` looks identical-from-Layer-2's-perspective to "no in-flight Task" — the only way to recover the cached value is to check `pendingResult` first. Inverting Layer 1 and Layer 2 would not break the cold path (Layer 2 also returns Some via the post-await drain), but would spawn unnecessary work in the rare "Layer 2 + already-completed Task" intersection (Swift's `await task.value` on an already-completed Task returns its value without re-running the body; cheap but not free). The comment ladder in the Swift source makes the order intent explicit so a future "simplification" doesn't accidentally swap them.
- **The `self.pendingResult = event` BEFORE return ordering is also load-bearing but not enforced.** Inverting the cache write into a defer would reintroduce the race (the defer clearing `nextResultTask` runs first or last depending on order; if `pendingResult` write is in a later-firing defer, a caller can see `nextResultTask == nil && pendingResult == nil` between defers). Documented with an explicit "ORDER MATTERS / inverting this would reintroduce the #143 race" comment, but not pinned by anything mechanical because we have no Swift test harness.

## Patterns to reuse, not reinvent

New from this session:

- **For an async-iterator-cancellation race where the value is held in a completed Task that nobody is awaiting, the fix is a one-field cache written by the Task body before any cleanup defer fires.** No `withCheckedContinuation` needed — the existing `await task.value` already gives the synchronisation point. The cache merely ensures the value survives the narrow window where the awaiter is dropped after the task completes. This is structurally simpler than the issue body's continuation sketch suggested; resist the temptation to over-engineer.

- **Layer-1-before-Layer-2 branch order in cache-then-await patterns is load-bearing.** When a cache check and an in-flight-future await both might recover the same value, the cache check MUST run first — once the future completes and writes to the cache, both branches see "no in-flight Task" and would re-do work otherwise. Comment the order explicitly so a future "simplification" doesn't swap them.

- **When implementing a race-fix in a Swift sidecar with no test harness, force a full rebuild via `cargo clean -p <crate>` before declaring success.** Cargo's caching can skip the `swiftc` invocation if it thinks nothing relevant changed (the Swift sources are listed in `rerun-if-changed=swift-sources` but the dependency-tracking is not as precise as for Rust source). Touching the Swift file and re-running may produce a "Finished 0.08s" line that looks like success but didn't actually exercise the type-checker on the new code. `cargo clean -p primer-speech` followed by `cargo build -p primer-speech --features macos-native-26` is the load-bearing verification command on Swift-only changes.

- **When the race is documented in the code already (via a docstring), addressing it with a structural fix is a one-paragraph commit message away.** The issue body's "sketch of a fix" with `withCheckedContinuation` is one design option, not the only one. Read the existing docstring carefully — it often already names the invariant the new code needs to preserve.
