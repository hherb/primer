## Patterns to reuse, not reinvent

New from this session:

- **Once a feature gate that was previously broken starts building, all of its previously-unreached files get clippy'd for the first time.** Expect a one-time burst of pre-existing lints in code that has rotted unnoticed in the broken-gate file. Per "implement quick fixes inline", fix them in the same PR that unblocks the build — they're trivially mechanical and each one only takes a moment, and rolling them into a follow-up issue would just slow the next dev down.
- **`let Some(x) = y else { return None; };` → `let x = y?;`** when the surrounding return type is `Option<T>`. This is what clippy's `question_mark` lint enforces. The `let-else` form is only needed when the early-return value isn't `None` (e.g. a default, an error, or `continue`/`break`/panic).
- **`fn f(arg: &PathBuf)` → `fn f(arg: &Path)`** is structurally equivalent and saves the `PathBuf` requirement at the call site. Same for `&String` → `&str`, `&Vec<T>` → `&[T]`. Clippy enforces via `ptr_arg`.
- **`format!("foo{}", VAR)` → `format!("foo{VAR}")`** when `VAR` is in scope and a path-y identifier (not an expression). Clippy enforces via `uninlined_format_args`.
