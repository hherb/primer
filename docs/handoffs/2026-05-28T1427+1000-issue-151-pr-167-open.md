### Why clippy, not check?

Rationale documented in the YAML comment block. The existing Linux feature-combos job uses `cargo check`; the macOS job uses `cargo clippy -- -D warnings`. Clippy catches both build breaks AND lint rot; check catches only the former.

## Patterns to reuse, not reinvent

New from this session:

- **"Verify locally first, then encode in CI" is the right discipline for new CI feature-combo entries.** Before adding a `cargo clippy -- -D warnings` step to a CI workflow, run the exact same command locally on `main` first. If it's clean, the CI entry is a pure drift-guard. If it's not, the inline-quick-fix policy applies — fix the rot in the same PR rather than landing a CI step that fails immediately. This session's verification matrix is the artifact of that pattern. Cleaner than "land the YAML and let CI tell you what's broken," which forces a follow-up commit on the same PR.
- **"`cargo clippy -- -D warnings` over `cargo check` for drift-guards" is the policy when the rotted-lint bug class exists.** The existing Linux feature-combos job uses `cargo check` (cheaper, catches only build breaks). The new macOS job uses `cargo clippy -- -D warnings` (slightly more expensive, catches lint rot too). The rule-of-thumb: if a code path has ever rotted a clippy lint that the default workspace-clippy didn't catch (PR #150 is the canonical example), prefer clippy over check on that path's drift-guard.

Carried forward (from prior sessions; see prior handoff trail for two-layer drain, capacity-1-channel-as-sync-lever, pure-helper extraction, `#[path]` + `#[cfg(test)] mod` two-load pattern for build-script TDD, gate-narrowing, RFC-2229 whole-struct capture, trait-directive refusal documentation patterns, bundle-struct + free-function shape for shared closure builders, one-file-one-cargo-feature-gate as a quick file-size win, public-re-export protecting call sites from internal renames, generic-single-slot-cache + mechanical-caller-wiring as a TDD shape, `Option<T>` taken in finalize-AND-Drop for consume-on-success-AND-on-panic, and put-back-before-propagate for caches keyed by expensive construction).
