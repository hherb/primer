Issue #22 asked to cache prepared statements on the Phase 0.2 corpus-bootstrap path, where `insert_passage` is called for the whole encyclopedia corpus and the per-row `format!` + `prepare()` re-parse is real cost (irrelevant at conversation-turn rates, dominated there by LLM latency).

- **Part 1 (cache locale-suffixed table-name `String`s) was already done** in a prior PR — the issue's suggested step 1 needed no work. This PR finishes **part 2**.
- **New private `LocaleSql` struct** (`LocaleSql::new(pack) -> LocaleSql`, pure) precomputes every per-locale hot-path statement string **once** at `open_for_locale`: `insert_content`, `insert_fts`, `insert_embedding`, `upsert_embedding`, `select_rowid_by_id`, `retrieve`, `knn_scan`. Single source of truth for hot-path SQL.
- **Routed the hot paths through `Connection::prepare_cached`** (`insert_passage`/`insert_passage_inner`, `insert_passage_with_vector`, `upsert_embedding`, `retrieve`, `knn_scan`) — each call is now a statement-cache hash lookup, not a re-parse. `insert_passage_inner` now takes `&LocaleSql` (works against both `Connection` and `Transaction` via `prepare_cached`).
- **Dropped the now-unused `passages_table` field**; `content_table`/`embeddings_table` remain only for the one-shot diagnostics (`passage_count`, `list_passage_ids`, `passages_missing_embedding`).
- **Benchmark** (`crates/primer-knowledge/tests/bulk_insert_bench.rs`, `#[ignore]`): times the cached path vs a raw `format!`+`execute` baseline over 50k in-memory rows. **~1.14x in release** (~12%, ~83 ms/50k rows). Honest result — modest because FTS5 tokenization/b-tree work dominates SQL parse; the cached path is *strictly faster even while paying an extra mutex lock per row* the raw baseline skips, so it's a conservative lower bound. Added `rusqlite` as a dev-dep (already a normal dep, zero extra build cost) for the raw baseline.
- **TDD:** two new `LocaleSql` unit tests (pack-id interpolation across every statement; FTS insert binds rowid as `?4`). Written red → green.

## Patterns to reuse, not reinvent

New from this session:
- **`Connection::prepare_cached(&stable_sql)` is the bulk-write idiom in this codebase** — it caches the compiled `sqlite3_stmt` by SQL text, so a *stable* (precomputed, not per-call-`format!`'d) string turns each call into a hash lookup. Pair it with a per-instance struct (`LocaleSql`) that builds every per-locale statement once at open. Works transparently against a `Transaction` too (derefs to `Connection`), so a static helper taking `&LocaleSql` serves both the autocommit and the in-transaction paths.
- **When a perf refactor's measured win is modest, report the number honestly and say *why*** (here: FTS5 tokenization/b-tree dominates SQL parse). A printed comparative `#[ignore]` bench (cached vs the literal pre-refactor `format!`+`execute` baseline, in-memory to remove `fsync` noise) is the right evidence — assert correctness (row counts), print the ratio, and only guard against a *pathological* regression (`cached < baseline * 3`), never assert a tight wall-clock bound (CI flake).
- **In-memory SQLite (`Path::new(":memory:")`) isolates CPU cost from disk in a write benchmark** — `fsync` per autocommit would otherwise swamp the parse-elimination signal.
- **TDD for a pure SQL-builder is a real test, not an echo:** pin the *meaningful* properties (pack-id appears in every statement → locale isolation; the FTS insert binds rowid as `?4` not `?1` → a silent param reshuffle is caught), not verbatim string equality.
