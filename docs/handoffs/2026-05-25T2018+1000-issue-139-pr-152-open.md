## What we shipped this session

**Closed issue #139** — pre-speak mic samples no longer leak past the SPEAK→LISTEN transition in the macos-native-26 audio thread. The original implementation only short-circuited the **append** step while `is_speaking` was true, so any samples already in the `pending: Vec<f32>` from before the SPEAK transition sat there waiting, then leaked into the post-speak transcription as the first slice of the next chunk. The fix is one line: unconditional `pending.clear()` on every speaking iteration — same pattern the sibling macos-native and whisper paths already use via `raw_buf.clear()`.

### Pure-helper extraction (per the project's pattern)

The state-transition logic now lives in a pure `PendingMicBuffer` helper at [`crates/primer-speech/src/voice_loop/macos26_audio_buffer.rs`](src/crates/primer-speech/src/voice_loop/macos26_audio_buffer.rs) — same shape as the existing `vad_debounce` and `phrase_split` modules. The audio thread imports `PendingMicBuffer::new(in_chunk_samples)` and calls `buffer.feed(&tmp[..popped], is_speaking_now)` per iteration; the resampler call and `audio_tx.try_send` stay inline because they depend on the rubato/tokio types the thread owns.

### TDD red→green verification

8 unit tests cover empty/partial/full/oversized inputs, FIFO ordering across calls, the `is_speaking` discard, the load-bearing regression test `pre_speak_samples_do_not_leak_past_speak_transition`, and a `repeated_speak_iterations_stay_clear` sanity check. Verified the regression test really catches the bug by temporarily reverting the `pending.clear()` and watching it fail at the explicit `#139` assertion line with `left: 256, right: 0`.

## Patterns to reuse, not reinvent

New from this session:

- **When a bug lives in a long imperative loop, lift the state-transition core into a pure helper and TDD-test it.** `PendingMicBuffer` is the third instance of the pattern in `primer-speech` (after `vad_debounce` and `phrase_split`). The audio thread keeps the cpal/rubato/tokio wiring; the helper carries the invariant. Test the helper red→green: write the regression test first, comment out the fix, watch the test fail at the explicit issue-number assertion, restore the fix.
- **`pub(crate) mod foo;` with `#[cfg(...)] mod tests` inside is the right shape for a small per-feature helper.** No need to expose it on the public API surface; the test binary still picks up the tests because `#[cfg(test)]` flips on under `cargo test`.
- **Use explicit issue-number assertion messages** (e.g. `"pre-speak buffer must be cleared on speak transition (#139)"`) so when a test fires in the future, the issue context is one grep away. Pays off when the regression resurfaces and you can immediately jump to the closed issue.
- **A bug that "shows up only in real audio" can still be tested structurally** if you can isolate the state-update path from the I/O path. The unit test won't catch a real-audio regression caused by a different mechanism (Swift analyzer, resampler, ringbuf), but it pins the *specific* invariant the issue called out.
