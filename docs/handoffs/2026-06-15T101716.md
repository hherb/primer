## What we shipped this session (PR #232, branch `a11y-mobile-drawer-aria-modal`)

**`7631360` — feat(gui): complete the mobile drawer ARIA modal — role=dialog + strict focus trap (#231).** PR #230 left the drawer a *partial* modal (chat `<main>` inert, focus moved in/out). This closes the two completeness gaps #231 flagged:

1. **ARIA announcement.** `applyDrawerModality` now toggles `role="dialog"` + `aria-modal="true"` on `#sidebar` on open and clears them on close, via a new `setDrawerDialogRole(modal)` helper. JS-toggled (not static markup) and mobile-only — the same `#sidebar` is the persistent desktop column, where a dialog role would wrongly announce a modal over the two-column layout. The element's existing `aria-label="Evaluation sidebar"` names the dialog.
2. **Strict focus trap.** While the drawer is open every interactive header control *except* the close toggle is made `inert` (new `setHeaderChromeInert(inert)` + `nonToggleHeaderControls()` helpers in `app.js`; `dom.header = document.querySelector("header.app-header")`). Combined with the already-inert chat `<main>`, the only tabbable elements are the drawer contents and `#sidebar-toggle`, so `Tab`/`Shift+Tab` wrap naturally between them with **no manual keydown handler** — exactly what `aria-modal="true"` promises. The toggle stays live (it is the close affordance).
3. **Shared teardown.** New `releaseDrawerModality()` helper clears the chat inert, the header-chrome inert, and the role/aria-modal in one place. Both the close branch of `applyDrawerModality` and the breakpoint-change handler funnel through it, so a drawer resized up to desktop can't strand any of the non-media-scoped modal DOM state (the old change handler cleared only the chat inert).

**Product decision (the issue asked us to decide):** **strict trap**, chosen with the owner. Settings / switch-session are reachable only after the drawer is closed — not left tabbable underneath. Implemented by inert-ing the non-toggle header chrome (the issue's accepted option) rather than a manual Tab/Shift+Tab keydown trap, which keeps the code pure (reuses `inert`) and avoids a brittle key handler.

**Tests added (test-first, red → green):** 4 new `responsive_layout_contract` shape-assertions — `app_js_announces_open_drawer_as_modal_dialog`, `app_js_strict_trap_inerts_header_except_toggle`, `app_js_clears_modal_state_on_breakpoint_change` (scoped to the `change`-handler slice), and `index_html_has_app_header_for_trap_target` (guards the `querySelector("header.app-header")` selector target, mirroring the existing chat-main guard). 3 went red before the impl; the 4th is a guard on an already-present element (passes immediately, like `index_html_has_chat_main_for_inert_target`).

- **Strict-trap design choice is locked in.** The trap is implemented by inert-ing the non-toggle header controls, NOT a manual `Tab`/`Shift+Tab` keydown handler. Don't "add a proper focus trap with key handling" later — `inert` + `aria-modal` is the standard pattern and the natural Tab cycle (toggle ⇄ drawer contents) is the intended behaviour. The `el !== dom.sidebarToggle` exclusion is load-bearing (the toggle must stay reachable as the close affordance).
- **`inert` browser support.** Relies on the global `inert` attribute (WebKit 15.4+ / Chromium 102+ — all Tauri WebView targets qualify). Same risk profile as PR #230; low on modern devices.

## Patterns to reuse, not reinvent

New this session:
- **Strict mobile-modal focus trap = inert the non-toggle background controls, not a keydown Tab handler.** With the modal's own container focusable (`tabindex="-1"`) and *everything reachable except the modal + its close control* marked `inert`, the browser's native Tab cycle already wraps within the surviving set — no `keydown`/`Shift+Tab` bookkeeping. `inert` also satisfies the `aria-modal="true"` "background is inert" promise in one stroke. The one exclusion that must be preserved is the close affordance (`el !== dom.sidebarToggle`).
- **Toggle ARIA-modal role in JS, mobile-only, when the modal element is dual-purpose.** `role="dialog"`/`aria-modal="true"` are *added on open and removed on close* (not static markup) because the same element is also a non-modal desktop landmark. Pair with a single `releaseDrawerModality()` teardown that both the close path and the breakpoint-`change` handler call, so non-media-scoped DOM state (inert, role, aria-modal) can't drift between the two cleanup sites.
- **Contract-test the `change`-handler delegation by slicing to the handler body** (`find("addEventListener(\"change\"")` → next `});`) and asserting the teardown call is inside that slice — tighter than a whole-file `.contains`, which would false-pass on the close path's copy of the same call.

Carried (still true): the mobile-only modal-drawer guard pattern from PR #230 (named `applyDrawerModality(open)` that early-returns off-mobile; breakpoint-`change` handler unconditionally clears non-media-scoped DOM state); frontend a11y "TDD" here = extend `responsive_layout_contract.rs` with `include_str!` shape-assertions using `strip_js_comments` + the `sidebar_open_tag` tag-extractor.
