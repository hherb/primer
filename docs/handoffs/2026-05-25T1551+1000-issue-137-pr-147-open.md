If a future refactor narrows either of those to a single feature, expect clippy to fire and just drop the inner attr.

## Patterns to reuse, not reinvent

New from this session:

- **Feature gates should reflect *true* dependencies at each level.** When a function lives in a file whose `#![cfg(...)]` is broader than what the function itself uses, you have two options: (a) leave it — small over-coupling, no functional harm, easy to forget; (b) extract into a sibling module with the narrower gate — makes the dependency graph honest and lets future feature flag changes flow correctly without re-touching downstream code. Choose (b) when there's more than one such function in the file, or when a documented future change (e.g., loosening the parent gate) will need the inner gates to already be accurate.
- **Cross-module shared types belong in a `_common.rs`-style sibling at the narrowest gate they can support.** Don't define them in the same file as one consumer if multiple consumers with different feature requirements exist. Otherwise, every consumer inherits the file's over-broad gate via use-statements.
- **Clippy's `duplicated_attributes` lint is sensitive to single-feature vs multi-feature cfg forms.** A `#![cfg(feature = "X")]` inner attribute on a module file whose parent has `#[cfg(feature = "X")] pub mod ...;` will lint. The same pattern with `all(feature = "X", feature = "Y", ...)` won't. When you must reduce a multi-feature inner cfg to a single feature, drop the inner attribute and rely on the parent gate.
- **When in doubt about whether to keep an inner `#![cfg(...)]`** that matches the parent module declaration's cfg: drop it. The parent gate is sufficient and the inner attribute is just a tripwire.
