## What we shipped this session

All on branch `llamacpp-bos-per-model-201`:

- **`54b2055` — llamacpp per-model BOS handling (#201).** `RealLlamaEngine::infer` no longer hard-codes `AddBos::Always`. Confirmed against `llama-cpp-2 0.1.146` source: `str_to_token` calls `llama_tokenize` with `parse_special = true` (hard-coded), so a chat template that embeds a **literal** BOS marker (Gemma `<bos>`, Llama 3 `<|begin_of_text|>`) already yields a BOS token — `AddBos::Always` then double-encoded it. The decision flows through two **pure, host-tested** helpers in [src/crates/primer-inference/src/llamacpp/params.rs](src/crates/primer-inference/src/llamacpp/params.rs):
  - `parse_add_bos_metadata(&str) -> Option<bool>` — parses `tokenizer.ggml.add_bos_token` (case-insensitive, trims, tolerant of "1"/"0"; `None` on anything unrecognised).
  - `should_prepend_bos(rendered, bos_piece, meta_add_bos) -> bool` — skip the prepend when `rendered` already starts with the BOS piece; else honour the metadata; else default to add-once (historical behaviour).
  `RealLlamaEngine` resolves the BOS piece (`token_to_piece(special=true)`) + metadata (`meta_val_str`) **once at model load**, caches both, and maps the per-call decision to `AddBos::Always`/`Never`. The same commit adds the ROADMAP §1.1 entry.

## Open decisions / risks

- **#201's behaviour change is unverified against a real GGUF.** The pure logic is host-tested and the feature compiles, but the actual token streams can only be checked with real Gemma/Qwen3 GGUFs on a C++/GPU toolchain. Until task #2 runs, the double-BOS fix is "implemented + host-tested," not "confirmed end to end." The blast radius is bounded: the only behaviour change vs. before is for models whose template embeds a literal BOS *or* whose `add_bos_token` metadata is `false` — common chat models (no literal BOS, no/true metadata) keep the exact prior add-once path.
- **Metadata-string format assumption.** `parse_add_bos_metadata` assumes llama.cpp's `meta_val_str` returns the bool as `"true"`/`"false"` (or `"1"`/`"0"`). If a converter emits something else (e.g. `"True"` is handled, but an unexpected encoding would return `None`), the code falls back to the historical add-once default — safe, but the metadata leg would be silently inert. The real-model smoke (#2) would surface this if it bites.

## Patterns to reuse, not reinvent

- **llamacpp pure-helper pattern.** `llamacpp/params.rs` holds the feature-INDEPENDENT logic (always compiled + host-tested); the `#[cfg(feature = "llamacpp")] mod real` block in `engine.rs` consumes it. Add new decision logic as a pure fn in `params.rs` with unit tests, then wire it into `RealLlamaEngine` — never put untested logic inside the feature-gated module. This mirrors the `MockLlamaEngine` seam (the `LlamaEngine` trait is always compiled).
- **Verifying llama-cpp-2 API before designing.** The crate isn't fetched on a default build. To inspect it: `cd /tmp && mkdir lcp2probe && … cargo fetch` a one-line project depending on `llama-cpp-2 = "=0.1.146"`, then read `~/.cargo/registry/src/index.crates.io-*/llama-cpp-2-0.1.146/src/model.rs`. Key facts found this session: `str_to_token` hard-codes `parse_special=true`; `meta_val_str(key)`, `token_bos()`, `token_to_piece(token, &mut decoder, special, lstrip)` are the accessors used.
- **Host gates for a llamacpp change:** `cargo test --workspace` (default) + `cargo clippy -p primer-inference --features llamacpp --all-targets` (the CI drift guard) + `cargo build -p primer-inference --features llamacpp`. Building the feature compiles bundled C++ llama.cpp (cached after first build, ~5 s incremental).
- **Android device-debug recipe (carried, for task #3):** logcat is dead on this ROM.
