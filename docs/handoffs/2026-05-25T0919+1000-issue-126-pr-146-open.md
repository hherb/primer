## What we shipped this session

**Closed issue #126** — `voice_loop` call sites for `SynthesisSession::push_text` / `finalize`. The pre-existing call shape passes the trait directive at [primer-core/src/speech.rs:239-246](src/crates/primer-core/src/speech.rs#L239-L246) — "async callers MUST wrap session calls in `tokio::task::spawn_blocking`". The issue's acceptance criteria allow either (a) actually wrapping or (b) documenting why the wrap is intentionally skipped. The brief's tentative recommendation was (a) as "the safer choice".

Option (a) turns out to be wrong, and the reason is non-obvious enough that it deserves the call-site comment block.

### Why option (a) breaks the macOS-native CLI

The macOS-native CLI is wired to `Builder::new_current_thread()` on the OS main thread (see [primer-cli/src/main.rs#L480-L498](src/crates/primer-cli/src/main.rs#L480-L498)). All async tasks poll on main; `MacosTtsSession::push_text` is called directly from main; `synthesize_streaming` sees `is_main_thread() == true` and takes its main-thread `runUntilDate` drain path — designed for synchronous use.

If we wrap `push_text` in `spawn_blocking`:
1. tokio hops the call onto a blocking-pool worker thread (NOT main).
2. `synthesize_streaming` sees `is_main_thread() == false` and takes the background-thread branch.
3. The background branch bounces work onto GCD main via `dispatch2::DispatchQueue::main().exec_async(...)`.
4. Per CLAUDE.md's macOS-native bullet (the section about `dispatch2`): **the GCD-main bounce only drains when the OS main thread is owned by `NSApplicationMain` or `dispatch_main()`**. The CLI's main thread is running tokio, not AppKit.
5. The bounced work is queued on GCD main but nothing dequeues it. Synthesis never returns. Deadlock.

The other two production backends tolerate the direct call as is:
- **macOS-native GUI**: multi-thread tokio + Tauri's `NSApplicationMain` on main. The direct call blocks one tokio worker for the synth duration; other workers keep running; the GCD bounce drains because Tauri owns main.
- **Piper Linux**: multi-thread tokio, no main-thread requirement. The direct call blocks one worker for the synth duration; other workers continue; ONNX inference proceeds on the worker thread.

The comment also documents the conditions under which the wrap *would* become correct (multi-tenant non-Apple runtime starving background tasks during SPEAK; new backend with no main-thread access constraint; macOS-native CLI moving off `new_current_thread`).

## Patterns to reuse, not reinvent

New from this session:

- **Trait-level `# Blocking` directives like "async callers MUST wrap in `spawn_blocking`" are advice, not law.** They are right by default — but a specific async caller may have an architectural reason to call sync directly, and that reason is worth documenting at the call site. Concretely: when the surrounding async runtime is `Builder::new_current_thread()` and the callee needs to run on the runtime's exact thread (the OS main thread, in macOS-native's case, for AVFoundation/main-thread-bound APIs), `spawn_blocking` would defeat the architecture by hopping the call off-thread. Pattern recurs anywhere a single-threaded runtime is paired with a "this work must happen on a specific thread" callee — Apple frameworks, GTK main loop, X11 server thread, anywhere with `#[thread_local]` resources, anywhere with a runloop that owns specific objects.
- **When refusing a trait directive at a call site, document three things**: (i) what the directive says and why it's the right default; (ii) the per-deployment-target rationale for skipping it (one bullet per production backend); (iii) explicit conditions under which the wrap would become correct again (so a future caller adding a fourth backend or a new runtime knows whether their case fits the existing exception or not). Without (iii) the comment ages badly — readers can't tell whether their use case is covered.
