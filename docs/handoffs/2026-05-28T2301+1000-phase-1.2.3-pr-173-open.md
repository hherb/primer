## What we shipped this session

**Phase 1.2 step 1.2.3** — the per-token streaming bridge for `QnnBackend`. Replaces the single-shot `GenieDialog::query_blocking` with `query_streaming(prompt, sender)` that drives the Genie C-ABI token callback into an mpsc channel. The receiver is wrapped as a `TokenStream`; `GenieDialog_query` runs inside `tokio::task::spawn_blocking` so multi-second decodes never starve the tokio runtime. Step 1.2.4 (CLI wiring under `--backend qnn`) is the natural next step — same crate, but expanding outward to `primer-engine` + `primer-cli`.

### Implementation topology

- **Trait change (`primer-inference/src/qnn/genie/mod.rs`)** — `query_blocking(&self, prompt) -> Result<String, GenieCallError>` is gone; replaced by `query_streaming(&self, prompt, sender: UnboundedSender<Result<TokenChunk>>)`. The sender's items are `Result<TokenChunk, PrimerError>` (not `GenieCallError`) so callers downstream see one error type without translating at every yield point. The trait method is synchronous-blocking — the real impl is invoked from `spawn_blocking`, the mock returns essentially instantly.

- **C-ABI bridge (`primer-inference/src/qnn/genie/real.rs`)** — `query_streaming` does:
  1. `Box::new(sender)` → `Box::into_raw` to get a stable `*mut UnboundedSender<...>` for the `user_data` slot.
  2. Register `streaming_token_callback` (an `unsafe extern "C" fn` that borrows the sender via `&*` and forwards each token as `Ok(TokenChunk { text, done: false })`).
  3. Call `dialog_query` (blocking; callbacks fire synchronously inside).
  4. After `dialog_query` returns: `Box::from_raw` reclaims the sender; on non-success status it emits one `Err(PrimerError::Inference(...))`, on success it emits one final `Ok(TokenChunk { text: "", done: true })`. The box drops, closing the channel.
  - Genie's documented "callbacks fire only inside `dialog_query`" contract is the load-bearing assumption that makes the lifetime bookkeeping linear: one `into_raw`, one `from_raw`, an unbounded number of non-consuming borrows in between. Step 1.2.0's chatapp_android validation is what would empirically confirm this against a real `libGenie.so` — until then it's structural-only.
  - Helper `reclaim_sender_box` centralises the `unsafe { Box::from_raw }` call with the safety contract documented once.
  - Helper `genie_to_primer_error` centralises `GenieCallError → PrimerError::Inference(InferenceError::Other(...))` so the dev-facing string is identical at every error site.

- **Mock (`primer-inference/src/qnn/genie/mock.rs`)** — new `Script` enum (`Tokens(Vec<String>) | TokensThenError { tokens, err }`) drives `query_streaming`. Three response-shape constructors:
  - `new_with_response("text")` — back-compat: one body chunk + one done chunk. Used by all existing tests that don't care about chunk count.
  - `new_with_tokens([t1, t2, t3])` — N body chunks (`done = false`) + 1 done chunk (`done = true`). Canonical step-1.2.3 shape.
  - `new_with_tokens_then_error([t1, t2], err)` — N body chunks + 1 `Err` chunk, no done. Simulates the real impl's mid-stream error path.
  - Existing `new_failing_open` and `new_failing_query` retained for the open-error and smoke-check-error paths. `new_failing_query` now emits one `Err` chunk on the channel (was: returned an error from `query_blocking`).

- **Backend (`primer-inference/src/qnn/backend/mod.rs`)** — `generate_stream` now:
  1. Renders the prompt through the chat template on the calling task (pure CPU, cheap).
  2. Creates `mpsc::unbounded::<Result<TokenChunk>>()`.
  3. `Arc::clone(&self.dialog)` into a `tokio::task::spawn_blocking` task.
  4. Task body: `dialog.blocking_lock()` (serialises concurrent callers — Genie forbids concurrent queries against the same dialog handle), then `guard.query_streaming(&rendered, tx)`. Sender drops when `query_streaming` returns → channel closes.
  5. Returns immediately with the receiver pinned as a `TokenStream`. Consumer drives the stream without blocking the runtime even when the underlying Genie call takes seconds.
  - `smoke_check` is now `async`, drains the streaming receiver to surface any `Err` chunk. Same FFI sequence as production turns, so a passing smoke check empirically verifies the streaming path.

## Open decisions / risks

Newly surfaced this session:

- **Decision to revisit at 1.2.3 resolved**: replaced `query_blocking` with `query_streaming` rather than keeping both. The previous brief's tentative call was to replace; the actual rationale held up — the smoke check converts cleanly to drain the streaming receiver, and production code only ever needs one path. The trait surface is now smaller by one method.
- **Sender error-types crossed the `GenieCallError → PrimerError` boundary at the trait edge.** Channel items are `Result<TokenChunk, PrimerError>`, not `Result<TokenChunk, GenieCallError>`. The mock and real impls both convert at the send site via the new `genie_to_primer_error` helper (real) or an inline `PrimerError::Inference(InferenceError::Other(err.to_string()))` (mock — because `GenieCallError` is not `Clone` we can't store one for repeated mock queries). Downstream consumers (`DialogueManager`, future `--backend qnn` wiring) see one error type. The i18n routing at step 1.2.4 can still discriminate on `GenieCallError` variants by re-parsing the dev string — not ideal, but the alternative (carrying `GenieCallError` through the channel) would force `Clone` on `GenieCallError` which transitively requires it on `std::io::Error`. **Decision to revisit at 1.2.4:** if the i18n surface really needs to discriminate `PlatformUnsupported` from `NonSuccess`, introduce a `PrimerError::Inference(InferenceError::PlatformUnsupported)` variant at the boundary rather than substring-matching the dev string.
- **Mid-stream cancellation is not supported.** Genie has no in-flight-cancellation API; if the consumer drops the receiver, body chunks continue to be sent via `unbounded_send` which returns `Err(SendError)` — we ignore those errors and let `dialog_query` run to completion. Acceptable for v1; if a future cancel-on-mic-resume contract needs to abort the LLM mid-decode (Phase 2 voice loop), the only escape valve is to wait for QAIRT to add a cancellation entry point or to teardown the dialog (much heavier).
- **The single-flight mutex acquired via `blocking_lock` inside `spawn_blocking` is the correct pattern**, but it does mean every `generate_stream` reserves a blocking-thread-pool worker for the full decode duration. tokio's default blocking pool is 512 workers — comfortably above any plausible concurrent-call count for the Primer (one child at a time). Worth flagging if a future use case (e.g. batch evaluation harness) drives many concurrent generations.

## Patterns to reuse, not reinvent

New from this session:

- **`Box::into_raw(Box::new(sender))` → C-ABI user_data → `Box::from_raw` after the synchronous-callback-firing call returns** is the cleanest pattern for forwarding tokens through an FFI callback into a Rust mpsc channel. The box is dropped exactly once (which closes the channel); the C-ABI callback only borrows the sender via `&*`, never consumes it. Genie's "callbacks only inside dialog_query" contract makes the lifetime bookkeeping linear. Same pattern fits any C API that takes a single user_data pointer and fires callbacks synchronously during a single blocking call.
- **`tokio::sync::Mutex::blocking_lock` inside `spawn_blocking`** is the canonical way to serialise access to a non-`Sync` resource (the Genie dialog handle) from an async-streaming entry point. The mutex enforces the single-session-per-dialog C contract; `blocking_lock` does not poll the runtime so it cannot deadlock the current-thread runtime; `spawn_blocking` keeps multi-second decodes off the async worker pool.
- **"Replace `Box::default()` accumulator with `Box::new(sender)` accumulator-channel" is the natural migration from collect-then-return to per-token streaming over an FFI callback.** The trait method signature gets one extra parameter (the sender); the C-ABI callback body changes from `acc.push_str(&token)` to `let _ = sender.unbounded_send(Ok(TokenChunk { ... }))`; the lifetime story is the same in both directions. Useful precedent for any future "non-streaming → streaming" upgrade across an FFI boundary.
- **"Sender error-types cross the boundary at the trait edge"** — the trait method takes `UnboundedSender<Result<TokenChunk, PrimerError>>` so all error translation happens once, at the mock/real impl side rather than in every consumer. The cost is that downstream i18n can no longer pattern-match on the original `GenieCallError` variant; the benefit is that consumers see one error type and the trait surface is unified with the rest of the codebase's `Result<T>` alias.
- **Smoke-check-through-the-streaming-path** discipline: the smoke check now drains the streaming receiver instead of having its own bespoke return-type. This means a passing smoke check exercises EXACTLY the FFI sequence (`set_token_callback` → `query` → callback fires → final done chunk + channel close) that production conversation hits. Same shape as "use the production code path for the canary test" patterns elsewhere — minimises the risk that the smoke check passes against a different code path than the one it's meant to canary.
- **`Script` enum-driven mock with per-shape helpers** (`emit_tokens` + `emit_tokens_then_done`) — clean way to factor the test surface so each shape (canned response, multi-token script, mid-stream error) is one constructor + one match arm, not three duplicate query implementations.
