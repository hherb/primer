## Patterns to reuse, not reinvent

New from this session:

- **"Rescue an orphaned branch via clean cherry-pick onto current `main`, not a merge/rebase of the stale branch."** When a branch is branched off old history but its changes are isolated, `git checkout -b <fresh-name> main && git cherry-pick <commit>` gives a clean, conflict-free PR branch without dragging the stale base. Verified the cherry-pick was clean with `git merge-tree $(git merge-base …)` first.
- **"Verify in an isolated worktree before opening the PR."** Built/tested the feature in `/tmp/primer-<x>-verify` (a `git worktree add` off `main` + cherry-pick) so the main checkout stayed clean, then removed the worktree. Lets you confirm CI-relevant facts (clippy/test green) before pushing, without risking the working tree.
- **"A `cargo check` drift-guard is not a clippy drift-guard."** If a feature combo can rot lints, add an explicit `cargo clippy --features <x> -- -D warnings` job — `cargo check --features <x>` will happily pass while clippy warnings accumulate.
- **"Fix the latent pre-existing lint inline when it blocks your PR's lint gate, and say so in the commit."** The example clippy rot wasn't introduced by #175, but it blocked `clippy --features supertonic -- -D warnings`; fixing it inline (separate commit, clearly labelled "latent since Stage A") is cheaper than a follow-up issue and keeps the gate green.
- **"Convert pip/venv → uv even in prose/docs."** The uv-only rule applies to documentation examples too, not just code — a runbook telling developers to `pip install` is still a uv-rule violation.
