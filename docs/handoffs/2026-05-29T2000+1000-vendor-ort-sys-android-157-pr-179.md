## What we shipped this session

### PR #179 — `fix(deps): vendor ort-sys rc.10 with an android cache_dir arm (#157)`

- Branch `deps/vendor-ort-sys-android-157`, single commit `399c665`. Implements issue #157's **unblocking path (b)**.
- **The fix:** vendored `ort-sys 2.0.0-rc.10` verbatim at `src/vendor/ort-sys/` (12 files, byte-identical to the crates.io release) with exactly **one cfg edit** in `src/internal/dirs.rs` — the Linux `cache_dir()` arm broadened from `#[cfg(target_os = "linux")]` to `#[cfg(any(target_os = "linux", target_os = "android"))]`. Wired via `[patch.crates-io] ort-sys = { path = "vendor/ort-sys" }` in `src/Cargo.toml` and added to `workspace.exclude`.
- **Why:** rc.10 gates `cache_dir()` on `windows | linux | macos` with **no catch-all**, so on a native Android host (Termux, `host == aarch64-linux-android`) the function is configured out and `build.rs:26`'s `use ...::cache_dir` fails to resolve (E0432) before any compile — the blocker that stopped `cargo build -p primer-cli --features embedding` on the RedMagic. Android shares the Linux XDG layout (Termux populates `$HOME`/`$XDG_CACHE_HOME`; `unix::home_dir()` already has an `android` arm), so the Linux logic works unchanged.
- **Why NOT path (a) (upstream PR):** I cloned `pykeio/ort` and wrote the equivalent main-branch fix, then found pykeio/ort's `CONTRIBUTING.md` **explicitly rejects AI-assisted PRs** ("PRs containing AI-generated or AI-assisted code/docs will be rejected"). Also, upstream `main` has **already** refactored `cache_dir()` to be unconditional (`ORT_CACHE_DIR` override + catch-all `None`), so the rc.10 E0432 no longer exists on main — only our load-bearing rc.10 pin needs a local fix. Path (a) is therefore both forbidden and moot.
- **Verification (from macOS).** The E0432 only manifests with an Android *host*; a cargo cross-compile from macOS can't reproduce it (build scripts always compile for the host). So the cfg behaviour was verified directly with `rustc --target aarch64-linux-android` on a `build.rs`-mimicking consumer module:
  - Unpatched rc.10 `dirs.rs` → **reproduces E0432** for the android target (exact issue symptom: "no `cache_dir` in `dirs`… found an item that was configured out").
  - Patched copy → **compiles clean** for the android target (and for the macOS host — no regression).
  - A counterfactual (broaden linux arm but forget the catch-all) → E0428 duplicate-definition — confirms the rc.10 shape needs only the one edit (rc.10 has no catch-all to adjust, unlike main).
  - `cargo metadata` confirms `[patch.crates-io]` resolves `ort-sys` to the vendored path.
  - **Default workspace test suite: 885 passed, 0 failed** (real `cargo test --workspace` exit 0 — re-run cleanly after an earlier `| tail` masked the exit code; see Patterns).
  - **Host `cargo build -p primer-embedding --features fastembed`** compiles the vendored `ort-sys` + `fastembed` cleanly in 30.52s, no unused-patch warning.

## Patterns to reuse, not reinvent

New from this session:

- **"Read CONTRIBUTING.md before drafting an upstream PR."** The pivot from path (a)→(b) hinged on finding pykeio/ort's anti-AI policy *before* opening a PR that would have been rejected on sight. Check a target repo's contribution rules (and whether the bug is already fixed on `main`) before investing in an external PR.
- **"A build-script E0432 is a HOST-cfg error; verify it at the cfg level with `rustc --target`, not `cargo`."** `cargo build --target X` compiles build scripts for the *host*, so it can't reproduce a "fails to build natively on X" error. The honest reproduction is a small consumer module compiled with `rustc --target X --emit=metadata` (no NDK linker needed), shown to fail on the original and pass on the patch.
- **"`| tail` masks the upstream exit code — never trust a piped exit code for a pass/fail claim."** An early `cargo test … | tail -25` reported exit 0 (tail's), hiding the real result. Re-ran as `cargo test … > log 2>&1; echo $?` to capture cargo's true exit before claiming 885/0.
- **"Vendor verbatim, patch minimally, document the drop-condition."** The vendored `ort-sys` is byte-identical to rc.10 except one cfg arm, with a comment + CLAUDE.md note saying exactly when to delete it — same discipline as the silero/piper/supertonic vendors.
