- **#201's behaviour change is still unverified against a real GGUF.** The pure logic is host-tested and the feature compiles, but the actual token streams can only be checked with real Gemma/Qwen3 GGUFs on a C++/GPU toolchain. Blast radius is bounded: the only behaviour change vs. before is for models whose template embeds a literal BOS *or* whose `add_bos_token` metadata is `false` — common chat models (no literal BOS, no/true metadata) keep the exact prior add-once path.
- **Metadata-string format assumption.** `parse_add_bos_metadata` assumes llama.cpp's `meta_val_str` returns the bool as `"true"`/`"false"` (or `"1"`/`"0"`, case-insensitive). An unexpected encoding returns `None` and the code falls back to the historical add-once default — safe, but the metadata leg would be silently inert. The real-model smoke (#1) would surface this.

## Patterns to reuse, not reinvent

- **llamacpp pure-helper pattern.** `llamacpp/params.rs` holds the feature-INDEPENDENT logic (always compiled + host-tested); the `#[cfg(feature = "llamacpp")] mod real` block in `engine.rs` consumes it. Add new decision logic as a pure fn in `params.rs` with unit tests, then wire it into `RealLlamaEngine` — never put untested logic inside the feature-gated module.
- **Verifying llama-cpp-2 API before designing.** The crate isn't fetched on a default build. To inspect it: `cd /tmp && mkdir lcp2probe && …` a one-line project depending on `llama-cpp-2 = "=0.1.146"`, then read `~/.cargo/registry/src/index.crates.io-*/llama-cpp-2-0.1.146/src/model.rs`. Key facts: `str_to_token` hard-codes `parse_special=true`; `meta_val_str(key)`, `token_bos()`, `token_to_piece(token, &mut decoder, special, lstrip)` are the accessors.
- **Host gates for a llamacpp change:** `cargo test --workspace` (default) + `cargo clippy -p primer-inference --features llamacpp --all-targets` (the CI drift guard) + `cargo build -p primer-inference --features llamacpp`.
