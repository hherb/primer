### PR #177 — `feat(pedagogy): per-backend context window budget (Phase 1.2 step 1.2.5)`

- Branch `feat/qnn-1.2.5-context-budget`, single commit `bb256a8`.
- **Behaviour:** small-context (≈4K-token) inference backends — today only QNN, whose `name()` is `"qnn:<model_id>"` — now use a **12-turn** recent-context window (vs 20) and a **3-passage** KB top-K (vs 5), so the assembled prompt fits a 4K model. Cloud/large-context backends are unchanged.
- **Cross-crate move:** `QNN_NAME_PREFIX` relocated to `primer_core::backend` (re-exported as `primer_inference::qnn::QNN_NAME_PREFIX`, so that public path still resolves). This is load-bearing: `primer-pedagogy` matches on the prefix but cannot depend on `primer-inference` (the DM only sees `&dyn InferenceBackend`). New pure helper `primer_core::backend::is_small_context_backend(name)`.
- **Config:** `PedagogyConfig` gains `context_window_turns_small_context` + `kb_top_k_small_context` (`Option`, default `Some(12)`/`Some(3)`) and methods `effective_context_window_turns(name)` / `effective_kb_top_k(name)`. Named `*_small_context` (not `*_qnn`) so a future 4K-bound non-Qualcomm backend reuses them by adding its prefix to `is_small_context_backend` — no rename. `None` forces the global value.
- **DM wiring:** every per-turn read of the window/top-K routes through the effective methods — `turn.rs` (prompt window), `summary.rs` (**both** refresh cadences, so the summary boundary never diverges from the prompt window), `retrieval.rs` (LTM window + KB top-K). LTM final-top-K (`LTM_FINAL_TOP_K`=3) was deliberately left alone — out of §1.2.5 scope and already small.
- **Consts:** new `consts::pedagogy::{DEFAULT_CONTEXT_WINDOW_TURNS=20, DEFAULT_CONTEXT_WINDOW_TURNS_SMALL_CONTEXT=12}` + `consts::retrieval::KB_FINAL_TOP_K_SMALL_CONTEXT=3`. The previously-inline `context_window_turns: 20` default now references the const (no-magic-numbers tidy ridden inline).
- **Tests (14 new):** 5 in `primer_core::backend` (prefix predicate, incl. empty + non-anchored substring), 6 in `PedagogyConfig` (effective methods + `None`-fallback both fields), 3 DM-level — resume under a `qnn:` backend lands the summary boundary at `total-12` (=13 for 25 turns) not `total-20`; KB retrieval requests top-K 3 under `qnn:` and 5 under cloud. Test support gained `ScriptedBackend::with_name(...)` and a `TopKRecordingKnowledge` mock.

## Open decisions / risks

Newly surfaced this session:

- **§1.2.5 design fork resolved toward backend-name detection, not token-capacity budgeting.** The plan left open "name fields backend-class-by-class" vs "add `context_capacity_tokens` to `primer-meta.json` and auto-budget on a prompt-token estimate." Chose the **name-prefix** approach because (a) the acceptance criteria are written against it, (b) the brief explicitly said pin detection via `QNN_NAME_PREFIX`, and (c) the token-capacity path needs a prompt-token estimator (a magic-number heuristic) + a new trait method + meta plumbing — a bigger PR for marginal generality. If a backend ever ships with a non-4K but still-bounded window, revisit: the cleaner long-term design is the capacity-token budget. The current design generalises to *other* 4K backends for free (add a prefix), just not to *arbitrary* window sizes.
- **LTM retrieval top-K (`LTM_FINAL_TOP_K`=3) was NOT shrunk for small-context backends.** Only the KB top-K and the recent-turn window are budgeted. LTM is already 3 and the window shrink is the dominant prompt-size lever; widening the budget to LTM is out of §1.2.5 scope. If on-device 4K runs still overflow, LTM top-K is the next knob (mirror the same `effective_*` pattern).
- **The 12-turn / 3-passage values are untested against a real 4K tokeniser.** They're principled estimates, not measured. Step 1.2.0's on-device validation (or 1.2.6's bench) is what would confirm the prompt actually fits 4K with headroom. If it overflows, lower `DEFAULT_CONTEXT_WINDOW_TURNS_SMALL_CONTEXT` (it's a single const).

## Patterns to reuse, not reinvent

New from this session:

- **"Put the cross-crate naming constant in the shared layer, not the producer crate."** `QNN_NAME_PREFIX` had to move from `primer-inference` to `primer-core` so the *consumer* (`primer-pedagogy`, which can't depend on `primer-inference`) could match on it. Re-export from the producer (`primer_inference::qnn::QNN_NAME_PREFIX`) to keep the public path stable. When a string/enum identity must be agreed across a dependency boundary, it belongs in the crate both sides already depend on.
- **"Name the config knob by the constraint it addresses, not the one backend that has it today."** `context_window_turns_small_context`, not `_qnn` — detection stays QNN-specific (a prefix list) but the budget field is reusable, so the next 4K backend is a one-line addition with no rename and no migration.
- **"Route every reader through one effective-value method so derived boundaries can't drift."** `summary.rs` computes a boundary from the window; if it read the raw `context_window_turns` while `turn.rs` used the effective one, turns would silently fall between the summary and the prompt window. Funnelling all three call sites through `effective_context_window_turns(name)` makes the invariant structural.
- **"Genuine TDD red even for trivial pure functions."** Stubbed `is_small_context_backend` → `false` and the config methods → global, watched the positive cases fail for the right reason, *then* implemented. Cheap insurance that the test actually exercises the branch.
- **"A behaviour-preserving const tidy can ride inline."** Moved the inline `context_window_turns: 20` default to a named const while adding the new ones — no-magic-numbers fix in the same PR rather than a follow-up.
