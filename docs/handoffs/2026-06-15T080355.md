## What we shipped this session (PR #230, branch `a11y-mobile-drawer-focus`)

**`e253da6` — feat(gui): focus management + scroll-lock for the mobile sidebar drawer (#226).** The phone-width (<940px) evaluation-sidebar overlay drawer (added in PR #225) now behaves as a proper modal:

1. **Focus management.** Opening the drawer moves focus into it (`#sidebar` gained `tabindex="-1"` so it accepts programmatic focus without joining the tab order); closing (backdrop tap / Esc / toggle) restores focus to `#sidebar-toggle`. Driven by a new named `applyDrawerModality(open)` helper in `ui/app.js` and a module-level `drawerReturnFocus` capture.
2. **Inert background.** While the drawer is open the chat `<main>` behind the dim backdrop is marked `inert` — unfocusable and hidden from the accessibility tree, so the composer textarea and conversation can't be tabbed to or read by assistive tech underneath.
3. **Scroll-lock.** A CSS `body.sidebar-open .chat-scroll { overflow: hidden }` rule scoped inside the `@media (max-width: 940px)` block freezes the conversation behind the drawer.

**Design decision (the issue asked us to decide):** the drawer is **NOT** converted to a native `<dialog>` `showModal()`. The same `#sidebar` element doubles as the *persistent desktop column* in the two-column layout; dialog/modal semantics would wrongly apply there. So modality is hand-rolled and mobile-only — `applyDrawerModality` no-ops on desktop (breakpoint guard), and the breakpoint-change handler clears `inert` unconditionally so an open drawer resized up to desktop never strands an inert chat (the `inert` DOM attribute isn't media-scoped, unlike the CSS scroll-lock + backdrop, which auto-release). The header (which holds the close toggle) z-stacks *above* the drawer + backdrop on mobile, so it stays operable as the close affordance — only the chat content is made inert.

**Tests added:** 4 new `responsive_layout_contract` tests (the established `include_str!` shape-assertion pattern — the GUI frontend has no JS test runner): `index_html_sidebar_is_programmatically_focusable`, `app_js_centralises_drawer_modality`, `app_js_inerts_chat_behind_open_drawer`, `stylesheet_scroll_locks_chat_behind_open_drawer` + a `sidebar_open_tag` helper. ## Open decisions / risks

- **PR #230 is open, not merged.** Merge once `cargo test (default features)` is green (host suite passed locally). 2 commits.
- **`inert` browser support.** The change relies on the global `inert` attribute, supported in WebKit 15.4+ / Chromium 102+ (so the Tauri WKWebView / WebView2 / webkit2gtk / Android System WebView targets all qualify). If an ancient WebView ever surfaces, the fallback is the CSS scroll-lock still applies and the backdrop still dims — only the focus/inert layer would no-op. Low risk on modern devices.
- **Design choice locked in:** the drawer is NOT a native `<dialog>`. Don't "modernise" it to `showModal()` later without accounting for the fact that `#sidebar` is also the persistent desktop column — modal semantics there would break the two-column layout. The mobile-only `applyDrawerModality` guard is load-bearing.
- **Pedagogy on a 4B NPU model remains the headline open quality question** — technically excellent, pedagogically unverified at scale.

## Patterns to reuse, not reinvent

New this session:
- **Mobile-only modal drawer = hand-rolled focus + `inert`, not a native `<dialog>`, when the same element is also a persistent desktop region.** The guard pattern: a named `applyDrawerModality(open)` helper that early-returns `if (!mobileQuery.matches)`, plus a breakpoint-`change` handler that *unconditionally* clears the non-media-scoped DOM state (`inert`, pending focus restore) so a resize-up-to-desktop can't strand it. Media-scoped state (CSS scroll-lock, backdrop visibility) needs no JS cleanup because the media query simply stops applying.
- **Scroll-lock for a mobile overlay = a CSS rule scoped to `body.<open-class> .<scroll-container>` *inside* the breakpoint media query**, not a JS `overflow` toggle — it auto-releases across the breakpoint and keeps desktop byte-identical.
- **Frontend a11y "TDD" here = extend `responsive_layout_contract.rs`** with `include_str!` shape-assertions (sidebar `tabindex`, the `applyDrawerModality` helper name, the `setAttribute("inert"`/`removeAttribute("inert"` pair, the scroll-lock CSS selector). Use the `strip_js_comments` helper so a doc-comment mentioning the marker can't false-pass; scope attribute checks to a single element via a small tag-extractor (`sidebar_open_tag`) rather than whole-document `.contains`.

Carried (prior QNN/device handoffs, still true): the standalone `qnn_bench` example **can't reach the DSP** from a sideloaded/Termux process on this ROM — on-device throughput is instrumented **inside the APK** (read `<app_data>/.primer/qnn_metrics.jsonl` via `run-as cat`; now opt-in — tick Settings → Diagnostics or set `PRIMER_QNN_METRICS_PATH`). The metrics file is bounded (1 MiB + single `.1` backup rotation, PR #229). A stream decorator must finalize on the terminal `done` chunk, not only on `None`.
