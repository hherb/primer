## What we shipped this session — PR #198 (#41)

**Branch:** `claude/issue-41-disambiguation-lead-anchor` (merged + deleted)
**Squash-merge commit on `main`:** `b0c25c6` — `fix(ingest): lead-anchor disambiguation prose markers to cut false positives (#41) (#198)`
**PR:** https://github.com/hherb/primer/pull/198 (merged; CI was all-green before merge)

Issue #41 asked to scope `data/ingest`'s disambiguation regex to lead-sentence patterns. The detector matched subject-predicate marker phrases (`may refer to`, `can mean`, `can refer to`, German `steht für`, `kann sich beziehen auf`) **anywhere in the first 300 chars** of a Wikipedia lead, falsely rejecting genuine articles whose body merely *contains* the phrase.

- **Two-kind marker model.** A disambiguation page's lead always opens with the **title as the grammatical subject**, immediately followed by the marker (`Mercury may refer to:`). New pure `_lead_anchored_marker(phrase)` in [data/ingest/wiki/source.py](data/ingest/wiki/source.py) anchors subject-predicate markers to lead-start — they now match only within the first `_DISAMBIGUATION_SUBJECT_MAX_CHARS` (= 40) chars, right after the short title-as-subject (`^.{0,40}?\bphrase\b`, lazy, `.` excludes newlines → first-line-only). **Explicit self-declaration markers** (`is a disambiguation`, `ist eine Begriffsklärung`, `Begriffsklärungsseite`, `ist mehrdeutig`) stay **un-anchored** — they never appear in a non-disambiguation lead, so matching anywhere guards against a false negative.
- **Deliberate tradeoff:** a disambiguation subject longer than 40 chars before the marker is now missed (rare false negative), but the issue notes false negatives are hand-recoverable while the old un-anchored rule's false positives silently blocked good articles. The committed seed JSONL is **unaffected** (those articles already passed); this only changes which articles a *future* ingest run rejects — no corpus regeneration, no retrieval-test impact.
- **TDD red→green:** 2 integration regression tests (EN `Reference (computer science)` + DE `Ampel` — the fixtures the issue asks to keep unblocked; both went red against the un-anchored code, green after) + 1 integration pin (`is a disambiguation` past the window still raises — pins the two-kind contract) + 3 pure unit tests for the factory (window boundary at exactly 40; case-insensitivity; first-line-only).
- **Docs:** CLAUDE.md `data/ingest` bullet now documents the two-kind model, the `_lead_anchored_marker` factory, the 40-char window const, and the false-positive-vs-false-negative tradeoff. ## Patterns to reuse, not reinvent

New from this session:
- **Two-kind classification beats one tuple of patterns when markers differ in *where* they may legitimately appear.** Subject-predicate phrases ("may refer to", "steht für") are only diagnostic at lead-start (right after the title-as-subject); self-declaration phrases ("is a disambiguation") are diagnostic anywhere. Encode the difference *in the compiled pattern* (`^.{0,N}?\bphrase\b` for the anchored kind) so the matching call site (`any(p.search(head))`) and the dataclass shape stay unchanged — zero churn to `WikiSource` or its drift tests.
- **A positional lead-anchor (`^.{0,N}?`) is more robust than exact-title-matching** for "title-as-subject" detection: it works regardless of whether the bolded lead string exactly equals the requested title (redirects, normalisation, casing), so it adds **no** new false-negative risk for the standard "<subject> <marker>" shape — which matters when the issue explicitly says false negatives are worse than false positives.
- **When a regex window is the fix, make the window a named const (`_DISAMBIGUATION_SUBJECT_MAX_CHARS = 40`) and `%`-format it into the pattern** (`r"^.{0,%d}?\b%s\b" % (N, re.escape(phrase))`) — keeps it out of the magic-numbers bucket and avoids f-string brace-escaping (`{{0,{N}}}`) confusion. `re.escape(phrase)` is belt-and-suspenders for literal marker phrases (spaces aren't escaped in Py 3.7+, so multi-word markers still match).
- **TDD for a detector tightening:** the red tests are the *previously-false-positive* inputs asserting "does NOT raise" (they fail against the loose rule); keep the *true positives* green as regression guards; add a direct unit test at the exact threshold (here: marker at offset 40+5 → no match) that the integration tests (far-inside / far-outside) don't pin.

Carried forward (prior handoffs): `Connection::prepare_cached(&stable_sql)` is the bulk-write idiom (pair with a per-instance precomputed-SQL struct); report a modest perf win honestly with the *why* and a comparative `#[ignore]` bench (assert correctness + only a pathological-regression guard, never a tight wall-clock bound); in-memory SQLite isolates CPU from disk in a write bench; to find squash-merged branches safe to delete, match local names against `gh pr list --state merged --json headRefName`, not `git branch --merged`; a `backup/*` branch whose tip isn't in `main` is a deliberate artifact — surface it, don't delete it; split an over-500-line module into a directory module via `git mv` so blame stays put; model a multi-file asset bundle as N single-file `kind`s; clap `required_if_eq` is blind to default values (ArgGroup + runtime validator); a hidden GUI form field must still be sent by `gather()` if its DTO field is mandatory; `--features speech` clippy is invisible to `--workspace` clippy — run it explicitly when touching voice code; run cargo from `src/` so the 1.88 pin is honored.
