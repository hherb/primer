## Open decisions / risks

- **PR #234 is open, not merged.** Merge once `cargo test (default features)` is green (host suite passed locally). 2 commits.
- **Sticky `×` design choice is locked in.** The in-dialog close is a sticky top-right `×` reusing `.modal-close`, NOT a full-width labelled button or sr-only-only. Don't "make it bigger / add a text label" later without owner sign-off — the placement was a deliberate product decision.
- **Both close affordances are intentional.** The header `#sidebar-toggle` (outside the subtree, sighted-keyboard path) AND the in-dialog `#drawer-close` (inside the subtree, screen-reader path) both stay live and both route through `setSidebarOpen(false)`. Don't remove one as "redundant" — they serve different a11y audiences.
- **Mobile-only scoping is load-bearing.** `.drawer-header` is `display: none` by default and only revealed in the `@media (max-width: 940px)` block, because the same `#sidebar` is the persistent desktop column where an in-panel close `×` makes no sense. Same scoping discipline as the rest of the drawer modality.
- **`inert` browser support** (carried from #230/#232): relies on the global `inert` attribute (WebKit 15.4+ / Chromium 102+ — all Tauri WebView targets qualify). Low risk on modern devices.
- **Design choice carried from #230/#232:** the drawer is NOT a native `<dialog>`. The `#sidebar` element is also the persistent desktop column; `showModal()` semantics there would break the two-column layout. The mobile-only `applyDrawerModality` guard remains load-bearing.
- **Pedagogy on a 4B NPU model remains the headline open quality question** — technically excellent, pedagogically unverified at scale.

## Patterns to reuse, not reinvent

New this session:
- **In-dialog modal close = a sticky control inside the `aria-modal` subtree, reusing the existing `.modal-close` glyph.** When a modal's container is dual-purpose (desktop column / mobile dialog), put the close button *inside* the container, hide it off-mobile (`display: none` default, revealed in the media query), make it `position: sticky; top: 0` so it survives a scrolling panel, and route its click through the SAME open/close function the toggle/backdrop/Esc use — so focus-restore + modality teardown stay in one place. Keep the out-of-subtree toggle live too (it serves the sighted-keyboard path); the two close controls are complementary, not redundant.
- **Scope a markup contract-test to a subtree with a `<aside>…</aside>` extractor.** `sidebar_subtree(html)` spans the opening `<aside` to its matching `</aside>` so a `.contains("id=\"drawer-close\"")` assertion can't false-pass on a stray element elsewhere in the document — the analogue of the existing `sidebar_open_tag` opening-tag extractor, but for inner content rather than attributes.

Carried (still true): the strict mobile-modal focus trap = inert the non-toggle background controls, not a keydown Tab handler (#232); toggle ARIA-modal role in JS, mobile-only, when the modal element is dual-purpose, with one `releaseDrawerModality()` teardown shared by close + breakpoint-change (#232); the mobile-only modal-drawer guard pattern from #230 (`applyDrawerModality(open)` early-returns off-mobile; breakpoint-`change` handler unconditionally clears non-media-scoped DOM state); frontend a11y "TDD" here = extend `responsive_layout_contract.rs` with `include_str!` shape-assertions using `strip_js_comments` + the tag/subtree extractors.
