## What we shipped this session

**Issue #149** — split `backends_macos.rs` (742 lines, the largest CLAUDE.md guideline violator) into two sibling files, one per builder, each behind exactly one cargo feature; extracted the post-mic construction (speaker + output resampler + `on_audio` + drain hook + input resampler) into shared helpers in `backends_common.rs`.

### Refactor topology

- **Builder split** (file → feature gate is now 1:1):
  - `backends_macos_native.rs` (306 lines, gate `target_os = "macos" + cpal + macos-native`) — SFSpeechRecognizer + Silero + AVSpeechSynthesizer. Carries the generic `run_audio_thread_stt` audio-thread body since no other builder uses it.
  - `backends_macos_native_26.rs` (282 lines, gate `target_os = "macos" + cpal + macos-native-26`) — SpeechAnalyzer + derived VAD + AVSpeechSynthesizer. Inline `std::thread::Builder::spawn` closure body pulled out to `run_audio_thread_macos26` so the builder stays linear.
- **Shared helpers** in [`crates/primer-speech/src/voice_loop/backends_common.rs`](src/crates/primer-speech/src/voice_loop/backends_common.rs) — consumed by all three voice-loop builders (`backends.rs` whisper+piper also adopts them):
  - `SpeakerPipeline::start(tts_rate, verbose) -> Result<Self>` — opens the speaker + builds the output resampler. Holds the `Arc<Mutex<HeapProd>>` + `Arc<AtomicBool>` (errored flag) + `Arc<Mutex<Option<Resampler>>>` that every builder threads through both the `on_audio` closure and the drain hook.
  - `make_on_audio(spk_prod, spk_errored, output_resampler, output_chunk_in, need_output_resample) -> Box<dyn FnMut(Vec<f32>) + Send>` — synthesised-PCM → speaker-ringbuf closure (resampler-tail wrap + end-of-turn flush sentinel). Free function with explicit primitives so the closure logic is testable without a real `SpeakerSink`.
  - `make_drain_hook(spk_prod, spk_errored) -> DrainHook` — `spawn_blocking` + `wait_for_drain` hook the loop calls at the end of each SPEAK phase.
  - `open_mic_with_resampler(target_rate, target_chunk_samples, verbose) -> Result<MicPipeline>` — mic + input resampler as a bundle.
- **Constants centralised**: `OUTPUT_RESAMPLER_CHUNK_IN`, `PUSH_RETRY_SLEEP`, `DRAIN_POLL_TICK`, `DRAIN_CONSECUTIVE_ZERO_CHECKS`, `DRAIN_GRACE`, `DRAIN_MAX_WAIT`, `FLUSH_SILENCE_CHUNKS` move from per-builder inline literals into one `pub(super) const` block. Single source of truth — change a tunable once, all three builders see it.
- **Three new TDD unit tests on `make_on_audio`** in `backends_common::tests` (vanilla `ringbuf::HeapRb`, no cpal device needed): pushes straight through without resampling; empty flush sentinel is a no-op without resampling; bails before pushing when `spk_errored` is already set. First unit-test coverage of the on_audio closure body — previously it had only smoke coverage via real-audio examples.

## Patterns to reuse, not reinvent

New from this session:

- **Bundle-struct + free-function shape for shared closure builders.** `SpeakerPipeline` is a struct that owns the cpal-driven state (mic stream, ringbuf producer mutex, errored flag, resampler); `make_on_audio` + `make_drain_hook` are free functions taking explicit primitives (the Arcs). Why split it this way: the bundle struct can only be tested by opening a real cpal device, but the free functions accept enough primitives to be tested against a vanilla `ringbuf::HeapRb` with no cpal involvement. The bundle gives readers the cohesive "open the speaker pipeline" verb; the free functions give tests a unit-of-test seam. Reuse this shape when a piece of glue code carries both "needs a real device to construct" state AND "pure logic that operates on that state" — split the verbs by who can test them.
- **One file = one cargo feature gate** is a quick mechanical win on file-size violations. The old `backends_macos.rs` carried a *union* gate (`any(macos-native, macos-native-26)`) because two builders shared a file. Splitting by builder gives each file the tightest possible gate, which also makes the feature dependency graph easier to reason about (read the top of the file → know which feature you need).
- **Public re-exports protect call sites from internal renames.** PR #162 documents this in CLAUDE.md and uses it as the migration target for the two latent-broken paths. If a future refactor wants to rename `backends_macos_native_26.rs` → `backends_apple_26.rs` (likely once an iOS host application lands), no caller has to change.

Carried forward (from prior sessions; see prior handoff trail for two-layer drain, capacity-1-channel-as-sync-lever, pure-helper extraction, `#[path]` + `#[cfg(test)] mod` two-load pattern for build-script TDD, gate-narrowing, RFC-2229 whole-struct capture, and trait-directive refusal documentation patterns).
