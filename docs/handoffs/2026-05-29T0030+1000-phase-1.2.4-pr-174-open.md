### Implementation topology

- **`primer-engine` (new `qnn` feature; new `build_qnn_backend` helper)** — `Cargo.toml` adds `qnn = ["primer-inference/qnn"]`. `wiring::BackendParams` gains `qnn_bundle_dir: Option<PathBuf>` and `qnn_qairt_lib_dir: Option<PathBuf>` (always present in the struct so callers don't have to cfg-gate field initializers — the only feature-gated piece is the dispatch arm itself). `build_backend("qnn", ...)` dispatches to `build_qnn_backend(params)`:
  - `#[cfg(feature = "qnn")]` arm: validates required `qnn_bundle_dir`, falls back to `default_qairt_lib_dir(bundle)` when `qnn_qairt_lib_dir` is unset, delegates to `QnnBackend::new(bundle, lib_dir)`.
  - `#[cfg(not(feature = "qnn"))]` arm: returns `Err` with a "rebuild with `--features primer-cli/qnn`" hint that's deliberately distinct from the generic "unknown backend" message — users need a different action (build-time vs typo-fix).
  - New pure helper `default_qairt_lib_dir(bundle_dir: &Path) -> PathBuf` resolves the conventional AI Hub apps layout (`<bundle>/../qairt/lib/aarch64-android/`). Re-exported from `lib.rs`.

- **`primer-cli` (new `qnn` feature, new flags, new warning)** — `Cargo.toml` adds `qnn = ["primer-inference/qnn", "primer-engine/qnn"]`. `main.rs` adds two clap flags, both `#[cfg(feature = "qnn")]`-gated so default `--help` stays clean:
  - `--qnn-bundle-dir <DIR>` with `required_if_eq("backend", "qnn")` and `env = "PRIMER_QNN_BUNDLE_DIR"`.
  - `--qnn-qairt-lib-dir <DIR>` with `env = "PRIMER_QNN_QAIRT_LIB_DIR"`; optional.
  - `--backend` doc-comment updated to enumerate the five accepted values (`stub|cloud|ollama|openai-compat|qnn`).
  - `--model` doc-comment notes that for qnn the flag is ignored (model id comes from `primer-meta.json`).
  - `main_model` resolution seeds a `"qnn-pending"` placeholder for `--backend qnn` and rebinds to `backend.name()` (e.g. `"qnn:Qwen3-4B"`) after `build_backend` returns so the classifier / extractor / comprehension identifiers downstream carry the real model id.
  - Banner row added: `"Using qnn (Qualcomm NPU) backend with {main_model}."`.
  - "Unknown backend" hint updated to mention `qnn`.
  - `BackendParams` initialiser plumbs the new fields, cfg-gated to populate only when feature is on:
    ```rust
    #[cfg(feature = "qnn")]
    qnn_bundle_dir: cli.qnn_bundle_dir.clone(),
    #[cfg(not(feature = "qnn"))]
    qnn_bundle_dir: None,
    ```
  - New `warn_on_npu_serialisation(cli)` is called from the banner block. Backed by a pure unit-testable `npu_serialisation_warning(classifier, extractor, comprehension) -> Option<String>` that:
    - Returns a warning when every subsystem resolves to `qnn` (all background LLM work serialises behind the chat turn through the dialog mutex).
    - Returns a warning when every subsystem is explicitly `stub` (conversation loses classifier-driven features).
    - Returns `None` for any mixed configuration (the canonical "stub classifier, NPU chat" setup is silent — that's the recommended shape).
  - The "cloud-classifier with missing `ANTHROPIC_API_KEY`" case from the plan is **not** added here — it's already structurally caught by `build_backend("cloud", ...)` which errors when `api_key` is `None`. Adding a CLI-layer pre-check would be duplicative.

- **`primer-gui` (forced `BackendParams` literal update)** — `wiring.rs` already builds a `BackendParams`; it picks up two new fields set to `None`. The GUI has no UI for picking a QNN bundle yet, so this is the deliberate "wired but not surfaced" shape: a GUI user picking `--backend qnn` will get the "qnn-bundle-dir required" error path. Tracked alongside other Phase 1.2 GUI work — when the GUI settings page gains a QNN section, populate these fields from the picker.

- **`primer-inference/src/qnn/backend/mod.rs` (Send-bounds latent fix)** — surfaced by the new engine arm being reachable through the GUI's Tauri command futures (`generate_handler!` requires the command futures to be `Send`). The construction-time smoke check previously held a `&dyn GenieDialog` across an `.await` inside `smoke_check(dialog: &dyn GenieDialog) -> Result<()>`, which forced `dyn GenieDialog: Sync`. The `GenieDialog` trait deliberately omits `Sync` (the C-side forbids concurrent queries; the type-level Send-only bound is the safer of the two options). Fix: split into two free fns:
  - `fire_smoke_check_query(dialog: &dyn GenieDialog) -> UnboundedReceiver<...>` — sync; the borrow ends with the function's return.
  - `drain_smoke_check_receiver(rx: UnboundedReceiver<...>) -> Result<()>` — async; captures only the owned receiver, no dialog reference. Future is `Send` without `dyn GenieDialog: Sync`.
  - Caller (`QnnBackend::new_with_library`) chains them: `let rx = fire_smoke_check_query(&*dialog); drain_smoke_check_receiver(rx).await?;`. The borrow doesn't cross the await.

## Open decisions / risks

Newly surfaced this session:

- **`BackendParams` fields are unconditional, not feature-gated.** Two `Option<PathBuf>` fields (`qnn_bundle_dir`, `qnn_qairt_lib_dir`) are always present in the struct regardless of the `qnn` cargo feature. The only feature-gated piece is the dispatch arm itself in `build_qnn_backend`. The trade-off: callers (CLI, GUI) don't need to cfg-gate struct-literal fields — they cfg-gate the field VALUES (CLI populates from cli args under `cfg(feature = "qnn")`, otherwise sets to `None`; GUI always sets `None`). This is more ergonomic than feature-gating the struct shape, at the cost of two unused `Option`s on default builds (a few bytes each). The alternative — feature-gating fields — would propagate `#[cfg]` to every BackendParams literal across `primer-cli` AND `primer-gui`, which is exactly the kind of feature-cfg-leak the project avoids.
- **`--backend qnn` ignores `--model`.** The model id is read from the bundle's `primer-meta.json` and the CLI rebinds `main_model` to `backend.name()` post-construction. Documented in the `--model` doc-comment. The alternative (require `--model` to match the bundle's id, or reject when `--model` is set) is more user-hostile — the user already gave us the bundle path, why force them to repeat the id from inside it? Decision-to-revisit if a future scenario surfaces where the user needs to override the bundle's reported id (e.g. fine-tune branding).
- **Two warnings (all-qnn, all-stub) are emitted as informational `eprintln!("Warning: ...")`, not gated by `--verbose`.** Visible to every user who picks `--backend qnn`. The reasoning: both configurations are subtle pitfalls a first-time user might not anticipate (especially the all-qnn → serialised case, which is functionally correct but degrades the conversation cadence on memory-constrained devices). Tuck-into-`--verbose` would defeat the purpose. If a future use case wants the warnings silenced, add a `--no-warn-npu-serialise` flag rather than gating behind verbosity.
- **The smoke-check split is a load-bearing pattern for FFI traits that are deliberately `Send`-only.** Whenever a trait can be `Send` but not `Sync` (e.g. C-side enforces exclusive access at runtime, not at the type level), holding `&dyn Trait` across an `.await` will force `Sync`. Split the fire (sync, takes `&dyn`) from the drain (async, takes owned receiver/handle) to keep the borrow off the async path. Documented at the new helpers; reuse this shape for any future FFI-trait + async-callback work. Same rationale that motivated the original `tokio::task::spawn_blocking` move for `GenieDialog_query` — but at the construction boundary instead of the steady-state boundary.

## Patterns to reuse, not reinvent

New from this session:

- **"Feature-gate the dispatch arm, not the struct shape"** for `BackendParams`-style configuration types that thread through multiple binaries. Keeping the struct's fields unconditional lets every call site stay free of `#[cfg]` litter; the feature-gating concentrates in one place (the dispatch helper). Two unused `Option<PathBuf>`s on default builds is a worthwhile trade against propagating cfg-noise across every literal.
- **"Distinct error messages for build-time vs runtime causes"** when a feature isn't compiled in. `"qnn backend requires the qnn cargo feature"` and `"unknown backend: <typo>"` are different *actions*: rebuild vs fix the spelling. The unit test `qnn_without_feature_returns_build_hint` pins both halves: the new error mentions `qnn + feature`, AND does not contain the generic `"unknown backend"` substring.
- **"Sync-fire, async-drain"** for FFI-trait method calls whose returned channel must be drained asynchronously, when the trait is deliberately `Send`-only (no `Sync`). The sync function takes `&dyn Trait`; the async function takes the owned receiver. The borrow ends before the await — `dyn Trait: Sync` is never required, and the trait keeps its deliberately-tight bound. Reusable for any future FFI-trait + async-callback combination.
- **"Post-construction model-name rebind"** for backends whose model id is discovered inside the bundle, not provided by the user. Seed a placeholder pre-construction so the dispatch helper has a String to take; rebind `main_model = backend.name().to_string()` after construction. Threads cleanly through the existing classifier/extractor/comprehension dispatch matrix without any matrix-level changes.
- **"`required_if_eq` + `env` clap attributes compose"** — `--qnn-bundle-dir` is required when `--backend qnn` UNLESS `PRIMER_QNN_BUNDLE_DIR` is set in the env. clap treats `env` as a satisfying value source for `required_if_eq` automatically. The CLI test handles the degenerate case where the env var is set externally (skips the assertion rather than failing).
