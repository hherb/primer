  1. **The #166 smoke** — `src/crates/primer-speech/tests/whisper_stream_reuse.rs`. Exercises the *actual* `WhisperStt` cache-reuse path (`open_session` → `push_audio` → `finalize`, twice) rather than only the `StreamCache<i32>` mechanism PR #164's unit tests cover. Proves **reuse-invariance**: utterance B is transcribed twice with identical greedy params — once *cold* (fresh `WhisperStt`, empty cache → freshly-constructed `WhisperStream`) and once *reused* (single `WhisperStt` where A constructs+caches the stream and B reuses it via `reset()`). Greedy decode is deterministic, so the transcripts must match unless the preserved `WhisperState` bled A's decoder/KV state into B — a strictly stronger, ground-truth-free form of "B doesn't contain A's terms". `#[cfg(feature = "whisper")]` + `#[ignore]`'d; reads `PRIMER_WHISPER_MODEL` + two 16 kHz mono WAVs from `PRIMER_WHISPER_AUDIO_A` / `_B`.   2. **The macOS C++ link fix** — `src/crates/primer-speech/build.rs`. Building the smoke surfaced that `--features whisper` *standalone* fails to link on macOS (`___gxx_personality_v0` undefined). `whisper-cpp-plus-sys` links the C++ runtime for Linux (`stdc++`) but **not** macOS, relying on a transitive libc++ (e.g. via onnxruntime under `silero`/`piper`). A whisper-only build has none. `build.rs` now emits `cargo:rustc-link-lib=c++` on macOS when the `whisper` feature is active (no-op when libc++ is already linked). This is why no whisper-only test target ever linked here before — every prior whisper consumer (`primer-cli --features speech`, `voice-loop`) also pulled `ort`.

## Open decisions / risks

- **Branch not yet pushed at handoff** (item #0) — outward-facing, owner-authorised.
- **The build.rs `c++` link fix is macOS-only and correct for modern macOS** (libc++, not libstdc++). It is redundant-but-harmless when libc++ is already linked transitively (linking the same dylib twice is a no-op), so the combined-feature builds (`voice-loop`, `primer-cli --features speech`) are unaffected. If a future macOS whisper build ever needs libstdc++ instead, revisit — but that is not the case today.
- **The reuse-invariance assertion is exact normalised-equality** between cold-B and reused-B. Greedy `best_of = 1` decode is deterministic for a fixed model+params+audio, so this should be stable; if the owner ever sees a spurious mismatch from decode nondeterminism (threading), the failure message makes the diff obvious and the assertion can be relaxed to a term-overlap check. Not expected.
- **The smoke is `#[ignore]`'d** — CI never exercises the model path; only the owner does (item #1). This matches the project's established "pure helpers host-tested, model-needing path `#[ignore]`'d" pattern (mirrors `whisper::stream_cache_tests` and the bench harness).

## Patterns to reuse, not reinvent

- **`whisper-cpp-plus-sys` does not link the C++ runtime on macOS** — only Linux (`stdc++`). Any whisper-*only* target (no `ort`/onnxruntime in the feature set) needs `cargo:rustc-link-lib=c++` or it fails to link with `___gxx_personality_v0` undefined. `primer-speech/build.rs::link_cxx_runtime_on_macos` now handles this for the crate; remember it if you ever add a whisper-only consumer elsewhere.
- **Build scripts compile for the HOST**, so `#[cfg(target_os)]` in a build.rs reflects the host, not the build target — read `CARGO_CFG_TARGET_OS` from env for target-OS branching (as both `whisper-cpp-plus-sys` and the new helper do).
- **The whisper.cpp first build is ~20 min** (bundled C++ via the sys crate's build script). The compiled objects are cached in the build-script `OUT_DIR` and survive `RUSTFLAGS` changes (which only re-link/re-check the Rust crates), so iterating on link directives is cheap after the first build.
- **"pure helpers ungated + model-needing test gated/`#[ignore]`'d"** is the house pattern for backend-dependent tests — keeps the plumbing CI-covered without a model on disk. See this file's `helper_tests` (always run) vs `reuse_smoke` (`#[cfg(feature = "whisper")]` + `#[ignore]`).
