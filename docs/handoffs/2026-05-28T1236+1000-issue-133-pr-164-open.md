## What we shipped this session

**Issue #133** — eliminated the per-utterance ~500 ms cold-start cost on the Whisper STT path. The `WhisperStream::new()` allocation block (KV cache + compute buffers) used to fire on every `VadEvent::SpeechStart`; now the stream is held alive inside `WhisperStt` and reused via `WhisperStream::reset()` between utterances.

### Implementation topology

- **`StreamCache<T>` (new, ~30 lines)** — private module in `crates/primer-speech/src/whisper.rs`. Single-slot `Mutex<Option<T>>` with `take()` / `put()` / `new()`. Generic so the invariants can be unit-tested against `i32` (constructing a real `WhisperStream` needs a GGML model on disk, which is outside the unit-test budget). Per-instance contention is impossible by construction (the voice loop runs one VAD-driven session at a time per backend) so a plain `Mutex` keeps the type `Send+Sync` without atomics. Poisoned-mutex paths silently drop the value — symmetric with cache-miss behaviour from the caller's perspective.
- **`WhisperStt` carries `cache: Arc<StreamCache<WhisperStream>>`** — constructed empty in `new()`. The first `open_session` `take()`s `None`, falls through to the cold-start path (`WhisperStream::new(&self.ctx, params)`); every subsequent call `take()`s `Some(stream)`, calls `reset()` (clears `audio_buf`, `pcmf32_old`, `prompt_tokens`, `n_iter`, `total_samples_processed`; KV cache + compute buffers stay), and reuses it.
- **`WhisperSession` was reshaped from `{ stream: WhisperStream }` to `{ stream: Option<WhisperStream>, cache: Arc<StreamCache<WhisperStream>> }`.** The `Option` only exists so `finalize` and `Drop` can `take()` ownership out of the session for the put-back. `push_audio` derefs through the `Option` with an `expect()` — the only paths that take from it are exit paths, so an in-use session always sees `Some`.
- **`finalize` puts the stream back even on a `flush()` error.** A flush failure would otherwise force every subsequent utterance to pay the cold-start tax for the rest of the process lifetime; the put-back is unconditional and the error propagates afterwards.
- **`Drop` puts the stream back if `finalize` was never called** (`push_audio` returned an error and the caller dropped the box; a panic upstream). `reset()` in the next `open_session` normalises any partial state.

## Open decisions / risks

Newly surfaced this session:

- **The "TDD with unit-tested generic + mechanical integration" pattern is the only practical TDD shape for the Whisper cache.** A real `WhisperStream` needs a GGML model on disk to construct, so the cache invariants can only be unit-tested by parameterising the cache over `T` and testing against `i32`. The integration glue (open_session takes / finalize+Drop puts) stays mechanical and is covered structurally by existing voice-loop integration tests (the trait shape is unchanged). If a future bug surfaces in the take/put wiring itself, the only way to add direct coverage is either (a) ship a tiny GGML test fixture in the repo, or (b) restructure `WhisperStt` to take a stream factory closure so the production constructor can be mocked. Both have a real cost; defer until the bug actually appears.
- **`flush()`-error vs put-back policy was a deliberate design choice.** `WhisperSession::finalize` puts the stream back into the cache **before** propagating any `flush()` error. The alternative — propagate first, drop the stream — would force every subsequent utterance for the rest of the process lifetime to pay the cold-start tax. The cost of putting back is a `reset()` call in the next `open_session`; the cost of not putting back is hundreds of ~500 ms latency spikes across a long session. The trade-off is asymmetric in favour of put-back.
- **`Drop` is the panic / early-return safety net, but it duplicates `finalize`'s put-back logic.** Both call `self.cache.put(stream)`. The duplication is cheap (one line each) and the `Drop` impl is the only path that catches a panic in the caller before the session reaches `finalize`. Skipping `Drop` would make a panic-in-`push_audio` permanently degrade the backend.
- **`cargo test -p primer-speech --features whisper` (whisper-feature alone) fails to link with C++ stdlib symbols.** Pre-existing; the workspace's default-features build, the voice-loop feature combos, and the macos-native combos all build & test cleanly. Not blocking; not in scope for #133. Worth a follow-up issue if any future PR wants to run whisper-only tests in isolation.

## Patterns to reuse, not reinvent

New from this session:

- **"Generic single-slot cache + mechanical caller wiring" is the right TDD shape when the cached type can't be constructed without expensive system resources.** `StreamCache<T>` is generic so the cache logic gets dense unit-test coverage against a cheap type (`i32`); the caller-side wiring (`WhisperStt::open_session` take, `WhisperSession::finalize` + `Drop` put) carries no logic worth testing in isolation — it's three lines of glue covered transitively by every voice-loop integration test that runs. Cleaner than threading a stream-factory closure through the trait shape and far cleaner than shipping a test GGML fixture.
- **`Option<T>` inside a session struct, taken in `finalize` AND `Drop`, is the standard pattern for "consume on success AND on panic".** `finalize` is the success path; `Drop` is the bailout path. Both call `Option::take()` and put the resulting `Some(value)` back to a shared cache. `push_audio` and other in-use methods see `Some` by invariant and unwrap via `expect()` — the panic message documents the invariant rather than guarding against it.
- **Put-back-before-propagate is the asymmetric policy for caches keyed by expensive construction.** When the put-back is cheap and the cold-start is expensive, return the resource before propagating the error. The reverse (propagate then drop) makes a single transient error permanently degrade the system.

Carried forward (from prior sessions; see prior handoff trail for two-layer drain, capacity-1-channel-as-sync-lever, pure-helper extraction, `#[path]` + `#[cfg(test)] mod` two-load pattern for build-script TDD, gate-narrowing, RFC-2229 whole-struct capture, trait-directive refusal documentation patterns, bundle-struct + free-function shape for shared closure builders, one-file-one-cargo-feature-gate as a quick file-size win, and public-re-export protecting call sites from internal renames).
