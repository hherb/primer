## Patterns to reuse, not reinvent

- **Always run the full sweep before claiming a backlog is empty.** The correct survey is `find crates -name '*.rs' -not -path '*/vendor/*' -not -name 'tests.rs' | xargs wc -l | awk '$1>500'` then `grep -c '#[cfg(test)]'` each to find the ones with a big inline test module. Do this first.
- **Test-module extraction is the lowest-risk, zero-churn oversized-file cut.** Move `#[cfg(test)] mod tests { … }` into a sibling file and replace the inline block with `#[cfg(test)]\nmod tests;`. The test module is a *descendant*, so `use super::*` reaches all parent privates (incl. cfg(test) helpers) with no visibility edits. Twelve PRs in a row (#268–#279) used exactly this. Prefer it over moving production code down (which costs `pub(super)`/`pub(crate)` churn).
- **The test module name may NOT be `tests`.** A prior session's was `v4_tests`. Match the declaration (`mod <name>;`) and child filename (`<name>.rs`) to the actual module name — grep the `mod …` line, don't assume `tests`.
- **Mind the path form: crate root vs. non-root module.** A `lib.rs`/`main.rs` (crate root) submodule resolves to `src/<name>.rs` directly. A non-root module `foo.rs`'s submodule resolves to `src/foo/<name>.rs`. A root-level module like `cpal_io.rs` (declared in `lib.rs`) is itself non-root for *its* submodule → `src/cpal_io/tests.rs` (this session, needing a new `cpal_io/` dir). Pick the right one or the build fails to find the module.
- **Use ABSOLUTE paths or a single confirmed `cd` in `sed`/`head`/`mkdir` commands — the Bash tool's cwd persists across calls and a leading `cd` that fails silently runs the rest in the wrong dir.** This session worked cleanly from `cd /Users/hherb/src/primer/src` with `crates/…`-relative paths after confirming cwd. Always `git status --short` after to catch stray files.
- **Verify the marker count before the `sed` cut.** `grep -c '#[cfg(test)]'` — exactly `1` means a single clean trailing module; `>1` means there may be standalone `#[cfg(test)] fn` helpers OR multiple sibling test modules. Multiple sibling test modules each carry their own `use super::*`, which resolves differently once moved down a level — find every marker line and confirm the boundary + `super` handling by hand.
- **Relocate blocks with `sed` line ranges, never hand-transcription.** `head -N` to trim the parent, `sed -n 'A,Bp' > child.rs` to extract the module *body* (inner lines only — drop the `mod tests {` wrapper and closing brace), then `cargo fmt` to dedent. **Confirm the move is clean with `git diff -- parent.rs | grep '^+' | grep -v '^+++'`** — the only addition to the parent should be `mod <name>;`.
- **`cargo fmt` may leave one blank line above the first relocated line that `--check` then rejects.** After moving a module body, run `cargo fmt --all -- --check` and delete the leading blank line if flagged (`sed -i '' '1{/^$/d;}'`). One-line fix; don't be surprised by it.
- **For a refactor, the test COUNT is the regression guard.** Capture a pre-move baseline (here: cpal 45 + 1 ignored) and confirm the identical counts pass post-move under the new module path. No new tests authored, none needed.
- **Check `cfg(feature …)` in the moved span AND the crate's feature setup before trusting a default `cargo test`.** Here the entire `cpal_io` module (and thus its tests) is gated on `--features cpal`, so a default `cargo test -p primer-speech` would never even compile the moved tests — the cpal feature run is the load-bearing regression guard.
