## Open decisions / risks

- **The macOS fastembed CI step is a dual-purpose gate.** It proves the cdn.pyke.io ort-runtime download AND implicitly asserts macOS hybrid is buildable at all. If it fails on `macos-latest` because no rc.10 ort binary exists for that arch, do NOT paper over it — it means reassessing whether macOS should ship hybrid default-on (or pinning the macOS runner arch). The failure is the correct signal.
- **First-run UX change.** A fresh default build now downloads ~570 MB BGE-M3 on first conversation (with the "Loading fastembed model…" banner), falling back to BM25-only + a `tracing::warn!` on download failure. Intended, but it IS a behaviour change for casual build-and-run users. `--embedder-backend none` or `--no-default-features` opts out.
- **Default CI build time increased.** The default Linux test job now compiles ort + fastembed for both `primer-cli` and `primer-gui` and downloads the ort runtime on a cold cache. `Swatinem/rust-cache` keeps subsequent runs warm; first run after merge is slower.
- **Android remains BM25-only by guidance AND by CI structure now.** The `--no-default-features` android-guard pin is the structural enforcement. If #157 on-device validation later proves the aarch64 ort download works, revisit whether to drop the pin and flip Android too.

## Patterns to reuse, not reinvent

New from this session:

- **Feature-aware runtime default = compile-time-conditional literal, not a runtime branch.** When a default must track a cargo feature AND the API takes a literal (clap `default_value`, a config `Default` impl), use `#[cfg_attr(feature = "x", ...)]` + a `not(...)` arm (CLI) or a cfg-gated `fn default_*() -> &'static str` helper (GUI/config). This lets a `--no-default-features` build degrade gracefully instead of hard-failing — strictly better than a naive flip that errors on a flagless run.
- **Flipping a cargo default has CI blast radius — audit every job that builds the flipped crate.** The non-obvious consequence here was the Android cross-compile guard: `cargo build -p primer-cli` (and `--features qnn`, which is additive to defaults) would have pulled the device-unverified aarch64 ort download. The fix (`--no-default-features` on those steps) ALSO happens to match the documented product guidance. Always grep CI for every `cargo build/check/clippy` of the crate whose default you flip.
- **A "prove the download works" CI gate is build-level, not run-level.** The cdn.pyke.io ort binary is fetched by `ort-sys`'s `build.rs`, so a `cargo check`/`clippy --features fastembed` triggers it — no need to download the 570 MB model or run inference in CI. Keep the heavy model-download recall test on-demand.
- **The final holistic review catches cross-file staleness the per-task reviews miss.** Per-task spec reviews are scoped to one file; the whole-branch review found stale "opt-in"/"default none" language in CLAUDE.md paragraphs the plan hadn't enumerated. Always run a final branch-wide review before PR, especially for doc-flip changes where the same claim is restated in many places.
