## Patterns to reuse, not reinvent

New from this session:
- **A new inference-routing/resilience behavior is a decorator over `Arc<dyn InferenceBackend>`, built by `build_main_backend`, not a DM change.** `RouterBackend` mirrors `FallbackBackend` exactly: wrap legs, delegate, `name()` → primary (load-bearing for `is_small_context_backend`), fall over ONLY at the pre-stream `…().await` `Result` boundary (never mid-stream). The router REUSES the fallback's `--fallback-*` secondary leg + `plan_main_backend` truth table rather than adding new flags.
- **Rich per-turn signals reach a decorator via an `Option<T>` field on `GenerationParams` that every other backend ignores.** Adding `routing: Option<RoutingSignals>` was a one-field core change + a confined `turn.rs` threading change; no other backend or call site cares. Default `None` ⇒ pure fn scores 0.0 ⇒ safe no-panic default. Use this shape for any future "the DM knows something the bare `Prompt` doesn't" signal.
- **Keep the routing/scoring POLICY pure and in `primer-core`** (`RouterMode`/`RoutingSignals`/`complexity_score`/`order_legs`), separate from the MECHANISM (`RouterBackend` in `primer-inference`). The policy is then unit-tested on the default `cargo test` with zero I/O, and `primer-pedagogy` (which can't depend on `primer-inference`) can reference the types.
- **An exhaustive `match` over a closed enum (no wildcard) is a deliberate compile-time tripwire** — `intent_weight`'s match forces a new `PedagogicalIntent` variant to be scored explicitly rather than silently defaulting to 0.
- **Subagent-driven execution: inline-verify trivial pure-additions (consts/enum), dispatch full reviewers for integration tasks.** Over 11 tasks, the controller read the diff for the trivial pure modules and spent dedicated spec+quality reviewers on the cross-cutting ones (wiring, GUI). When a subagent dies on an infra error (socket close) with nothing committed, re-dispatch or — for a verbatim file the controller authored — write it directly; verify via the task's own `cargo test`.
- **The GUI `validate()` module is the right home for cheap structural cross-field invariants** (e.g. "routing mode requires a fallback secondary") — it surfaces the error inline at save time, mirroring the CLI's early validation, instead of waiting for the identical construction-time error at `start_session`.

Carried forward (prior handoffs): a benchmark for a new inference backend is "thin example + shared `bench` core" (not a fresh harness); all-`Option` `BenchTargets` + vacuous-pass `evaluate` is the one mechanism for both hard-gate and measurement-only; mirroring a CLI feature into the GUI is a 3-struct (`Config`/`View`/`Update`) + wiring + frontend pattern (the `Update` DTO has no `#[serde(default)]`, so every new field is MANDATORY in every IPC test payload + `settings.js::gather()`); a reliability/routing decorator for any `Arc<dyn Trait>` is a GoF decorator; split a construction decision into a pure truth-table fn + thin glue; a new pub fn in a sub-module needs an explicit crate-root re-export; adding a non-`Option` field to a shared struct breaks EVERY literal (grep `StructName {`) — but literals using `..Default::default()` are immune; for a new in-process inference backend, the QNN/llamacpp trait-seam + mock + `spawn_blocking` + two-cfg `build_*_backend` is the template; run cargo from `src/` with `+1.88`; GPU-variant cargo features must each chain to the base feature in every crate; docs-only PRs are CI-path-ignored (#168) → BLOCKED → admin-merge/direct-push.
